## 1. Как устроена HashMap внутри?

`HashMap` хранит данные в массиве `Node<K,V>[] table`, где каждая ячейка (бакет) содержит связный список узлов. Каждый `Node` хранит четыре поля: `int hash`, `K key`, `V value` и `Node<K,V> next` — ссылку на следующий узел в том же бакете. Размер массива всегда является степенью двойки, что позволяет вычислять индекс через битовую операцию `hash & (capacity - 1)` вместо медленного деления по модулю.

При вызове `put(key, value)` HashMap вычисляет `key.hashCode()`, затем применяет «перемешивающую» функцию `(h = key.hashCode()) ^ (h >>> 16)` — XOR старшей половины хэша с младшей. Это снижает коллизии для ключей, у которых hashCode различается только в старших битах. Полученное значение маскируется: `index = hash & (n - 1)`, где `n` — текущий capacity. Начиная с Java 8, когда в одном бакете накапливается 8 и более узлов (и общий capacity ≥ 64), список превращается в красно-чёрное дерево для поддержания O(log n) в худшем случае.

Реальная ошибка: разработчик ожидает, что `HashMap` всегда работает за O(1), и не думает о качестве hashCode. Если все ключи возвращают одинаковый hashCode (например, `return 42`), все элементы попадут в один бакет — и структура деградирует до списка O(n) или дерева O(log n).

```java
// Упрощённая структура узла (из исходников JDK)
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;

    Node(int hash, K key, V value, Node<K,V> next) {
        this.hash = hash;
        this.key = key;
        this.value = value;
        this.next = next;
    }
}

// Вычисление индекса бакета (из JDK)
// hash(key): (h = key.hashCode()) ^ (h >>> 16)
// index: (n - 1) & hash
```

❗ На собесе часто спрашивают: «почему размер таблицы всегда степень двойки?» — чтобы можно было заменить дорогое `hash % n` на быстрое `hash & (n-1)`. Ещё ловушка: `table` инициализируется лениво — при создании `new HashMap()` массив ещё не выделен, выделение происходит при первом `put`.

**→ Уточняющий вопрос:** Что происходит с внутренним массивом при вызове `new HashMap()` без аргументов — массив выделяется сразу или отложенно?

**↳ Ответ:** Отложенно (lazy initialization). При `new HashMap()` поле `table` остаётся `null`, память под массив не выделяется. Только при первом вызове `put()` вызывается `resize()`, который аллоцирует массив размером 16. Это оптимизация памяти: если map создаётся но никогда не заполняется, она не занимает лишние байты.

## 2. Что такое bucket (корзина) в HashMap?

Бакет — это одна ячейка внутреннего массива `Node[] table`. По сути это «слот», к которому привязан ноль или больше узлов, образующих связный список (или дерево при большом числе коллизий). Когда два ключа дают одинаковый индекс, их узлы добавляются в один бакет и соединяются через поле `next`.

Индекс бакета вычисляется по формуле `(n - 1) & hash`, где `n` — текущий capacity. Например, при capacity=16 ключ с hash=17 попадёт в бакет с индексом `16-1 & 17 = 15 & 17 = 1`. При resize capacity удваивается, и элементы перераспределяются по новым бакетам — элемент либо остаётся на том же месте, либо переходит на позицию `oldIndex + oldCapacity`.

Реальная проблема: если все ключи попадают в один бакет (атака хэш-коллизиями или плохой hashCode), время поиска деградирует. До Java 8 это был вектор DoS-атаки на веб-приложения. Java 8 решила это через treeification.

❗ На собесе: «сколько бакетов в HashMap по умолчанию?» — 16. Не путать с «количеством элементов».

**→ Уточняющий вопрос:** Если два ключа попали в один бакет, как HashMap находит нужный ключ при вызове `get()`?

**↳ Ответ:** HashMap проходит по цепочке (или дереву) в бакете и для каждого узла проверяет: `e.hash == hash && (e.key == key || key.equals(e.key))`. Сначала сравниваются хэши — это дёшево. Только при совпадении хэшей вызывается `equals()` — это дороже. Правило: если `equals()` дорогой — убедись, что `hashCode()` хорошо распределяет объекты, чтобы минимизировать количество вызовов `equals()`.

## 3. Как HashMap определяет, в какой bucket положить элемент?

Процесс состоит из двух шагов. Шаг первый — вычисление хэша: берётся `key.hashCode()`, затем применяется «spreading» функция: `h ^ (h >>> 16)`. Это XOR значения самого с собой, сдвинутым на 16 бит вправо. Цель — «смешать» старшие и младшие биты, чтобы учесть информацию со всего диапазона int при выборе бакета.

Шаг второй — вычисление индекса: `(capacity - 1) & hash`. Поскольку capacity — всегда степень двойки, это эквивалентно `hash % capacity`, но работает в разы быстрее. Например, для capacity=16: `(16-1) & hash = 0b1111 & hash` — берутся только последние 4 бита хэша.

Проблема без spreading: если у объектов hashCode отличается только в старших битах (что часто бывает у `Long`, `Double` или ключей типа `new Point(x, y)`), без XOR все они дали бы одинаковые младшие биты и попали бы в один бакет.

```java
// Метод hash() из HashMap JDK (вспомогательный)
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

// При put:
// int index = (table.length - 1) & hash(key);
```

❗ На собесе: «зачем XOR со сдвигом?» — это самый тонкий вопрос по HashMap. Ответ: младшие биты определяют бакет, а старшие биты hashCode в них не участвуют без spreading — растут коллизии.

**→ Уточняющий вопрос:** Что произойдёт, если у всех ключей hashCode() возвращает одно и то же число?

**↳ Ответ:** Все элементы попадут в один бакет. При capacity < 64 HashMap будет делать resize за resize'ом (потому что treeification не включается до 64), увеличивая таблицу без пользы. При capacity ≥ 64 бакет превратится в дерево — `get()`/`put()` деградируют до O(log n). Структура данных работает корректно, но в разы медленнее. Это именно модель HashDoS-атаки: подобрать ключи с одинаковым хэшем и перегрузить сервис.

## 4. Что такое коллизия в HashMap?

Коллизия — это ситуация, когда два или более различных ключа дают одинаковый индекс бакета. Это математически неизбежно: количество возможных ключей (практически бесконечно) намного превышает количество бакетов (например, 16). Коллизия сама по себе не является ошибкой — HashMap умеет с ней работать через цепочки (chaining).

Важно понимать разницу: два ключа могут иметь одинаковый `hashCode()` (→ точно в один бакет), или разные hashCode, но одинаковый индекс бакета после маскирования `(n-1) & hash` (→ тоже коллизия бакета). Первый случай — «полная коллизия хэша», второй — «частичная коллизия индекса».

Реальная проблема: если приложение использует нехорошие ключи (например, объекты с `hashCode() { return 1; }`), HashMap становится связным списком. Это был вектор атаки на приложения, принимающие входные данные от пользователей (HashDoS).

❗ На собесе: «Java 8 решила проблему коллизий» — только частично. Деревья помогают при DoS-атаке, но не устраняют сам факт коллизии. Качественный hashCode по-прежнему важен.

**→ Уточняющий вопрос:** Как HashMap определяет, что среди нескольких узлов в бакете именно этот — нужный?

**↳ Ответ:** По паре хэш + equals: `e.hash == targetHash && (e.key == targetKey || targetKey.equals(e.key))`. Первое условие — быстрая проверка, что хэши совпадают (int сравнение). Второе — полная проверка ссылочного или структурного равенства. Если оба условия выполнены — это нужный узел. Именно поэтому нарушение контракта (равные объекты → разные хэши) ломает поиск: HashMap смотрит не в тот бакет.

## 5. Как HashMap обрабатывает коллизии?

HashMap использует метод цепочек (separate chaining): все элементы с одинаковым индексом бакета хранятся в одной «цепочке» — связном списке узлов. При `get(key)` HashMap переходит в нужный бакет и проходится по всем узлам, сравнивая `hash == nodeHash && (key == nodeKey || key.equals(nodeKey))`. Проверка хэша идёт первой — она дешевле, чем `equals()`.

Начиная с Java 8, когда в бакете накапливается 8+ узлов и общий capacity ≥ 64, список преобразуется в красно-чёрное дерево (treeification). Это снижает время поиска в переполненном бакете с O(n) до O(log n). При удалении узлов, если их в дереве осталось 6 или меньше, дерево снова преобразуется в список (untreeification).

```java
// Упрощённая логика поиска в бакете
Node<K,V> e = table[hash & (n-1)];
while (e != null) {
    if (e.hash == hash &&
        (e.key == key || (key != null && key.equals(e.key)))) {
        return e; // нашли
    }
    e = e.next;
}
return null; // не нашли
```

❗ На собесе: «условие treeification — это только 8 элементов в бакете?» — нет! Нужно ещё `capacity >= TREEIFY_THRESHOLD = 64`. Если capacity < 64, HashMap сначала сделает resize, а не treeification. Это типичная деталь, которую забывают.

**→ Уточняющий вопрос:** При каком условии дерево в бакете обратно превращается в список?

**↳ Ответ:** Когда в бакете остаётся ≤ 6 узлов (`UNTREEIFY_THRESHOLD = 6`) — происходит при удалении элементов или при resize (когда бакет разбивается пополам). Порог 6, а не 8, чтобы не было «пинг-понга»: добавили 8-й → дерево, удалили → список, добавили → дерево... Гистерезис в 2 элемента делает переключение стабильным.

## 6. Что происходит при достижении 8 элементов в одном bucket?

При накоплении 8 элементов в одном бакете HashMap проверяет общий `capacity` таблицы. Если он меньше 64 (`MIN_TREEIFY_CAPACITY`), HashMap выполняет `resize()` — удваивает массив. Это помогает разбить переполненный бакет на два. Только если capacity уже ≥ 64, список преобразуется в красно-чёрное дерево (`TreeNode`).

Пороговые константы: `TREEIFY_THRESHOLD = 8` (список → дерево), `UNTREEIFY_THRESHOLD = 6` (дерево → список при удалении). Порог 6, а не 8, для отмены дерева введён намеренно: чтобы не было дорогостоящего «пинг-понга» между структурами при добавлении/удалении 8-го элемента.

Красно-чёрное дерево в бакете использует `TreeNode` — подкласс `Node`, у которого есть дополнительные ссылки `parent`, `left`, `right` и флаг `red`. Из-за этого каждый узел в дереве занимает примерно вдвое больше памяти, чем обычный `Node`.

❗ На собесе: «что происходит при 8 элементах в бакете?» — нельзя просто ответить «список превращается в дерево». Нужно добавить условие `capacity >= 64`, иначе ответ неполный. Это показывает знание исходников JDK.

**→ Уточняющий вопрос:** Почему порог возврата к списку равен 6, а не 8 (как порог перехода к дереву)?

**↳ Ответ:** Гистерезис: если бы оба порога были 8, при чередующихся add/remove на границе 8 элементов структура конвертировалась бы при каждой операции, что дорого. Разрыв в 2 создаёт «буферную зону»: нужно удалить ещё 2 элемента после treeification, чтобы вернуться к списку. Стандартный приём при проектировании систем с гистерезисом.

## 7. Что такое контракт equals() и hashCode()?

Контракт — это набор правил, которым обязаны следовать реализации `equals()` и `hashCode()` совместно. Ключевое правило: **если `a.equals(b)` возвращает `true`, то `a.hashCode()` должен быть равен `b.hashCode()`**. Обратное не требуется: объекты с одинаковым hashCode не обязаны быть равны (это просто коллизия хэша).

Требования к `equals()`: рефлексивность (`x.equals(x) == true`), симметричность (`x.equals(y) == y.equals(x)`), транзитивность (если `x.equals(y)` и `y.equals(z)`, то `x.equals(z)`), консистентность (результат не меняется, если объекты не изменились), `x.equals(null) == false`.

Требования к `hashCode()`: консистентность (тот же объект возвращает тот же хэш при повторных вызовах), согласованность с `equals()` (равные объекты — равные хэши), желательно: разные объекты — разные хэши (для производительности, но не обязательно).

```java
// Корректная реализация через Objects.hash()
public class Point {
    private final int x;
    private final int y;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y); // корректно: равные Point дадут равный хэш
    }
}
```

❗ На собесе: «назови контракт одной фразой» — «равные объекты должны иметь одинаковый hashCode». Обратное — нет. Нарушение этого правила ломает все хэш-коллекции.

**→ Уточняющий вопрос:** Что будет, если нарушить симметричность в `equals()` — например, `a.equals(b) = true`, но `b.equals(a) = false`?

**↳ Ответ:** Поведение непредсказуемо: `map.get(a)` может найти значение (если `a` был ключом), но `map.get(b)` — нет, хотя `a.equals(b)`. `HashSet.contains(b)` вернёт `false`, даже если в сете есть `a`. Нарушение симметрии — наиболее коварная ошибка при наследовании: подкласс переопределяет `equals()` с проверкой типа через `instanceof`, и `parent.equals(child)` работает иначе чем `child.equals(parent)`.

## 8. Если два объекта равны по equals(), что можно сказать об их hashCode()?

Они обязаны быть равны. Это прямое следствие контракта `hashCode()`/`equals()`. HashMap и другие хэш-структуры используют это допущение: сначала ищут бакет по hashCode, затем внутри бакета находят узел через `equals()`. Если равные объекты попадут в разные бакеты (разный hashCode), `get()` никогда не найдёт значение по «логически равному» ключу.

Это требование не является рекомендацией — это жёсткий контракт, нарушение которого вызывает непредсказуемое поведение. Проблема коварна тем, что код компилируется и работает без исключений, но возвращает неверные результаты: элементы «теряются», `contains()` возвращает `false` для присутствующих объектов.

```java
// Нарушение контракта — equals переопределён, hashCode — нет
class BadKey {
    int id;
    BadKey(int id) { this.id = id; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof BadKey)) return false;
        return this.id == ((BadKey) o).id;
    }
    // hashCode НЕ переопределён — наследуется от Object (адрес объекта)
}

Map<BadKey, String> map = new HashMap<>();
BadKey k1 = new BadKey(1);
map.put(k1, "value");

BadKey k2 = new BadKey(1);  // equals(k1) == true
System.out.println(map.get(k2)); // null ! разные hashCode → разные бакеты
System.out.println(k1.equals(k2)); // true
```

❗ На собесе: IDE-генерация `equals()` без `hashCode()` (или наоборот) — реальная ошибка новичков. IntelliJ IDEA и Lombok (`@EqualsAndHashCode`) всегда генерируют пару. Принципиально важно понимать, почему это требование несимметрично.

**→ Уточняющий вопрос:** В каком именно месте кода `HashMap.get()` сломается, если два равных объекта дали разный hashCode?

**↳ Ответ:** Сломается на вычислении индекса бакета: `(n-1) & hash(key)`. Если `k1.hashCode() != k2.hashCode()`, то `hash(k1) != hash(k2)`, значит и индексы будут разные. HashMap заглянет в бакет для k2 — там либо пусто, либо другие элементы. До вызова `equals()` вообще не дойдёт. Именно поэтому нарушение контракта «тихое» — код без исключений, просто `get()` возвращает `null`.

## 9. Если два объекта имеют одинаковый hashCode(), обязательно ли они равны по equals()?

Нет, это не обязательно. Одинаковый hashCode означает лишь, что объекты попадут в один бакет. Внутри бакета `equals()` является финальным арбитром при поиске конкретного ключа. Ситуация, когда неравные объекты имеют одинаковый hashCode, называется коллизией хэша и является допустимой.

Крайний пример: возвращать `return 42` из `hashCode()` технически не нарушает контракт — все объекты будут равны по хэшу, но разными по `equals()`. Коллекция будет работать корректно, просто медленно: все элементы попадут в один бакет и поиск займёт O(n) вместо O(1).

Хорошая реализация hashCode стремится распределять объекты равномерно по всем бакетам, чтобы минимизировать коллизии. Именно поэтому `Objects.hash(field1, field2, ...)` предпочтительнее примитивных реализаций вроде `return field1 + field2`.

❗ На собесе: «что такое коллизия?» — не путать «коллизию хэша» (одинаковый hashCode у разных объектов) и «коллизию бакета» (разные hashCode, но одинаковый индекс после маскирования). Оба случая обрабатываются одинаково, но это разные явления.

**→ Уточняющий вопрос:** Если hashCode() возвращает одно константное значение (например, 42), как это повлияет на производительность HashMap?

**↳ Ответ:** Все элементы попадут в один бакет. `get()` и `put()` деградируют до O(log n) (дерево) после 8+ элементов, или O(n) (список) при capacity < 64 пока не вырастет таблица. HashMap продолжает работать корректно — результаты верные — но производительность убита: поиск среди 10 000 элементов займёт ~13 шагов вместо ~1.

## 10. Что произойдёт, если переопределить equals() но не переопределить hashCode()?

HashMap полностью сломается для логически равных объектов. При поиске `get(key)` HashMap сначала вычисляет hashCode ключа и идёт в нужный бакет. Если два логически равных объекта имеют разный hashCode (а без переопределения `hashCode()` это так — наследуется `Object.hashCode()`, который зависит от адреса объекта), они окажутся в разных бакетах. `get()` заглянет в «неправильный» бакет и ничего не найдёт.

Это нарушение основного контракта: «равные по `equals()` объекты должны иметь одинаковый `hashCode()`». Нарушение не вызывает исключений — код молча работает неверно. Особенно коварно в больших кодовых базах, где класс используется как ключ только в некоторых местах.

```java
class User {
    String name;
    User(String name) { this.name = name; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof User)) return false;
        return name.equals(((User) o).name);
    }
    // hashCode НЕ переопределён!
}

Map<User, Integer> scores = new HashMap<>();
User alice1 = new User("Alice");
scores.put(alice1, 100);

User alice2 = new User("Alice");
System.out.println(alice1.equals(alice2)); // true
System.out.println(scores.get(alice2));    // null — alice2 попал в другой бакет!
System.out.println(scores.containsKey(alice2)); // false — хотя логически ключ есть
```

❗ Самый популярный вопрос на собесе по HashMap. Правило: переопределяй `equals()` и `hashCode()` **всегда в паре**. Lombok `@EqualsAndHashCode`, `record`, или IDE-генерация гарантируют это.

**→ Уточняющий вопрос:** Какие инструменты или паттерны помогают не забыть переопределить оба метода?

**↳ Ответ:** 1) `record` (Java 16+) — автоматически генерирует `equals()` + `hashCode()` по всем компонентам. 2) Lombok `@EqualsAndHashCode` — аннотация генерирует оба метода. 3) IDE (IntelliJ IDEA) — «Generate → equals() and hashCode()» всегда создаёт пару. 4) `Objects.equals()` + `Objects.hash()` — стандартный шаблон при ручной реализации. Правило: если класс предназначен для использования как ключ или в множестве — всегда используй `record` или Lombok.

## 11. Что произойдёт, если переопределить hashCode() но не переопределить equals()?

`equals()` из `Object` сравнивает ссылки (`==`). Два разных объекта с одинаковым содержимым будут считаться разными даже в одном бакете. В `HashSet` появятся логические дубликаты — одинаковые объекты, лежащие рядом. В `HashMap.get()` нужный ключ не будет найден, если передать не тот же самый экземпляр, а «логически равный».

Этот случай менее катастрофичен, чем первый (с точки зрения «молчаливой» ошибки): элементы будут правильно группироваться в бакеты, но проверка равенства внутри бакета всегда будет сравнивать ссылки. Это можно заметить раньше, потому что `HashSet` будет содержать дубликаты.

```java
class Product {
    String sku;
    Product(String sku) { this.sku = sku; }

    @Override
    public int hashCode() {
        return sku.hashCode(); // хэш переопределён
    }
    // equals НЕ переопределён — сравнивает ссылки (==)
}

Set<Product> catalog = new HashSet<>();
catalog.add(new Product("ABC-123"));
catalog.add(new Product("ABC-123")); // другой объект, тот же sku

System.out.println(catalog.size()); // 2 — дубликаты! equals сравнил ссылки
```

❗ На собесе: «какой из двух случаев опаснее — забыть `equals` или `hashCode`?» — оба плохи, но забытый `hashCode` при переопределённом `equals` коварнее: `get()` молча возвращает `null` вместо исключения.

**→ Уточняющий вопрос:** Что вернёт `contains()` у HashSet, если в нём два «одинаковых» объекта из-за несовместимого equals/hashCode?

**↳ Ответ:** Зависит от случая. Если `hashCode` переопределён правильно но `equals` нет — оба объекта попадут в один бакет, но `contains()` вернёт `false` для «второго» (т.к. `equals` сравнивает ссылки). Если `hashCode` не переопределён — они попадут в разные бакеты и `contains()` тоже вернёт `false`. В обоих случаях `size()` будет 2 вместо 1. Ответ на собесе: «корректность `contains()` в HashSet полностью зависит от корректности `equals()` + `hashCode()`».

## 12. Можно ли использовать изменяемый объект как ключ в HashMap?

Технически — да, HashMap не запрещает это и не проверяет. Но это крайне опасно: если изменить поля объекта-ключа, которые участвуют в вычислении `hashCode()`, после того как он уже был добавлен в HashMap, запись фактически «потеряется» — станет недостижимой через `get()`, хотя физически остаётся в таблице.

Проблема в том, что HashMap запоминает хэш только при вставке и не пересчитывает его при модификации ключа. При `get()` он вычислит новый hashCode, получит новый индекс бакета, заглянет туда — и не найдёт запись, потому что она лежит в старом бакете.

Единственный безопасный сценарий: изменяемый ключ, но поля, используемые в `equals()`/`hashCode()`, никогда не меняются после добавления в map. Это требует строгой дисциплины и хорошей документации класса. В реальных проектах лучше использовать иммутабельные ключи.

❗ На собесе: «что плохого в изменяемом ключе?» — многие отвечают абстрактно. Нужен конкретный механизм: ключ лежит в бакете по старому hashCode, после мутации hashCode меняется, `get()` ищет в новом бакете → `null`.

**→ Уточняющий вопрос:** Что произойдёт с памятью, если большое количество ключей «потеряется» таким образом в долгоживущей HashMap?

**↳ Ответ:** Утечка памяти. Объекты ключей и значений остаются реферируемыми через `Node` в таблице — GC не может их собрать. `size()` показывает реальное количество (включая потерянные), capacity растёт при resize, что создаёт ещё больше аллоцированных слотов. В долгоживущих сервисах это ведёт к постепенному росту heap и в конечном счёте к OutOfMemoryError. Решение: не мутировать ключи, либо использовать иммутабельные объекты.

## 13. Что случится, если изменить ключ после добавления в HashMap?

При вставке `put(key, value)` HashMap вычисляет `hash(key)` и сохраняет запись в соответствующем бакете. После изменения объекта-ключа его `hashCode()` вернёт другое значение. При следующем вызове `get(modifiedKey)` HashMap вычислит новый хэш, получит новый индекс бакета, пройдётся по нему — и не найдёт запись, которая лежит в старом бакете.

Запись при этом не удаляется автоматически. Она «зависает» в таблице: не достижима через `get()`, но занимает память и учитывается в `size()`. Через `entrySet().iterator()` её всё ещё можно увидеть, потому что итератор проходит по всем бакетам напрямую, игнорируя hashCode.

```java
List<Integer> mutableKey = new ArrayList<>(Arrays.asList(1, 2, 3));
Map<List<Integer>, String> map = new HashMap<>();
map.put(mutableKey, "value");

System.out.println(map.get(mutableKey));  // "value" — пока ключ не менялся

mutableKey.add(4);  // мутируем ключ — hashCode изменился!

System.out.println(map.get(mutableKey));  // null — записи нет в новом бакете
System.out.println(map.size());           // 1 — запись физически всё ещё в map
System.out.println(map.containsValue("value")); // true — итератор её находит
```

❗ На собесе: демонстрация с `List` как ключом — классический пример, который сразу показывает понимание механизма. Вывод: используй иммутабельные ключи — `String`, примитивные обёртки, `record`, либо классы с `final` полями в `hashCode`.

**→ Уточняющий вопрос:** Как такой «потерявшийся» ключ влияет на результат `map.size()` и на поведение итератора по `entrySet()`?

**↳ Ответ:** `map.size()` включает потерявшиеся записи — они всё ещё в таблице, просто недостижимы через `get()`. Итератор `entrySet()` их видит и возвращает, потому что обходит все бакеты напрямую, без пересчёта хэша. Это создаёт парадокс: `map.size() == 1`, но `map.get(key) == null` при этом `map.entrySet().iterator().next().getValue()` вернёт значение.

## 14. Какие требования к ключу HashMap?

Формальных требований два: класс ключа должен корректно реализовывать `equals()` и `hashCode()` в соответствии с контрактом. Больше HashMap ничего не требует — нет интерфейса `Hashable`, нет проверок при вставке.

На практике хороший ключ для HashMap должен удовлетворять более широкому набору свойств. Стабильность hashCode: значение не должно меняться пока объект используется как ключ — иначе запись «потеряется». Иммутабельность: лучший способ гарантировать стабильность hashCode. Качественное распределение: hashCode должен распределять объекты равномерно, не давать слишком много коллизий. Быстрая реализация `equals()/hashCode()`: они вызываются при каждой операции, поэтому медленная реализация напрямую влияет на производительность.

`String` является эталонным ключом: иммутабельна, кэширует `hashCode` (вычисляется лениво и сохраняется в поле `hash`), имеет хорошо распределённый хэш.

❗ На собесе: «можно ли `null` использовать как ключ?» — да, HashMap поддерживает ровно один `null`-ключ, он хранится в бакете 0. `ConcurrentHashMap` и `Hashtable` не допускают `null`-ключей.

**→ Уточняющий вопрос:** Почему `String` кэширует свой hashCode, и когда именно он вычисляется?

**↳ Ответ:** `String` иммутабельна — её содержимое никогда не меняется, поэтому хэш вычисляется один раз и безопасно кэшируется. Вычисление происходит при первом вызове `hashCode()` — лениво. Кэш хранится в поле `private int hash`. Если вычисленный хэш оказался 0 (исключительно редко), он не кэшируется и пересчитывается каждый раз (это исправлено в Java 7 введением флага `hashIsZero`). Вывод: при частом использовании строк как ключей в HashMap второй и последующие вызовы `hashCode()` — бесплатные.

## 15. Почему String часто используется как ключ в HashMap?

`String` обладает всеми свойствами идеального ключа HashMap. Во-первых, она иммутабельна: после создания объект нельзя изменить, значит hashCode никогда не изменится за время жизни объекта в map. Во-вторых, `String` переопределяет `equals()` и `hashCode()` корректно и в соответствии с контрактом — это гарантировано спецификацией JDK.

В-третьих, `String` кэширует свой hashCode: при первом вызове `hashCode()` значение вычисляется и сохраняется в приватном поле `private int hash`. При последующих вызовах возвращается кэшированное значение (при условии, что оно не ноль). Это делает повторные операции `get()`/`put()` с одним строковым ключом очень дешёвыми.

Алгоритм `String.hashCode()` — polynomial rolling hash: `s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]`. Число 31 выбрано потому, что оно нечётное простое, и компилятор может заменить `31*i` на `(i << 5) - i` — битовый сдвиг без умножения.

```java
String key = "example";
// Первый вызов — вычисляется хэш и кэшируется
int h1 = key.hashCode(); // вычисление
int h2 = key.hashCode(); // возврат из кэша — O(1)

// String иммутабельна: новый объект, а не изменение старого
String modified = key + "!"; // создаётся новый String
```

❗ На собесе: «что значит "String кэширует hashCode"?» — поле `hash` в классе `String` инициализируется 0 (значение по умолчанию), при первом вызове `hashCode()` вычисляется и сохраняется. Если хэш вычислился равным 0 — он не кэшируется и пересчитывается каждый раз (крайне редкий случай, исправлен через отдельный флаг в Java 7+).

**→ Уточняющий вопрос:** Что произойдёт, если у двух разных строк совпал hashCode (коллизия)? Как HashMap их различит?

**↳ Ответ:** Обе строки попадут в один бакет. При поиске HashMap пройдёт по цепочке бакета и для каждого узла вызовет `String.equals()` — посимвольное сравнение. `equals()` вернёт `false` для разных строк, и поиск продолжится до нужного узла. Коллизия не нарушает корректность, только чуть снижает скорость. Пример: `"Aa"` и `"BB"` имеют одинаковый hashCode в Java.

## 16. Что такое load factor в HashMap?

Load factor (коэффициент загрузки) — это пороговое соотношение `size / capacity`, при превышении которого HashMap выполняет resize. По умолчанию равен `0.75`. Это означает: при capacity=16 и load factor=0.75 resize произойдёт когда будет добавлен 13-й элемент (`16 * 0.75 = 12`, порог 12 — resize при добавлении 13-го).

Load factor — это баланс между памятью и производительностью. Высокий load factor (например, 0.9) экономит память — таблица будет заполнена плотнее, но больше коллизий, медленнее поиск. Низкий load factor (например, 0.5) снижает коллизии и ускоряет поиск, но таблица занимает вдвое больше памяти.

Значение 0.75 выбрано математически: при равномерном распределении хэшей и load factor 0.75 вероятность коллизии в каждом бакете достаточно мала. Это оптимум между памятью и временем доступа для большинства сценариев.

```java
// Создание HashMap с явным load factor
Map<String, Integer> map = new HashMap<>(16, 0.75f); // default
Map<String, Integer> memoryOptimized = new HashMap<>(16, 0.9f); // больше элементов до resize
Map<String, Integer> speedOptimized = new HashMap<>(64, 0.5f);  // меньше коллизий
```

❗ На собесе: «при каком размере HashMap 16 будет resize?» — при добавлении **13-го** элемента (не 12-го!). Порог `threshold = capacity * loadFactor = 16 * 0.75 = 12`. Когда size превышает threshold, запускается resize.

**→ Уточняющий вопрос:** Можно ли задать load factor > 1.0? Что это будет означать?

**↳ Ответ:** Да, технически можно — `new HashMap<>(16, 1.5f)`. Это означает: resize произойдёт только когда число элементов превысит `capacity * 1.5`, то есть в таблице будет больше элементов чем бакетов. Коллизий будет много, производительность упадёт. Это оправдано лишь в крайне memory-sensitive сценариях, где важна компактность и медленный поиск допустим. На практике никогда не используется.

## 17. Что такое capacity в HashMap?

Capacity — это размер внутреннего массива бакетов `Node[] table`. По умолчанию начальный capacity равен 16 (`DEFAULT_INITIAL_CAPACITY`). Capacity всегда является степенью двойки — это гарантируется конструкторами (если передать `new HashMap(15)`, реальный capacity будет 16).

Capacity не равно числу элементов в map. В map с capacity=16 может быть 0, 5 или 12 элементов. Реальное количество элементов хранится в поле `size`. Порог для resize — `threshold = capacity * loadFactor`. При превышении threshold capacity удваивается (`capacity <<= 1`).

При создании `HashMap` с конструктором `new HashMap<>(initialCapacity)` таблица всё равно не выделяется немедленно — это происходит при первом `put()`. Переданный `initialCapacity` используется для вычисления начального threshold.

```java
Map<String, Integer> map = new HashMap<>();
// Реальный capacity = 16 (DEFAULT_INITIAL_CAPACITY)
// Порог resize = 12 (16 * 0.75)

Map<String, Integer> custom = new HashMap<>(100);
// JDK округлит до ближайшей степени двойки >= 100 → capacity = 128
// Порог resize = 96 (128 * 0.75)
```

❗ На собесе: «чему равен capacity по умолчанию?» — 16. Важное уточнение: `new HashMap(15)` даст capacity 16, а не 15, потому что HashMap всегда округляет до степени двойки вверх. Ещё деталь: при `new HashMap(0)` — capacity будет 1.

**→ Уточняющий вопрос:** Что произойдёт, если создать `new HashMap(1_000_000)` и сразу добавить один элемент — когда будет выделен массив на миллион элементов?

**↳ Ответ:** Массив выделится при первом `put()`. JDK округлит 1 000 000 до ближайшей степени двойки ≥ 1 000 000 = 1 048 576. Это ~8 МБ для массива ссылок (8 байт × 1M). Выделяется сразу при первом `put()`, не при конструкторе — конструктор только сохраняет `initialCapacity` для будущего использования. Если вы добавите 1 элемент и забудете — 8 МБ памяти будут заняты впустую.

## 18. Когда происходит rehashing в HashMap?

Rehashing (пересхэширование, resize) происходит когда `size` превышает `threshold = capacity * loadFactor`. Точнее: после добавления нового элемента, если `++size > threshold`, запускается `resize()`. По умолчанию это происходит при добавлении 13-го элемента в пустую HashMap (capacity=16, threshold=12).

Также resize может быть вызван принудительно при treeification если capacity < 64 — HashMap предпочитает увеличить таблицу вместо создания дерева в бакете.

После resize threshold обновляется: `newThreshold = newCapacity * loadFactor`. Для capacity=32 и loadFactor=0.75 новый threshold = 24. Следующий resize произойдёт при добавлении 25-го элемента.

```java
// Чтобы избежать нежелательного rehashing при известном числе элементов:
int expectedSize = 100;
// Формула: initialCapacity = expectedSize / loadFactor + 1
Map<String, String> map = new HashMap<>((int)(expectedSize / 0.75) + 1);
// Это гарантирует, что 100 элементов не вызовут resize
```

❗ На собесе: «когда происходит rehashing?» — недостаточно сказать «когда map заполняется». Нужна формула: `size > capacity * loadFactor`. И важно уточнить: проверка происходит **после** добавления каждого элемента.

**→ Уточняющий вопрос:** Rehashing — это дорогая операция. Как можно её полностью избежать, если заранее известно количество элементов?

**↳ Ответ:** Задать `initialCapacity = (int)(expectedSize / loadFactor) + 1`. Для 100 элементов: `(int)(100 / 0.75) + 1 = 134`. JDK округлит до 256 — threshold = 192, что больше 100, resize не произойдёт. Guava предоставляет `Maps.newHashMapWithExpectedSize(n)` с этой же логикой. Правило: если известен размер заранее — всегда указывай `initialCapacity`, это O(1) экономия вместо нескольких O(n) resize.

## 19. Что происходит при rehashing?

При rehashing HashMap создаёт новый массив вдвое большего размера (`newCap = oldCap << 1`). Затем каждый элемент из старого массива нужно поместить в новый. Наивный подход — пересчитать `hash % newCap` для каждого элемента. Но HashMap поступает умнее.

Поскольку capacity всегда степень двойки, удвоение capacity означает: новый индекс элемента либо равен старому, либо равен `oldIndex + oldCapacity`. Это определяется проверкой одного бита: `(oldHash & oldCapacity) == 0` — тогда индекс не меняется, иначе прибавляется `oldCapacity`. Это позволяет разбить каждый бакет на две группы за O(n/buckets) без полного пересчёта хэшей.

Rehashing — операция O(n), и она выполняется в одном потоке (HashMap не является потокобезопасной). При конкурентном rehashing в двух потоках в Java 7 был известный баг с бесконечным циклом (race condition при перестановке указателей в связном списке). В Java 8 алгоритм resize изменён и этот баг исправлен, но HashMap всё равно небезопасна в многопоточной среде.

❗ На собесе: «что такое O(n) при rehashing?» — нужно понять, что в среднем эта операция амортизируется до O(1) на один `put()`, потому что происходит логарифмически редко по мере роста map. Но в конкретный момент времени она занимает O(n) — это может быть проблемой для систем с требованиями к latency.

**→ Уточняющий вопрос:** Если rehashing — это O(n), то как можно утверждать что `put()` в HashMap работает за амортизированное O(1)?

**↳ Ответ:** Амортизация — суммарная стоимость всех операций делённая на их количество. Resize с capacity N происходит после N*0.75 вставок. Стоимость resize — O(N). На одну вставку приходится O(N)/O(N) = O(1) «доля» стоимости resize. Между resize'ами каждая вставка — O(1). Итого: суммарно на n вставок тратится O(n) работы → амортизированно O(1) на вставку, хотя конкретная вставка при resize стоит O(n).

## 20. Какова временная сложность операций get() и put() в HashMap?

В среднем — O(1). Это достигается за счёт прямого доступа по индексу массива (вычисленному из hashCode) и отсутствия коллизий или малого их числа. Вычисление индекса, переход к бакету и сравнение одного-двух элементов — константное количество операций.

В худшем случае (до Java 8) — O(n), когда все элементы попали в один бакет и образовали список. Начиная с Java 8 — O(log n) при treeification переполненного бакета. Формально это и есть гарантированный худший случай в современном JDK.

При rehashing отдельный `put()` занимает O(n), но амортизированно по всем вставкам сложность остаётся O(1): rehashing происходит экспоненциально редко по мере роста map, и суммарная стоимость всех rehashing делённая на все вставки стремится к константе.

```
Сводная таблица:
Операция    | Средний случай | Худший (Java 8+) | Худший (Java 7-)
get(key)    | O(1)          | O(log n)         | O(n)
put(key)    | O(1) аморт.   | O(log n)         | O(n)
remove(key) | O(1)          | O(log n)         | O(n)
contains()  | O(1)          | O(log n)         | O(n)
```

❗ На собесе: «HashMap это O(1) — можно ли на это всегда рассчитывать?» — нет. O(1) — это **среднее** при предположении равномерного распределения хэшей. Атака хэш-коллизиями (HashDoS) намеренно делает все ключи коллидирующими → деградация до O(n) или O(log n).

**→ Уточняющий вопрос:** Что такое амортизированная O(1) и чем она отличается от O(1) в строгом смысле?

**↳ Ответ:** O(1) в строгом смысле — каждая отдельная операция ограничена константой. Амортизированное O(1) — среднее по последовательности операций. Конкретная операция может стоить O(n), но это редко, и суммарно n операций занимают O(n) → среднее O(1). Разница важна для latency-sensitive систем: `put()` обычно быстрый, но раз в while выполняется resize — пауза O(n) в реальном времени. Для систем с SLA на p99-латентность это проблема.

## 21. Когда временная сложность может стать O(n)?

До Java 8: когда все элементы попадали в один бакет (деградация списка), операции в этом бакете занимали O(n). Это мог спровоцировать злоумышленник, зная алгоритм hashCode строк в Java, — подбирались строки с одинаковым хэшем (HashDoS-атака).

С Java 8: список в бакете при 8+ элементах становится деревом, что ограничивает поиск O(log n). Чистого O(n) для базовых операций больше нет. Однако сам resize (rehashing) — это O(n) операция, хотя и редкая.

На практике «деградация производительности» HashMap может произойти не только при атаке, но и при неудачном hashCode: например, если ключи — объекты `new Point(x, y)` с реализацией `hashCode() { return x + y; }`, многие точки дадут одинаковый хэш.

```java
// Плохой hashCode — вызывает деградацию
class BadPoint {
    int x, y;
    BadPoint(int x, int y) { this.x = x; this.y = y; }

    @Override
    public int hashCode() {
        return x + y; // много коллизий для точек на диагоналях (0,0), (1,-1), (2,-2)...
    }
}

// Хороший hashCode — минимум коллизий
    @Override
    public int hashCode() {
        return Objects.hash(x, y); // учитывает порядок, уникальный для каждой комбинации
    }
```

❗ На собесе: «Java 8 полностью решила проблему HashDoS?» — нет. Деревья снизили худший случай с O(n) до O(log n), но не устранили его. Для `String` Java также рандомизирует hashCode при каждом запуске JVM (`-XX:+UseStringDeduplication`, `JAVA_TOOL_OPTIONS=-Djava.lang.String.hash32`). Плюс деревья в бакете требуют, чтобы ключи реализовывали `Comparable` — иначе treeification не работает корректно.

**→ Уточняющий вопрос:** Что произойдёт при treeification, если ключи не реализуют `Comparable`?

**↳ Ответ:** HashMap не требует `Comparable` жёстко. Если класс ключей не реализует `Comparable`, при treeification JDK использует `System.identityHashCode()` как тай-брейкер для упорядочивания узлов дерева. Это детерминировано в рамках одного запуска JVM (но не между запусками). Дерево работает корректно — хуже только потенциальное качество балансировки. Ключи всё равно находятся через `equals()`, дерево используется для ограничения глубины поиска.

## 22. Как работает HashMap в многопоточной среде?

`HashMap` не является потокобезопасной. Конкурентная запись может привести к потере данных: если два потока одновременно выполняют `put()` и оба попадают в один бакет, один из них может затереть результат другого. Это «тихая» потеря данных без исключений.

При конкурентном resize в Java 7 мог возникнуть бесконечный цикл: два потока одновременно перестраивали связный список, и из-за race condition в операциях с указателями список мог свернуться в цикл. Итератор по такому HashMap зависал навсегда. В Java 8 алгоритм resize изменён (два отдельных списка вместо инвертирования одного), бесконечный цикл исправлен, но race condition при конкурентной записи всё равно приводит к потере данных.

Альтернативы: `ConcurrentHashMap` — для конкурентных записей; `Collections.synchronizedMap(new HashMap<>())` — для устаревшего кода (полная синхронизация, хуже производительность); `Hashtable` — legacy, не рекомендуется.

```java
// НЕПРАВИЛЬНО: HashMap в многопоточной среде
Map<String, Integer> unsafeMap = new HashMap<>();
// Конкурентные put → потеря данных, иногда ArrayIndexOutOfBoundsException

// ПРАВИЛЬНО: ConcurrentHashMap
Map<String, Integer> safeMap = new ConcurrentHashMap<>();
// Потокобезопасные операции, хорошая производительность

// ПРАВИЛЬНО (устаревший подход): synchronized wrapper
Map<String, Integer> syncMap = Collections.synchronizedMap(new HashMap<>());
// Нужна внешняя синхронизация при итерации!
synchronized (syncMap) {
    for (Map.Entry<String, Integer> e : syncMap.entrySet()) { ... }
}
```

❗ На собесе: «можно ли использовать `Collections.synchronizedMap()` вместо ConcurrentHashMap?» — можно, но с важной оговоркой: итерация по синхронизированной map требует явного `synchronized`-блока, иначе `ConcurrentModificationException`. `ConcurrentHashMap` не требует этого.

**→ Уточняющий вопрос:** Чем конкретно `ConcurrentHashMap` лучше `Collections.synchronizedMap()` с точки зрения производительности?

**↳ Ответ:** `synchronizedMap` синхронизирует каждый метод целиком по одному объекту — только один поток работает с map в любой момент. `ConcurrentHashMap` блокирует только отдельный бакет при записи (fine-grained locking), а чтения (`get()`) вообще lock-free через `volatile`. При 16 потоках: `synchronizedMap` — 1 поток пишет, 15 ждут; `ConcurrentHashMap` — до 16 потоков могут писать в разные бакеты одновременно. Разница в throughput при высокой нагрузке — на порядки.

## 23. Что такое ConcurrentHashMap и чем он отличается от HashMap?

`ConcurrentHashMap` — потокобезопасная реализация `Map`, оптимизированная для высококонкурентного доступа. В отличие от `Hashtable`, который синхронизирует каждый метод целиком, `ConcurrentHashMap` использует более тонкую стратегию.

В Java 8+ `ConcurrentHashMap` использует: CAS (Compare-And-Swap) для добавления первого элемента в пустой бакет — без блокировок; `synchronized` блок только на уровне отдельного бакета (головного узла) для остальных операций записи; `volatile` поля для гарантии видимости при чтении — операции `get()` полностью lock-free. Это позволяет нескольким потокам параллельно писать в разные бакеты без блокировок друг друга.

Ключевые отличия от `HashMap`: потокобезопасен; не допускает `null` ключи и `null` значения (исключение: `NullPointerException`); `size()` возвращает приближённое значение (используется `LongAdder` внутри); итераторы не бросают `ConcurrentModificationException`, но отражают «слабо консистентный» снимок данных.

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// Атомарные операции — без гонок
map.putIfAbsent("key", 1);
map.computeIfAbsent("key", k -> expensiveCompute(k));
map.merge("counter", 1, Integer::sum); // атомарный инкремент

// Нельзя хранить null:
map.put(null, 1);   // NullPointerException
map.put("k", null); // NullPointerException
```

❗ На собесе: «почему ConcurrentHashMap не допускает null?» — потому что при конкурентном доступе невозможно различить «ключ отсутствует» и «значение null» без дополнительной синхронизации. В однопоточном `HashMap` это решается через `containsKey()`, но в конкурентной среде между `containsKey()` и `get()` другой поток может удалить запись.

**→ Уточняющий вопрос:** Что такое «слабая консистентность» (weakly consistent) итераторов ConcurrentHashMap?

**↳ Ответ:** Слабая консистентность означает: итератор гарантированно покажет все элементы, которые существовали в момент создания итератора. Элементы, добавленные или удалённые после создания итератора, могут или не могут быть показаны — гарантий нет. При этом итератор никогда не бросает `ConcurrentModificationException`. Это противоположность fail-fast итераторам `HashMap`, которые бросают CME при любом структурном изменении во время итерации.

## 24. Можно ли хранить null ключ в HashMap?

Да, `HashMap` поддерживает ровно один `null`-ключ. Для него не вызывается `hashCode()` (что было бы `NullPointerException`) — вместо этого HashMap помещает его напрямую в бакет с индексом 0 (`hash(null) = 0`).

Это исключительная обработка: в методе `hash()` первым делом идёт проверка `(key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)`. При `put(null, value)` значение кладётся в бакет 0 с захардкоженным хэшем 0. При `get(null)` аналогично — ищет в бакете 0.

Важно: `null` ключ работает только в `HashMap`. `ConcurrentHashMap`, `Hashtable`, `TreeMap` (без `Comparator`) не допускают `null` ключей и бросят `NullPointerException`.

```java
Map<String, String> map = new HashMap<>();
map.put(null, "null_value");
map.put(null, "overwritten"); // перезапишет предыдущее
System.out.println(map.get(null)); // "overwritten"
System.out.println(map.containsKey(null)); // true
System.out.println(map.size()); // 1 — только одна запись с null-ключом
```

❗ На собесе: «сколько null-ключей можно хранить в HashMap?» — ровно один. Второй `put(null, ...)` перезапишет первый, как обычный duplicate key. Также: `get(null)` возвращает `null` если ключа нет — неотличимо от случая, когда значение равно `null`. Используй `containsKey(null)`.

**→ Уточняющий вопрос:** Как HashMap обрабатывает `get(null)` — каким образом он находит значение без вызова `hashCode()` на null?

**↳ Ответ:** Метод `hash(key)` сразу проверяет `key == null` и возвращает 0 без вызова `hashCode()`. Таким образом, `null`-ключ всегда маппится на хэш 0 → бакет 0. При `get(null)` HashMap идёт в бакет 0 и ищет узел с `key == null`. Это hard-coded поведение, исключение из общего потока. Поэтому `null` как ключ работает корректно — просто через специальный путь.

## 25. Можно ли хранить null значение в HashMap?

Да, в `HashMap` можно хранить сколько угодно `null`-значений — ограничений нет. Разные ключи могут маппиться на `null`. Это особенность дизайна `HashMap`, отличающая его от `ConcurrentHashMap` и `Hashtable`.

Главная проблема: метод `get(key)` возвращает `null` в двух принципиально разных случаях — ключ отсутствует в map, или ключ присутствует но его значение равно `null`. Различить эти случаи без дополнительного вызова нельзя.

Это особенно коварно в случаях, когда результат `get()` используется для принятия решений — например, как флаг присутствия ключа. Правильное решение: `containsKey(key)` для проверки наличия, `getOrDefault(key, defaultValue)` или `computeIfAbsent()` для работы с дефолтными значениями.

```java
Map<String, String> config = new HashMap<>();
config.put("host", "localhost");
config.put("debug", null); // явно задан null

// Опасный паттерн:
String host = config.get("timeout");   // null — ключа нет
String debug = config.get("debug");    // null — значение null
// Как отличить? Никак без containsKey!

// Правильный паттерн:
if (config.containsKey("timeout")) {
    // использовать значение
}
String val = config.getOrDefault("timeout", "30s"); // "30s"
```

❗ На собесе: «как отличить "ключ не найден" от "значение равно null" при вызове get()?» — это типичный практический вопрос. Ответ: `containsKey()`, `getOrDefault()`, или возврат к дизайну (лучше не хранить null значения в map — предпочтительнее `Optional` или явный sentinel объект).

**→ Уточняющий вопрос:** Предложи дизайн, при котором `null`-значения в Map не приводят к этой двусмысленности.

**↳ Ответ:** Три подхода: 1) `Map<K, Optional<V>>` — явно разделяет «значение есть, но оно null» и «ключа нет»; 2) sentinel-объект — специальный статический `ABSENT = new Object()` вместо `null`; 3) запретить `null`-значения и использовать `getOrDefault(key, defaultValue)` — `null` = «отсутствие», дефолт явный. Самый чистый: не хранить `null`, переосмыслить семантику. `Optional` в map — злоупотребление (Guava и `Optional` авторы рекомендуют против этого), но на практике встречается.

## 26. В чём разница между HashMap и Hashtable?

`Hashtable` — один из первых классов Java (с версии 1.0), созданный до введения Collections Framework. Каждый его публичный метод объявлен `synchronized`, что обеспечивает потокобезопасность, но создаёт «бутылочное горло»: в каждый момент только один поток может работать с таблицей.

`HashMap` появился в Java 1.2 как часть Collections Framework. Он не синхронизирован, работает быстрее в однопоточном коде, допускает один `null`-ключ и любое количество `null`-значений. `Hashtable` не допускает ни `null`-ключей, ни `null`-значений — бросает `NullPointerException`.

Сегодня `Hashtable` считается устаревшим (legacy). Для многопоточного кода используется `ConcurrentHashMap` — он обеспечивает потокобезопасность с гораздо лучшей производительностью через сегментированные блокировки и CAS. Для однопоточного — `HashMap`.

```
Сравнительная таблица:
Свойство             | HashMap       | Hashtable    | ConcurrentHashMap
Потокобезопасность   | Нет           | Да (sync)    | Да (fine-grained)
null ключи           | 1             | Нет          | Нет
null значения        | Любое кол-во  | Нет          | Нет
Производительность   | Высокая       | Низкая       | Высокая
Наследование         | AbstractMap   | Dictionary   | AbstractMap
Статус               | Актуален      | Legacy       | Актуален
```

❗ На собесе: «можно ли использовать Hashtable в новом коде?» — нет, это anti-pattern. Даже если нужна синхронизация, `ConcurrentHashMap` всегда предпочтительнее за счёт лучшей масштабируемости. `Collections.synchronizedMap()` — второй вариант, но хуже ConcurrentHashMap.

**→ Уточняющий вопрос:** Чем синхронизация на уровне всего объекта (как в Hashtable) хуже сегментированной блокировки (как в ConcurrentHashMap)?

**↳ Ответ:** Синхронизация на всём объекте — это монопольный доступ: только один поток может читать или писать в любой момент. При N потоках — N-1 ждут, масштабируемость O(1/N). Сегментированная блокировка разделяет таблицу на независимые бакеты: потоки, пишущие в разные бакеты, не блокируют друг друга. Чтение вообще lock-free. При N потоках и M бакетах — почти линейное масштабирование до min(N, M) потоков. На 16-ядерном сервере разница в throughput — 10-50x.

## 27. Что такое WeakHashMap?

`WeakHashMap` — это реализация `Map`, в которой ключи хранятся через слабые ссылки (`WeakReference`). Если на объект-ключ нет сильных ссылок нигде в программе (кроме самой WeakHashMap), GC может удалить его, и соответствующая запись автоматически исчезнет из map при следующей очистке `ReferenceQueue`.

Это делает `WeakHashMap` идеальной для кэшей: когда клиент «забывает» об объекте (теряет сильную ссылку), кэш автоматически освобождает память без ручного управления. Это предотвращает утечки памяти, характерные для обычного `HashMap`-кэша.

Важные ограничения: `WeakHashMap` не потокобезопасна; запись может исчезнуть в любой момент между операциями (GC работает асинхронно); не рекомендуется использовать иммутабельные объекты (строки-интерны, примитивные обёртки) как ключи — они могут не удаляться GC, если JVM их кэширует.

```java
WeakHashMap<Object, String> cache = new WeakHashMap<>();

Object key = new Object();  // сильная ссылка
cache.put(key, "cached_value");
System.out.println(cache.size()); // 1

key = null;  // убираем сильную ссылку
System.gc(); // подсказываем GC (не гарантия!)
Thread.sleep(100); // ждём...

System.out.println(cache.size()); // 0 — запись удалена GC

// Реальный use case: кэш вычислений без утечек
WeakHashMap<SomeExpensiveObject, ComputedResult> resultCache = new WeakHashMap<>();
```

❗ На собесе: «почему нельзя использовать String literals как ключи в WeakHashMap?» — строки-интерны (из пула строк) имеют сильные ссылки от JVM, поэтому никогда не будут собраны GC. `WeakHashMap` с такими ключами поведёт себя как обычный `HashMap` — не будет очищаться. Правильный ключ — `new String("value")` (не из пула).

**→ Уточняющий вопрос:** Чем `WeakHashMap` отличается от `SoftReference`-кэша? Когда использовать `WeakReference`, а когда `SoftReference`?

**↳ Ответ:** `WeakReference` — GC собирает объект при первой же возможности, как только нет сильных ссылок. `SoftReference` — GC собирает только при нехватке памяти (перед OutOfMemoryError). Правило выбора: `WeakReference` = «живёт пока кто-то другой использует объект» — идеально для кэша связанных метаданных (объект умер → кэш умер). `SoftReference` = «лучше держать в памяти, но можно выбросить при давлении» — идеально для кэша вычислений, которые дорого пересчитывать.

## 28. Как правильно выбрать начальный capacity для HashMap?

Если заранее известно примерное количество элементов N, правильная формула: `initialCapacity = (int)(N / loadFactor) + 1`. При loadFactor 0.75 это: `(int)(N / 0.75) + 1`. Это гарантирует, что вставка N элементов не вызовет ни одного resize.

Например, для 100 элементов: `(int)(100 / 0.75) + 1 = 134`. HashMap округлит до 256 (ближайшая степень двойки ≥ 134). Порог resize = `256 * 0.75 = 192` > 100 → rehashing не произойдёт.

Если задать `new HashMap<>(100)` напрямую — capacity будет 128, threshold = 96. При вставке 97-го элемента произойдёт resize! Это контринтуитивно: создал map «на 100 элементов», а resize случается уже на 97-м.

```java
// НЕПРАВИЛЬНО — вызовет resize при ~75 элементах
Map<String, String> wrong = new HashMap<>(100);

// ПРАВИЛЬНО — не вызовет resize при вставке 100 элементов
int expectedSize = 100;
int initialCapacity = (int)(expectedSize / 0.75) + 1; // = 134
Map<String, String> correct = new HashMap<>(initialCapacity);

// Guava предоставляет готовый метод:
// Maps.newHashMapWithExpectedSize(100) — делает это автоматически
```

❗ На собесе: «создал `new HashMap<>(100)` и добавляю 100 элементов — будет ли resize?» — да, будет! Порог = 75, resize произойдёт при 76-м элементе. Это частая ошибка в продакшн-коде, которая создаёт ненужные O(n) операции при заполнении map.

**→ Уточняющий вопрос:** Что хуже с точки зрения производительности — слишком маленький initial capacity или слишком большой?

**↳ Ответ:** Слишком маленький хуже для throughput: он вызывает несколько resize'ов (O(n) каждый), что замедляет вставку и создаёт GC-давление от временных массивов. Слишком большой — лишь тратит память (выделяется пустой массив), но не влияет на время операций. Правило: лучше передать `initialCapacity` с запасом (например, ×2 от ожидаемого), чем допустить resize. Память дешевле, чем latency-спайк от O(n) rehashing.

## 29. Что лучше: ArrayList или HashMap для хранения пар ключ-значение?

Это вопрос о правильном выборе структуры данных под задачу. Если нужен доступ по ключу — однозначно `HashMap`: поиск за O(1) против O(n) у `ArrayList`. Если хранить пары как `List<Map.Entry<K,V>>` или `List<Object[]>` — поиск потребует полного перебора.

`ArrayList` оправдан если: порядок важен и нет поиска по ключу; нужен доступ по позиции (индексу); необходима экономия памяти (каждая запись HashMap несёт накладные расходы на `Node` объект с хэшем и двумя ссылками).

Накладные расходы `HashMap` на один элемент: объект `Node` (16 байт заголовка + 4*8 = 32 байта полей на 64-битной JVM) + запись в массиве. Для небольшого числа элементов (например, < 10 пар) `ArrayList<AbstractMap.SimpleEntry>` может быть не хуже по производительности, потому что паттерн доступа последовательный и хорошо кэшируется CPU.

```java
// HashMap — O(1) поиск, порядок не гарантирован
Map<String, Integer> scores = new HashMap<>();
scores.put("Alice", 95);
scores.get("Alice"); // O(1)

// LinkedHashMap — O(1) поиск + порядок вставки
Map<String, Integer> orderedScores = new LinkedHashMap<>();

// TreeMap — O(log n) поиск + сортировка по ключу
Map<String, Integer> sortedScores = new TreeMap<>();

// ArrayList пар — O(n) поиск, но лучше для итерации, меньше памяти
List<Map.Entry<String, Integer>> list = new ArrayList<>();
list.stream().filter(e -> e.getKey().equals("Alice")).findFirst(); // O(n)
```

❗ На собесе: «когда HashMap хуже ArrayList?» — когда важна память (много мелких записей), нужен порядок вставки (используй `LinkedHashMap`) или сортировка (используй `TreeMap`). Ещё: если ключи — целые числа в небольшом диапазоне, обычный массив `int[]` быстрее любой Map за счёт лучшей локальности кэша.

**→ Уточняющий вопрос:** Чем `LinkedHashMap` отличается от `HashMap`, и когда его использовать вместо `TreeMap`?

**↳ Ответ:** `LinkedHashMap` = `HashMap` + двусвязный список через все узлы, сохраняющий порядок вставки (или LRU-порядок при `accessOrder=true`). Операции O(1) как у `HashMap`. `TreeMap` = красно-чёрное дерево, сортировка по ключу, операции O(log n). Правило: если нужен порядок вставки или LRU — `LinkedHashMap`. Если нужна сортировка по ключу или range-запросы (`headMap`, `tailMap`, `subMap`) — `TreeMap`. Цена `LinkedHashMap` — лишние 2 ссылки на узел (prev/next), ~16 байт на элемент.
