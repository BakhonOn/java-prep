## 1. Как работает String Pool?

String Pool — это специальная хэш-таблица внутри кучи (heap), где JVM хранит уникальные экземпляры строковых литералов. Когда JVM встречает строковый литерал `"hello"`, она сначала ищет строку с таким содержимым в пуле. Если находит — возвращает ссылку на существующий объект; если нет — создаёт новый объект и кладёт его в пул.

Механизм работает благодаря иммутабельности String: раз строку нельзя изменить, два клиента могут безопасно держать ссылку на один объект. Пул реализован через нативный `StringTable` в JVM — это hashtable фиксированного (но настраиваемого) размера. С Java 7 пул живёт в основном heap и участвует в GC.

Практический пример: при парсинге CSV-файла с миллионом строк, где поле «страна» принимает 200 уникальных значений, интернирование (`country.intern()`) позволяет хранить 200 объектов вместо миллиона дублей — экономия сотен мегабайт.

```java
String a = "Russia";
String b = "Russia";
String c = new String("Russia");

System.out.println(a == b);          // true  — один объект из пула
System.out.println(a == c);          // false — c в heap, не в пуле
System.out.println(a == c.intern()); // true  — intern() вернул объект из пула
```

❗ Ловушка собеса: кандидаты путают String Pool с PermGen. До Java 7 пул был в PermGen (фиксированный, не собирался GC). С Java 7+ — в heap. Размер пула настраивается флагом `-XX:StringTableSize` (по умолчанию 65536 бакетов до Java 11, потом больше).

**→ Уточняющий вопрос:** Что произойдёт с объектами в String Pool при работе GC — они когда-нибудь собираются?

**↳ Ответ:** С Java 7 пул находится в heap, и StringTable хранит WeakReference на пуловые объекты — GC может их собрать, если нет сильных ссылок из кода. Строковые литералы классов не собираются, пока жив их ClassLoader (сильная ссылка из константного пула класса). Динамически интернированные строки без других ссылок — собираются при следующем GC-цикле. Правило: пул не утекает сам по себе, проблема только при массовом `intern()` уникальных данных — они накапливаются до следующего GC.

## 2. В чём разница между созданием String через литерал и через new?

Литерал `"abc"` — компилятор помещает строку в константный пул класса (`.class` файл), а при загрузке класса JVM интернирует её в String Pool. При каждом обращении к литералу возвращается один и тот же объект из пула. `new String("abc")` всегда создаёт новый объект в heap, даже если `"abc"` уже есть в пуле; при этом сам литерал `"abc"` тоже попадает в пул как побочный эффект.

Разница важна при сравнении через `==` и при анализе расхода памяти. В реальном коде: если конфигурационный файл читается через `Properties.load()`, все строки-значения создаются через `new` (внутри парсера) и не попадают в пул автоматически.

```java
String literal = "hello";        // из пула
String newStr  = new String("hello"); // новый объект в heap

System.out.println(literal == newStr);          // false
System.out.println(literal == newStr.intern()); // true
System.out.println(literal.equals(newStr));     // true — по содержимому
```

❗ Ловушка собеса: `new String("abc")` создаёт ДВА объекта, если `"abc"` ещё не в пуле — один в пуле (от литерала-аргумента), один в heap (от `new`). Многие отвечают «один».

**→ Уточняющий вопрос:** Если сделать `new String("abc").intern()`, какой объект вернётся — тот, что в heap, или новый из пула?

**↳ Ответ:** `intern()` проверяет пул: если `"abc"` уже там есть — возвращает *существующий* пуловый объект (не heap-объект). Heap-объект становится кандидатом на GC, если нет других ссылок на него. Правило: `new String("abc").intern() == "abc"` всегда `true` — intern возвращает пуловый объект, и всегда именно его; использовать `new String(...)` + `intern()` бессмысленно — можно просто написать `"abc"`.

## 3. Когда стоит использовать intern()?

`intern()` помещает строку в String Pool (или возвращает уже существующий объект с тем же содержимым). Оправдан, когда приложение обрабатывает большой поток данных с высоким процентом повторяющихся строк, поступающих из внешнего источника: парсинг логов, чтение CSV, обработка сетевых пакетов с повторяющимися полями (имена хостов, коды статусов, названия полей JSON).

Под капотом `intern()` обращается к нативному `StringTable::intern` в JVM, который вычисляет хэш строки и ищет её в таблице. Это не бесплатно: каждый вызов — нативный переход + хэширование. Если повторяемость данных низкая (каждая строка уникальна), `intern()` только добавляет нагрузку и раздувает пул.

```java
// Парсинг CSV: поле "country" имеет ~200 уникальных значений из 1_000_000 строк
List<Record> records = new ArrayList<>();
try (BufferedReader br = new BufferedReader(new FileReader("data.csv"))) {
    String line;
    while ((line = br.readLine()) != null) {
        String[] parts = line.split(",");
        String country = parts[2].trim().intern(); // экономим память
        records.add(new Record(parts[0], country));
    }
}
```

❗ Ловушка собеса: бесконтрольный `intern()` для уникальных строк (например, UUID или токенов) приведёт к тому, что String Pool разрастётся и вызовет `OutOfMemoryError`. Кроме того, до Java 7 `intern()` был особенно опасен из-за PermGen с фиксированным размером.

**→ Уточняющий вопрос:** Как настроить размер String Pool и как проверить его заполненность на работающем приложении?

**↳ Ответ:** Размер задаётся флагом `-XX:StringTableSize=N` (число бакетов, должно быть простым числом для лучшего распределения). Статистику получают командой `jcmd <pid> VM.stringtable` — выводит число записей, загрузку бакетов, среднюю длину цепочек. Флаг `-XX:+PrintStringTableStatistics` печатает статистику при завершении JVM. Правило: если `Average Bucket Size` > 2-3 → таблица перегружена, нужно увеличить `-XX:StringTableSize` или сократить использование `intern()`.

## 4. Почему String является иммутабельным?

String сделан иммутабельным сознательно по нескольким причинам, которые усиливают друг друга. Во-первых, безопасность: `String` используется в критичных местах — имя класса для загрузки, URL для сетевого соединения, путь к файлу. Если бы строка была мутабельной, код мог бы проверить значение, а другой поток или злонамеренный код успел бы его изменить до фактического использования. Во-вторых, String Pool требует иммутабельности — нельзя безопасно разделять один объект между несколькими клиентами, если кто-то может его изменить.

В-третьих, `hashCode` кэшируется в поле `hash` и вычисляется один раз (при первом обращении). Это делает `String` эффективным ключом в `HashMap`: хэш не меняется, нет нужды пересчитывать при каждой операции. В-четвёртых, иммутабельность = потокобезопасность без синхронизации.

Технически иммутабельность обеспечивается: поле `value` (byte[]) объявлено `private final`, нет сеттеров, конструкторы копируют переданный массив, класс объявлен `final` (нельзя создать мутабельный подкласс).

```java
// hashCode кэшируется — безопасно, т.к. строка не меняется
String key = "user_id";
Map<String, Integer> map = new HashMap<>();
map.put(key, 42);
// если бы String был мутабельным, изменение key сломало бы lookup
```

❗ Ловушка собеса: кандидаты часто называют одну причину (безопасность или пул). Сильный ответ называет все четыре: безопасность, пул, кэширование hashCode, thread-safety.

**→ Уточняющий вопрос:** Раз String иммутабелен и thread-safe, зачем вообще нужен StringBuffer — разве String уже безопасен для многопоточной записи?

**↳ Ответ:** String thread-safe как *значение* — читать его одновременно из нескольких потоков безопасно. Но *сборка* строки через конкатенацию не атомарна: `str += x` — это три шага (создать StringBuilder, append, toString), между которыми другой поток может вмешаться. StringBuffer нужен, когда несколько потоков дописывают в *один общий буфер*. Правило: String безопасен для чтения, но не для совместной сборки; лучше давать каждому потоку свой StringBuilder и объединять результаты в конце.

## 5. Какие последствия иммутабельности String?

Главное последствие — любая «модификация» строки создаёт новый объект. `str.toUpperCase()`, `str.trim()`, `str.replace("a","b")` — все возвращают новые объекты, оригинал не тронут. Это означает, что конкатенация в цикле (`result += item`) квадратично дорога: на каждой итерации создаётся новый объект, предыдущий становится мусором. При 10 000 итераций это 10 000 временных объектов и O(n²) копирований.

Следствие для кода: использовать `StringBuilder` в циклах, `String.join()` или `Collectors.joining()` для агрегации. При формировании SQL-запроса из списка условий — всегда `StringBuilder` или `StringJoiner`.

```java
// Плохо: O(n²), 10_000 временных объектов при n=10_000
String result = "";
for (String item : items) {
    result += item + ",";  // каждый раз новый объект
}

// Хорошо: O(n), один StringBuilder
StringBuilder sb = new StringBuilder();
for (String item : items) {
    sb.append(item).append(',');
}
String result = sb.toString();

// Ещё лучше для join:
String result = String.join(",", items);
```

Второе следствие — позитивное: строку можно безопасно передавать в другие потоки, класть в кэш, использовать как ключ в Map без защитного копирования.

❗ Ловушка собеса: `str += "x"` внутри метода кажется безобидным, но в цикле это катастрофа производительности. Интервьюер часто показывает цикл и просит оценить сложность.

**→ Уточняющий вопрос:** Компилятор Java иногда оптимизирует `+` в StringBuilder автоматически — в каких случаях он это делает, а в каких нет?

**↳ Ответ:** Компилятор оптимизирует одно выражение: `"a" + b + "c"` → одна цепочка `StringBuilder.append()`. Но не переносит StringBuilder через итерации цикла: `str += x` в цикле — каждая итерация создаёт новый StringBuilder, копирует весь `str`, делает append, toString → O(n²). Правило: одно многочленное выражение → компилятор оптимизирует; накопление результата через `+=` в цикле → нет, нужен явный `StringBuilder sb` снаружи цикла.

## 6. Когда использовать StringBuilder, а когда StringBuffer?

`StringBuilder` — для однопоточного кода и для методов, где буфер не утекает наружу. Это 99% случаев использования. `StringBuffer` — когда один и тот же экземпляр буфера действительно разделяется между несколькими потоками и несколько потоков могут вызывать его методы одновременно. На практике такой сценарий крайне редок: обычно каждый поток строит свою строку независимо.

Важно понять: если несколько потоков формируют строки параллельно и потом результаты объединяются — каждый поток использует свой `StringBuilder`, синхронизация не нужна. `StringBuffer` нужен только если разные потоки вызывают `append` на одном объекте одновременно — а такое почти не встречается в корректно написанном коде.

```java
// Правильно: каждый поток строит свою строку через StringBuilder
public String buildQuery(List<String> conditions) {
    StringBuilder sb = new StringBuilder("SELECT * FROM users WHERE ");
    for (int i = 0; i < conditions.size(); i++) {
        if (i > 0) sb.append(" AND ");
        sb.append(conditions.get(i));
    }
    return sb.toString();
}

// StringBuffer нужен только при реальном шаринге:
// StringBuffer shared = new StringBuffer(); // передаётся в несколько потоков
```

❗ Ловушка собеса: многие думают, что раз приложение многопоточное — нужен `StringBuffer`. Нет: локальный `StringBuilder` в методе потокобезопасен по определению (у каждого потока своя копия в стеке). `StringBuffer` оправдан только при явном шаринге экземпляра.

**→ Уточняющий вопрос:** Если `StringBuffer` синхронизирован, означает ли это, что операция «проверить длину и добавить» (`if (sb.length() < 100) sb.append(...)`) атомарна?

**↳ Ответ:** Нет — каждый вызов (`length()` и `append()`) синхронизирован *по отдельности*, но между ними другой поток может успеть вставить свой `append`. Это классическая race condition «check-then-act». Правило: синхронизированный объект гарантирует атомарность *одного вызова*, но не составных операций; для атомарности check-then-act нужен внешний `synchronized (sb) { ... }` блок.

## 7. Почему StringBuffer медленнее StringBuilder?

Все публичные методы `StringBuffer` (append, insert, delete, reverse и т.д.) объявлены как `synchronized`. Это означает, что при каждом вызове JVM должна получить монитор объекта (или убедиться, что он свободен), выполнить операцию, и освободить монитор. Даже в однопоточном коде JIT-компилятор должен обработать эти барьеры — в современных JVM он часто выполняет lock elision (убирает бесполезные блокировки), но это дополнительная нагрузка на JIT и не гарантировано.

Практически: в микробенчмарках `StringBuilder` в однопоточном коде обгоняет `StringBuffer` в 1.5–3 раза. При интенсивном формировании строк (генерация HTML, построение SQL, сериализация JSON без библиотек) разница ощутима.

```java
// Демонстрация разницы (концептуально — реальный бенчмарк через JMH)
long start = System.nanoTime();
StringBuffer sbf = new StringBuffer();
for (int i = 0; i < 100_000; i++) sbf.append("x");
long sbfTime = System.nanoTime() - start;

start = System.nanoTime();
StringBuilder sbd = new StringBuilder();
for (int i = 0; i < 100_000; i++) sbd.append("x");
long sbdTime = System.nanoTime() - start;

// sbdTime < sbfTime в однопоточном коде
```

❗ Ловушка собеса: кандидат говорит «StringBuffer медленнее», но не может объяснить, почему JIT иногда устраняет разницу (lock elision, escape analysis). Хороший ответ упоминает, что JIT может оптимизировать `StringBuffer` если докажет, что объект не утекает из потока.

**→ Уточняющий вопрос:** Что такое escape analysis в JVM и как он влияет на производительность StringBuffer?

**↳ Ответ:** Escape analysis — анализ JIT, определяющий, «утекает» ли объект за пределы создавшего его потока или метода. Если StringBuffer создан и используется только внутри одного метода, JIT может убрать синхронизацию как бессмысленную (lock elision) — никакого другого потока нет, монитор никому не нужен. Правило: StringBuffer в локальном методе JIT оптимизирует до скорости StringBuilder; если передать объект во внешнее поле или другой поток — JIT лишается этой возможности.

## 8. Что происходит при конкатенации строк через оператор +?

Оператор `+` для строк — синтаксический сахар, который компилятор преобразует во время компиляции. До Java 9 компилятор javac превращал `a + b + c` в `new StringBuilder().append(a).append(b).append(c).toString()`. Начиная с Java 9, вместо прямой подстановки кода генерируется вызов `invokedynamic` со ссылкой на `StringConcatFactory`, что даёт JVM возможность оптимизировать стратегию конкатенации в runtime.

Ключевая проблема — в цикле. `str += item` разворачивается в `str = new StringBuilder(str).append(item).toString()` — каждая итерация создаёт новый `StringBuilder` и копирует все накопленные данные заново. Итого O(n²) для n итераций.

```java
// Что пишет программист:
String sql = "SELECT " + columns + " FROM " + table + " WHERE " + condition;

// Что генерирует компилятор (до Java 9, концептуально):
String sql = new StringBuilder()
    .append("SELECT ").append(columns)
    .append(" FROM ").append(table)
    .append(" WHERE ").append(condition)
    .toString();

// Проблема в цикле:
String log = "";
for (String entry : entries) {
    log += entry + "\n"; // каждый раз new StringBuilder(log) — O(n²)!
}
```

❗ Ловушка собеса: показывают цикл с `+=` и спрашивают «компилятор же оптимизирует?». Нет — компилятор оптимизирует одно выражение в одной строке, но не может вынести `StringBuilder` за пределы цикла, потому что каждая итерация использует результат предыдущей.

**→ Уточняющий вопрос:** Как именно StringConcatFactory в Java 9 отличается от прямой подстановки StringBuilder — в чём преимущество invokedynamic?

**↳ Ответ:** Прямая подстановка javac вшивает `new StringBuilder().append().toString()` статически — стратегия фиксирована на compile-time. С `invokedynamic` + `StringConcatFactory` JVM при первом вызове *в runtime* выбирает оптимальную стратегию: например, заранее вычисляет точный размер буфера (без resize), использует `Unsafe` для прямой записи в byte[] без промежуточных объектов. Правило: invokedynamic — «отложенное решение оптимизации в runtime», что позволяет JVM генерировать код лучше, чем статический компилятор.

## 9. Как компилятор Java оптимизирует конкатенацию строк?

До Java 9: javac компилирует `a + b + c` в линейную цепочку `StringBuilder.append()`. Это статическая оптимизация на уровне байткода — всегда одна стратегия. До Java 5 это была вообще ручная работа разработчика.

С Java 9: вместо генерации кода для StringBuilder компилятор генерирует вызов `invokedynamic`, который при первом выполнении вызывает `StringConcatFactory.makeConcatWithConstants()`. Фабрика генерирует оптимальный код в runtime — например, заранее рассчитывает нужный размер буфера, чтобы не делать resize, или использует `Unsafe` для прямой записи в байтовый массив. Это быстрее, потому что нет промежуточного `StringBuilder`-объекта.

Константные строки оптимизируются отдельно: `"foo" + "bar"` компилятор заменяет на `"foobar"` прямо в compile-time (constant folding). `"prefix_" + CONSTANT` тоже схлопывается, если `CONSTANT` — `static final`.

```java
// Constant folding — компилятор схлопывает в compile-time:
static final String PREFIX = "user_";
String key = PREFIX + "name"; // → "user_name" прямо в байткоде

// invokedynamic в Java 9+: более эффективно в runtime
String url = "https://" + host + ":" + port + "/api/" + endpoint;
// JVM выбирает оптимальную стратегию при первом вызове (bootstrap)
```

❗ Ловушка собеса: часто спрашивают «когда конкатенация через + хуже StringBuilder?» — ответ: только в цикле с накоплением результата. Одиночное многочленное выражение Java 9+ оптимизирует не хуже явного StringBuilder.

**→ Уточняющий вопрос:** Что такое constant folding и как проверить, применил ли его компилятор к конкретному коду?

**↳ Ответ:** Constant folding — замена выражений из константных значений их результатом в compile-time: `"foo" + "bar"` → `"foobar"` прямо в байткоде; `static final int N = 2 * 5` → `10`. Проверить: `javap -c ClassName` — если в байткоде `ldc "foobar"` вместо двух отдельных `ldc` + `invokevirtual StringBuilder.append` — folding применился. Правило: `static final` поля и литералы складываются компилятором; обычные переменные (даже `final` параметры методов) — нет.

## 10. Можно ли использовать == для сравнения String?

Технически можно — `==` скомпилируется и выполнится. Но семантически это почти всегда ошибка, потому что `==` сравнивает ссылки (адреса объектов в памяти), а не содержимое строк. Две строки с одинаковым текстом, созданные разными путями, будут разными объектами и дадут `false` при `==`.

Единственные безопасные случаи использования `==` со строками: сравнение с `null` (разумно, хотя `Objects.isNull()` читабельнее) и намеренная проверка identity (это тот же самый объект?). В продакшн-коде для проверки содержимого всегда `equals()` или `equalsIgnoreCase()`.

```java
String a = "hello";
String b = "hello";
String c = new String("hello");
String d = "hel" + "lo";           // constant folding → из пула
String e = "hel";
String f = e + "lo";               // runtime конкатенация → новый объект

System.out.println(a == b);  // true  — оба из пула
System.out.println(a == c);  // false — c создан через new
System.out.println(a == d);  // true  — компилятор схлопнул "hel"+"lo"
System.out.println(a == f);  // false — конкатенация в runtime → новый объект
System.out.println(a.equals(f)); // true — по содержимому
```

❗ Ловушка собеса: в тесте `a == b` для двух литералов даёт `true` из-за String Pool — кандидат делает вывод «==  работает для строк». Потом этот же паттерн применяется к строкам из HTTP-запроса или БД — и всё ломается.

**→ Уточняющий вопрос:** Если написать `if (status == "active")` в коде, где `status` приходит из HTTP-запроса — всегда ли это даст `false`? Почему?

**↳ Ответ:** Почти всегда `false` — строка из HTTP-запроса создаётся через парсинг (new String из байт) и живёт в heap, а не в пуле; у неё другой адрес, чем у литерала `"active"`. Исключение: если где-то был явный `status.intern()` — объект оказался бы в пуле и `==` дало бы `true`, но это случайность. Правило: для строк из внешних источников `== "literal"` семантически некорректно; всегда используйте `.equals()`.

## 11. В чём разница между == и equals() для String?

`==` проверяет идентичность объектов: возвращает `true` только если обе переменные ссылаются на один и тот же объект в памяти (один адрес). `equals()` у `String` переопределён и проверяет посимвольное равенство содержимого — возвращает `true` если обе строки содержат одинаковые символы в одинаковом порядке.

`String.equals()` сначала проверяет `==` (быстрый путь), затем сравнивает длины, потом побайтово или посимвольно сравнивает внутренние массивы. Метод `equalsIgnoreCase()` дополнительно нормализует регистр при сравнении.

```java
String s1 = new String("config");
String s2 = new String("config");

// == : разные объекты в heap
System.out.println(s1 == s2);          // false
// equals : одинаковое содержимое
System.out.println(s1.equals(s2));     // true

// Безопасный паттерн: литерал слева — защита от NPE
System.out.println("config".equals(s1)); // true, не кинет NPE если s1 == null
System.out.println(s1.equals("config")); // true, но NPE если s1 == null
```

Хорошая практика: при сравнении строки с константой или литералом — ставить константу слева (`"expected".equals(actual)`), чтобы избежать `NullPointerException` если `actual` равен `null`.

❗ Ловушка собеса: показывают `s.equals(null)` — возвращает `false` (не NPE). А `null.equals(s)` — NPE. Кандидаты путают, что на что вызывается.

**→ Уточняющий вопрос:** Как работает `compareTo()` у String и в каких случаях его используют вместо `equals()`?

**↳ Ответ:** `compareTo()` выполняет лексикографическое сравнение: возвращает 0 (равно), отрицательное (текущая < аргумент) или положительное (текущая > аргумент) — как разница первого несовпадающего char/byte. Используется для сортировки: `TreeMap<String, V>`, `Collections.sort()`, `Comparator.naturalOrder()`. Правило: `equals()` — для проверки равенства; `compareTo()` — для упорядочивания; никогда не пишите `compareTo() == 0` вместо `equals()` — это работает, но нарушает читаемость.

## 12. Где хранится String Pool (в какой области памяти)?

До Java 7: String Pool находился в **PermGen** (Permanent Generation) — области памяти за пределами обычной heap, с фиксированным размером по умолчанию (обычно 64–256 МБ). PermGen не участвовал в обычной сборке мусора, поэтому строки в пуле не удалялись. Бесконтрольный `intern()` или большое количество динамически загружаемых классов (со строковыми константами) легко вызывал `OutOfMemoryError: PermGen space`.

Начиная с Java 7: String Pool переехал в **основной heap** (область Young/Old Generation). Теперь объекты в пуле участвуют в GC как обычные объекты — если на них нет других ссылок кроме самого пула, они собираются. В Java 8 PermGen был полностью упразднён и заменён на **Metaspace** (память за пределами heap, но для метаданных классов, а не строк).

```
До Java 7:
[ Heap: Young | Old ] [ PermGen: классы + String Pool ] ← фиксированный размер

Java 7+:
[ Heap: Young | Old | String Pool ] [ Metaspace: классы ]
```

Размер StringTable настраивается: `-XX:StringTableSize=N` (N — число бакетов, должно быть простым числом). По умолчанию: 1009 до Java 7, 60013 в Java 8, 65536+ в Java 11+.

❗ Ловушка собеса: классический вопрос «где хранится String Pool» — и кандидат отвечает «в PermGen». Правильный ответ зависит от версии JVM. Java 7+ → heap. Незнание этой эволюции — сигнал, что кандидат не следит за изменениями платформы.

**→ Уточняющий вопрос:** После переезда String Pool в heap — может ли GC собрать объект из пула? При каких условиях?

**↳ Ответ:** Да — StringTable хранит WeakReference на объекты. Если нет сильных ссылок из кода (ни из переменных, ни из полей класса), GC соберёт объект и удалит запись из таблицы. Литералы не собираются: на них всегда есть сильная ссылка из константного пула класса (пока жив ClassLoader). Правило: интернированные строки, вышедшие из scope, собираются — пул не «вечная» утечка; риск OOM только при массовом `intern()` быстрее, чем GC успевает собирать.

## 13. Может ли String Pool вызвать OutOfMemoryError?

Да, может, и это реальная проблема в продакшн-коде. Поскольку с Java 7 пул в heap, OOM проявится как `OutOfMemoryError: Java heap space` (не PermGen). Причина: массовый вызов `intern()` на уникальных строках или строках с низкой повторяемостью накапливает тысячи объектов в пуле, они держат ссылки из `StringTable` и не собираются GC пока живёт таблица.

Типичный сценарий: микросервис читает JSON из очереди сообщений, где каждое сообщение содержит уникальные UUID или timestamp-строки, и по незнанию вызывает `intern()` на каждой — за несколько часов работы пул разрастается до нескольких гигабайт.

```java
// Опасный паттерн — intern() на уникальных строках
Random rnd = new Random();
for (int i = 0; i < 10_000_000; i++) {
    String unique = UUID.randomUUID().toString().intern(); // УТЕЧКА!
    // unique выходит из scope, но StringTable держит ссылку
}
// → OutOfMemoryError: Java heap space

// Безопасно — intern() только для строк с высокой повторяемостью:
String country = row.getCountry().intern(); // ~200 уникальных значений из миллионов строк
```

До Java 7 проблема была острее: PermGen с дефолтным размером 64 МБ переполнялся быстро. Теперь heap обычно больше, но проблема не исчезла.

❗ Ловушка собеса: интервьюер спрашивает «как избежать OOM от String Pool» — ответ должен включать: не интернировать уникальные значения, мониторить размер пула через `-XX:+PrintStringTableStatistics`, настраивать `-XX:StringTableSize`.

**→ Уточняющий вопрос:** Как можно мониторить количество интернированных строк на работающей JVM без перезапуска?

**↳ Ответ:** `jcmd <pid> VM.stringtable` — выводит текущую статистику: число записей, число бакетов, среднюю длину цепочек без остановки JVM. Для динамического мониторинга запускать команду с интервалом и сравнивать `Number of entries`. Также: Java Flight Recorder (JFR) собирает событие `jdk.StringTableStatistics`. Правило: если число записей неограниченно растёт — где-то бесконтрольный `intern()` на уникальных данных.

## 14. Что делает метод substring() и как он работал до Java 7?

`substring(beginIndex, endIndex)` возвращает подстроку от `beginIndex` включительно до `endIndex` исключительно. Если передать один аргумент — `substring(beginIndex)` — возвращает от указанной позиции до конца строки.

До Java 7 (update 6): `String` хранил три поля — `char[] value`, `int offset`, `int count`. Метод `substring()` создавал новый объект `String`, который **разделял тот же массив `value`** с оригиналом, но с другими `offset` и `count`. Это была O(1) операция — никакого копирования. Проблема: если взять подстроку из огромной строки (например, первые 10 символов из строки длиной 1 МБ), оригинальный массив на 1 МБ оставался в памяти, пока жила подстрока.

```java
// До Java 7 (концептуально):
// String bigString = loadMegabyteFile(); // char[1_000_000]
// String tiny = bigString.substring(0, 10);
// tiny держит ссылку на char[1_000_000] — весь мегабайт не собирается GC!

// Обходной приём до Java 7:
String tiny = new String(bigString.substring(0, 10)); // копирует в новый массив

// Java 7u6+: substring() сам копирует, обходной приём не нужен
String tiny = bigString.substring(0, 10); // создаёт новый char[10]
```

❗ Ловушка собеса: кандидат может не знать об этом изменении и описывать только старое поведение. Или наоборот — описывает только новое, не упоминая историческую утечку. Полный ответ требует обоих.

**→ Уточняющий вопрос:** Если бы старое разделяемое поведение было такой проблемой, почему его так долго не меняли? Какой сценарий делал его полезным?

**↳ Ответ:** Shared-buffer substring был O(1) — никакого копирования, только новые offset/count. При многократном извлечении подстрок из одной большой строки (парсинг XML-документа, разбор протокола) это давало огромный выигрыш по скорости. Проблема проявилась с паттерном «одна маленькая подстрока долго живёт» — оригинальный массив не собирался GC. Правило: оптимизация «быстро, но с утечкой» держалась, пока быстрый путь был важнее; убрали, когда компактные строки (JEP 254) потребовали чистой модели.

## 15. Почему изменили реализацию substring() в Java 7?

Изменение было ответом на практические проблемы с утечками памяти. Паттерн «взять маленький токен из большой строки» был крайне распространён: парсинг HTTP-заголовков, разбор CSV-строк, обработка XML. Каждая подстрока держала в живых огромный исходный массив. Разработчики были вынуждены писать `new String(str.substring(...))` — неочевидный workaround.

Кроме утечек, старая реализация усложняла внутреннюю оптимизацию: Compact Strings в Java 9 требовали чистой модели без shared-массивов. Новая реализация — `Arrays.copyOfRange(value, begin, end)` — O(n) по длине подстроки, но предсказуема по памяти. Жертва скоростью substring'а ради надёжности и читаемости кода.

```java
// Реальный пример — парсинг Authorization header в HTTP-сервисе
public String extractToken(String header) {
    // header = "Bearer eyJhbGciOiJIUzI1NiJ9...."  (может быть длинный JWT)
    if (header != null && header.startsWith("Bearer ")) {
        return header.substring(7); // Java 7+: копирует токен в новый массив
        // До Java 7: весь header жил бы в памяти пока жил token
    }
    return null;
}
```

❗ Ловушка собеса: кандидата спрашивают «что изменилось в substring в Java 7» — он должен объяснить не только что изменилось, но и почему. Ответ «стало медленнее» без объяснения причин и компромисса — неполный.

**→ Уточняющий вопрос:** Как можно воспроизвести старое быстрое поведение substring (разделяемый буфер) в современной Java, если это действительно нужно по производительности?

**↳ Ответ:** Стандартным `String` — никак, substring всегда копирует в Java 7u6+. Альтернативы без копирования: `CharBuffer.wrap(charArray, offset, length)` — view без аллокации; `ByteBuffer.slice()` для байтовых данных; библиотека `Rope` для очень больших текстов со структурными разделами. Правило: если нужен O(1) substring → используйте `CharSequence`-абстракцию с `CharBuffer`; для большинства задач современная JVM компенсирует копирование через эффективное управление памятью и JIT.

## 16. Как работает метод split()?

`split(String regex)` компилирует переданную строку как регулярное выражение и разбивает строку по всем совпадениям, возвращая `String[]`. Под капотом вызывается `Pattern.compile(regex).split(this)`. При вызове несколько раз с одним и тем же паттерном — паттерн компилируется каждый раз заново, что неэффективно. Оптимальный подход — предкомпилировать `Pattern` один раз.

Второй параметр `limit` управляет поведением: `limit > 0` — максимум `limit` частей; `limit < 0` — все части, включая пустые хвостовые; `limit = 0` (по умолчанию) — пустые строки в конце массива отбрасываются.

```java
// Ошибка: точка — это regex "любой символ", не литеральная точка
String[] bad = "192.168.1.1".split(".");  // → [] (каждый символ — разделитель)
String[] good = "192.168.1.1".split("\\."); // → ["192", "168", "1", "1"]

// Limit: разница поведения
String csv = "a,b,,c,,";
System.out.println(Arrays.toString("a,b,,c,,".split(",")));     // [a, b, , c]  — хвост обрезан
System.out.println(Arrays.toString("a,b,,c,,".split(",", -1))); // [a, b, , c, , ] — все части

// Предкомпилированный Pattern для горячего пути (парсинг миллионов строк):
private static final Pattern COMMA = Pattern.compile(",");
String[] parts = COMMA.split(line); // эффективно в цикле
```

❗ Ловушка собеса: `split(".")` — классическая ошибка. Точка в regex означает «любой символ». Правильно — `split("\\.")`. Аналогично `|` → `\\|`, `*` → `\\*`. Интервьюеры часто показывают именно этот пример.

**→ Уточняющий вопрос:** В чём разница между `split(",", 0)` и `split(",", -1)` — покажи на примере строки с пустыми хвостовыми полями?

**↳ Ответ:** `split(",", 0)` (и просто `split(",")`) отбрасывает пустые строки в *конце*: `"a,b,,"` → `["a", "b"]` (2 элемента). `split(",", -1)` сохраняет все части: `"a,b,,"` → `["a", "b", "", ""]` (4 элемента). Правило: при парсинге CSV с фиксированным числом полей всегда используйте `split(",", -1)` — иначе строка `"Name,,"` (пустые поля в конце) даст неверное число элементов и индексы съедут.

## 17. В чём особенность метода replace() vs replaceAll()?

Основная разница: `replace(CharSequence target, CharSequence replacement)` работает с **литеральной** подстрокой (без regex), а `replaceAll(String regex, String replacement)` интерпретирует первый аргумент как **регулярное выражение**. Название `replace` вводит в заблуждение — оно тоже заменяет **все** вхождения, а не только первое (это делает `replaceFirst`).

Под капотом `replace(CharSequence, CharSequence)` всё равно использует regex внутри, но передаёт строку как `Pattern.LITERAL` — символы не интерпретируются как мета-символы. Это важно: если нужно заменить литеральную строку с спецсимволами (например, `"("` или `"."`) — используй `replace`, а не `replaceAll`, иначе придётся экранировать.

```java
String text = "user.name@example.com";

// replace — литерально, безопасно
System.out.println(text.replace(".", "_"));      // user_name@example_com

// replaceAll — regex, точка = любой символ!
System.out.println(text.replaceAll(".", "_"));   // __________________ (всё заменено)
System.out.println(text.replaceAll("\\.", "_")); // user_name@example_com (правильно)

// Практика: очистка SQL-параметра от спецсимволов
String sanitized = input
    .replace("'", "''")   // литеральная замена — безопасно
    .replace("\\", "\\\\");

// replaceAll с группами захвата (только replaceAll умеет):
String camelToSnake = "myVariableName".replaceAll("([A-Z])", "_$1").toLowerCase();
// → my_variable_name
```

❗ Ловушка собеса: кандидат думает, что `replace` заменяет только первое вхождение, а `replaceAll` — все. На самом деле оба заменяют все. Разница — в интерпретации паттерна.

**→ Уточняющий вопрос:** Как заменить все вхождения регулярного выражения в строке, не компилируя Pattern каждый раз при вызове метода в горячем цикле?

**↳ Ответ:** Предкомпилировать `Pattern` в `static final` поле: `private static final Pattern P = Pattern.compile("[0-9]+")`, затем `P.matcher(input).replaceAll("N")`. `String.replaceAll()` вызывает `Pattern.compile()` внутри на каждый вызов. Правило: regex в hot path → `static final Pattern`; компиляция паттерна — O(n) по сложности выражения и занимает десятки микросекунд, при тысячах вызовов в секунду это ощутимо.

## 18. Что такое String encoding (кодировка)?

Кодировка — это правило сопоставления символов (Unicode code points) и последовательностей байт. Unicode определяет числовой код для каждого символа (например, 'А' = U+0410), а кодировка определяет, какими байтами этот код представлен. UTF-8 использует от 1 до 4 байт (ASCII-символы — 1 байт), UTF-16 — 2 или 4 байта (Java внутри использует UTF-16), ISO-8859-1 — 1 байт, только латиница.

Внутри JVM `String` хранит символы в UTF-16 (в Java 9+ — Latin-1 если возможно, см. compact strings). Проблемы возникают на границах: чтение файлов, HTTP-запросы/ответы, сериализация, БД. Если не указать кодировку явно — используется платформенная (`Charset.defaultCharset()`), которая отличается на разных системах (Linux: UTF-8, Windows: windows-1251 или cp866).

```java
// Правильно: кодировка всегда явная
String json = """
    {"name": "Кириллица"}
    """;

// Сериализация в байты для HTTP-ответа:
byte[] utf8Bytes = json.getBytes(StandardCharsets.UTF_8);
// Content-Length должен считаться по utf8Bytes.length, не json.length()!

// Чтение файла с явной кодировкой:
List<String> lines = Files.readAllLines(
    Paths.get("config.properties"),
    StandardCharsets.UTF_8
);

// Ловушка: getBytes() без кодировки
byte[] dangerous = json.getBytes(); // платформенная кодировка — непредсказуемо!
```

❗ Ловушка собеса: `json.length()` возвращает количество символов (char), а `json.getBytes(UTF_8).length` — количество байт. Для кириллицы в UTF-8 один символ = 2 байта. При установке Content-Length в HTTP-ответе — нужен `bytes.length`, а не `string.length()`, иначе клиент получит обрезанный ответ.

**→ Уточняющий вопрос:** Почему `string.length()` может возвращать неправильный результат для строк с эмодзи или иероглифами?

**↳ Ответ:** `String.length()` возвращает количество `char` (единиц UTF-16), а символы выше U+FFFF (эмодзи, часть CJK) кодируются двумя `char` — surrogate pair. Эмодзи "😀" (U+1F600) → `length() == 2`, но визуально это 1 символ. Правило: для подсчёта видимых символов (Unicode code points) используйте `s.codePointCount(0, s.length())`; для обрезки по символам — `s.codePoints().limit(n)`.

## 19. Как правильно конвертировать String в byte[] и обратно?

Золотое правило: **всегда явно указывать кодировку**. `str.getBytes()` и `new String(bytes)` без аргументов используют `Charset.defaultCharset()` — платформозависимо, нестабильно между средами. Код, отлично работающий на локальной Windows-машине, сломается в Linux-контейнере.

Для конвертации используют `StandardCharsets.UTF_8` (или другую нужную кодировку). `StandardCharsets` — enum-like класс с константами, не требует `try-catch` (в отличие от `Charset.forName("UTF-8")`, который декларирует `UnsupportedEncodingException`). Roundtrip должен давать идентичный результат: `new String(str.getBytes(UTF_8), UTF_8).equals(str)` — true для любого Unicode-текста в UTF-8.

```java
import java.nio.charset.StandardCharsets;

// Правильная конвертация:
String original = "Привет, мир! 🌍";
byte[] bytes = original.getBytes(StandardCharsets.UTF_8);
String restored = new String(bytes, StandardCharsets.UTF_8);
System.out.println(original.equals(restored)); // true

// Практика: отправка JSON в REST API
public HttpRequest buildRequest(String body) {
    byte[] bodyBytes = body.getBytes(StandardCharsets.UTF_8);
    return HttpRequest.newBuilder()
        .POST(HttpRequest.BodyPublishers.ofByteArray(bodyBytes))
        .header("Content-Type", "application/json; charset=UTF-8")
        .header("Content-Length", String.valueOf(bodyBytes.length))
        .build();
}

// Чтение байт из потока (например, из Kafka-сообщения):
byte[] kafkaPayload = record.value();
String message = new String(kafkaPayload, StandardCharsets.UTF_8);
```

❗ Ловушка собеса: `new String(bytes, "UTF-8")` vs `new String(bytes, StandardCharsets.UTF_8)`. Первый вариант требует `catch (UnsupportedEncodingException)` (хотя UTF-8 всегда поддерживается). Второй — предпочтительнее: нет checked exception, код чище. На собесе оба варианта правильны, но второй показывает знание API.

**→ Уточняющий вопрос:** Если byte[] содержит невалидную для UTF-8 последовательность байт, что вернёт `new String(bytes, UTF_8)` — исключение или что-то другое?

**↳ Ответ:** Исключения не будет — Java молча заменит невалидные байты символом-заменителем U+FFFD (`?`). Это поведение `CharsetDecoder` с `CodingErrorAction.REPLACE` по умолчанию. Правило: `new String(bytes, UTF_8)` никогда не кидает, но может испортить данные; если нужна строгая валидация — используйте `CharsetDecoder` с `CodingErrorAction.REPORT`, который выбросит `MalformedInputException`.

## 20. Что такое compact strings в Java 9+?

Compact Strings — оптимизация хранения строк, введённая в Java 9 ([JEP 254](https://openjdk.org/jeps/254)). До Java 9 `String` всегда хранил символы в `char[]`, где каждый `char` занимает 2 байта (UTF-16). Анализ реальных приложений показал: большинство строк (особенно в серверных приложениях) содержат только Latin-1 символы (ASCII + западноевропейские), которым хватает 1 байта.

С Java 9 `String` хранит данные в `byte[]` и имеет поле `byte coder` (0 = LATIN1, 1 = UTF16). Для Latin-1 строк каждый символ занимает 1 байт вместо 2 — вдвое меньше памяти. Переключение между режимами прозрачно: операции над строками проверяют `coder` и вызывают соответствующую реализацию. Если строка содержит хотя бы один не-Latin-1 символ — используется UTF-16 (2 байта на char).

```java
// Compact Strings прозрачны для разработчика — API не изменился
String latin = "Hello, World!"; // хранится как byte[13], 1 байт/символ
String cyrillic = "Привет!";    // хранится как byte[14], 2 байта/символ (UTF-16)

// Проверить через рефлексию (не для прода!):
Field coderField = String.class.getDeclaredField("coder");
coderField.setAccessible(true);
System.out.println(coderField.get(latin));    // 0 (LATIN1)
System.out.println(coderField.get(cyrillic)); // 1 (UTF16)

// Отключить, если нужно (редкий случай):
// java -XX:-CompactStrings ...
```

В типичных Java-серверах compact strings даёт 10–20% экономии heap: заголовки HTTP, имена полей JSON, SQL-запросы — всё ASCII.

❗ Ловушка собеса: кандидаты путают compact strings с String Pool или с сжатием данных. Это не сжатие — это другой способ хранения (1 байт вместо 2 для Latin-1). Производительность не ухудшается: Latin-1 операции даже быстрее из-за лучшей cache locality.

**→ Уточняющий вопрос:** Как compact strings взаимодействуют с String deduplication в G1 GC — дедуплицируются ли byte[] массивы с разными `coder`?

**↳ Ответ:** Нет — G1 deduplication сравнивает массивы только если у обоих строк одинаковый `coder`: Latin-1 дедуплицируется с Latin-1, UTF-16 с UTF-16. Строки с разным `coder` не могут быть дедуплицированы, даже если содержат одинаковый текст (их byte[] структурно отличаются). Правило: дедупликация работает на уровне byte[] с проверкой coder + побайтового совпадения; compact strings и deduplication вместе не конфликтуют, просто дают дополнительный уровень экономии.

## 21. Как узнать, сколько памяти занимает String?

Строка в Java — это объект с несколькими полями, плюс отдельный объект массива. В современной Java (8+, 64-bit JVM с CompressedOops):
- Заголовок объекта `String`: 16 байт
- Поле `byte[] value`: 4 байта (ссылка, compressed)
- Поле `int hash`: 4 байта
- Поле `byte coder` (Java 9+): 1 байт (выравнивается до 4)
- Итого объект `String`: ~24 байта
- Объект `byte[]`: 16 байт заголовок + 4 байта (длина) + N байт данных
- Для Latin-1 строки длиной N: ~24 + 16 + 4 + N байт ≈ 44 + N байт
- Для UTF-16 строки длиной N символов: ~24 + 16 + 4 + 2*N байт ≈ 44 + 2*N байт

Точно измерить можно через:
1. **Java Object Layout (JOL)** — библиотека от OpenJDK для анализа layout объектов
2. **Heap dump + MAT (Memory Analyzer Tool)** — анализ в postmortem
3. `Instrumentation.getObjectSize()` — в runtime через java agent

```java
// JOL (добавить зависимость org.openjdk.jol:jol-core):
import org.openjdk.jol.info.ClassLayout;
import org.openjdk.jol.info.GraphLayout;

String s = "Hello";
System.out.println(ClassLayout.parseInstance(s).toPrintable());
// Покажет layout полей объекта

long totalSize = GraphLayout.parseInstance(s).totalSize();
System.out.println("Total size including byte[]: " + totalSize + " bytes");
// "Hello" (5 Latin-1 chars) → ~56 bytes
```

❗ Ловушка собеса: кандидат называет только `str.length() * 2` (для char) или просто `length()`. Правильный ответ учитывает: заголовок объекта String, заголовок массива, выравнивание, и версию Java (compact strings). Без JOL или heap dump точное значение невозможно вычислить вручную.

**→ Уточняющий вопрос:** Во сколько раз больше памяти займёт 1 000 000 объектов `String("a")` по сравнению с одним интернированным `"a"` из пула?

**↳ Ответ:** Один интернированный `"a"`: ~24 байта (объект String) + ~21 байт (byte[1] с заголовком) ≈ ~45 байт плюс 1 000 000 ссылок по 4/8 байт. 1 000 000 `new String("a")` (неинтернированных): 1 000 000 × (~24 + ~21) ≈ ~43 МБ. Отношение ~1 000 000:1 по числу объектов. Правило: интернирование строк с высокой повторяемостью превращает O(n) объектов в O(1) объект плюс n ссылок — экономия пропорциональна числу дублей.

## 22. Можно ли изменить содержимое String через рефлексию?

Технически — да, но это опасный и контракт-нарушающий приём. `String.value` — private final `byte[]`, но рефлексия позволяет вызвать `field.setAccessible(true)` и изменить элементы массива. Поскольку `final` запрещает изменить ссылку на массив, но не содержимое самого массива, можно изменить байты внутри.

Последствия катастрофические: если строка находится в String Pool (например, это литерал), то все ссылки на этот объект в JVM увидят изменение. Кэшированный `hashCode` устареет — строка «потеряется» в HashMap. JVM может нарушить инварианты, на которых основана безопасность (SecurityManager, ClassLoader использует строки для имён классов). В Java 16+ с `--illegal-access=deny` это поведение по умолчанию запрещено модульной системой.

```java
// НИКОГДА НЕ ДЕЛАТЬ В ПРОДЕ — только для понимания механизма:
String original = "hello";
System.out.println(original);        // hello

Field valueField = String.class.getDeclaredField("value");
valueField.setAccessible(true);
byte[] bytes = (byte[]) valueField.get(original);
bytes[0] = (byte) 'J';               // меняем 'h' → 'J'

System.out.println(original);        // Jello — ИЗМЕНИЛИ "иммутабельную" строку!
System.out.println("hello");         // Jello — СЛОМАЛИ ПУЛОВЫЙ ОБЪЕКТ!
// Теперь любой литерал "hello" в этом ClassLoader → "Jello"
```

В Java 9+ с системой модулей `setAccessible(true)` на `java.lang.String` требует `--add-opens java.base/java.lang=ALL-UNNAMED` и генерирует warning. В будущих версиях Java планируется полный запрет.

❗ Ловушка собеса: вопрос проверяет понимание иммутабельности (она договорная, а не абсолютная) и знание инкапсуляции. Интервьюер хочет услышать «технически можно, но это ломает String Pool и кэшированный hashCode» — а не просто «нет, нельзя».

**→ Уточняющий вопрос:** Если изменить содержимое String-ключа в HashMap через рефлексию, что произойдёт при попытке найти это значение по ключу?

**↳ Ответ:** HashMap кэширует hashCode ключа при `put()` (в Node). При изменении содержимого строки через рефлексию кэшированный `hash` в объекте String и hash в Node устаревают — новый вызов `hashCode()` даст другое значение. При `get(key)` HashMap вычислит новый hash, попадёт в другой бакет и не найдёт значение — вернёт `null`. Правило: ключи Map нельзя изменять после вставки; именно иммутабельность String защищает от этой проблемы.

## 23. Что такое String deduplication в G1 GC?

String deduplication — оптимизация G1 GC (доступна также в ZGC с Java 18+), включаемая флагом `-XX:+UseStringDeduplication`. Во время сборки мусора G1 находит объекты `String`, у которых одинаковое содержимое `byte[]`, но разные массивы (разные объекты, но дубли по значению). Для таких строк GC заменяет их внутренние ссылки так, чтобы они указывали на один и тот же `byte[]` — дубликаты массивов удаляются.

Ключевое отличие от String Pool: сами объекты `String` остаются разными (ссылки на них не меняются), только внутренние массивы объединяются. Это безопасно, потому что `byte[]` внутри String нельзя изменить через публичный API. Дедупликация работает асинхронно в фоне, без stop-the-world пауз.

```
До дедупликации:
String s1 → [byte[] id=100: "config"]
String s2 → [byte[] id=200: "config"]   ← дубль, занимает лишние ~20 байт

После дедупликации:
String s1 → [byte[] id=100: "config"]
String s2 → [byte[] id=100: "config"]   ← ссылаются на один массив
// byte[] id=200 собирается GC
```

Эффективна когда: много дублирующихся строк из внешних источников (парсинг JSON, чтение из БД), которые не интернируются явно. Не заменяет `intern()` — пул работает на уровне JVM и даёт `==` сравнение, дедупликация работает только на уровне массивов памяти.

```java
// Включить и мониторить:
// java -XX:+UseG1GC -XX:+UseStringDeduplication
//      -XX:+PrintStringDeduplicationStatistics ...
// В логах появится:
// [GC concurrent-string-deduplication, 1234.5K->567.8K(666.7K), ...
```

❗ Ловушка собеса: кандидат путает deduplication с intern() / String Pool. Чёткое отличие: Pool — один объект String для одного значения, `==` работает; Deduplication — несколько объектов String, но один byte[] массив, `==` не меняется. Дедупликация снижает потребление памяти, но не влияет на корректность `==`.

**→ Уточняющий вопрос:** Если включены и String deduplication, и явный intern() — как они взаимодействуют? Может ли дедупликация навредить интернированным строкам?

**↳ Ответ:** Дедупликация работает на уровне внутреннего byte[] — объекты String остаются разными, меняется только их внутренний массив. Для интернированных строк это безопасно: гарантия идентичности (`s1 == s2` после intern) не нарушается, просто оба объекта теперь указывают на один byte[]. Для интернированных строк дедупликация практически бесполезна — пул и так хранит одну копию значения. Правило: intern() и deduplication работают на разных уровнях и не конфликтуют; deduplication полезна для heap-строк без intern().
