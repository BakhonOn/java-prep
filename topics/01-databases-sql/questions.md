## 1. Для чего нужны индексы?

Индексы — это вспомогательные структуры данных, которые позволяют СУБД находить строки без полного перебора таблицы (full table scan). Без индекса PostgreSQL читает все страницы таблицы, что на таблице `orders` в 50 млн строк занимает секунды. С индексом на `user_id` тот же запрос выполняется за миллисекунды: движок прыгает прямо на нужные блоки на диске.

На практике в микросервисной архитектуре индексы жизненно необходимы на FK-колонках (`orders.user_id`, `order_items.order_id`), по которым происходят JOIN-ы и выборки. Без них каждый HTTP-запрос к сервису заказов будет делать seq scan по миллионам строк, и latency улетает в потолок.

```sql
-- медленно без индекса (seq scan по 50M строк)
SELECT * FROM orders WHERE user_id = 12345;

-- создаём индекс
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- теперь PostgreSQL использует Index Scan
EXPLAIN SELECT * FROM orders WHERE user_id = 12345;
```

❗ На собесе проверяют понимание того, что индексы — это компромисс: ускоряют чтение, но замедляют запись. Частая ошибка — ставить индексы на все колонки «про запас», или создать индекс на колонку с низкой селективностью (`status` с тремя значениями) и ожидать ускорения.

**→ Уточняющий вопрос:** Какие операции замедляет наличие индекса, и почему?

**↳ Ответ:** INSERT, UPDATE, DELETE — при каждом изменении строки PostgreSQL обновляет все индексы таблицы. Чем больше индексов, тем дороже запись. Правило: read-heavy таблица (>90% чтений) — индексы оправданы; write-heavy таблица (логи, метрики, очереди) — минимизируй индексы, иначе каждая вставка тормозит весь пайплайн.

## 2. Как работает B-tree индекс?

B-tree (Balanced Tree) — дерево, у которого все листовые узлы находятся на одинаковой глубине, что гарантирует O(log n) для поиска, вставки и удаления. Листья содержат пары (ключ → ctid строки в heap), внутренние узлы — разделители для навигации. Дерево сбалансировано автоматически при изменении данных.

PostgreSQL использует B-tree по умолчанию. Он эффективен для операций `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `ORDER BY`, `LIKE 'prefix%'`. При range-запросе (`BETWEEN '2024-01-01' AND '2024-12-31'`) движок находит левую границу по дереву, а затем идёт по связанному списку листьев — это называется Index Range Scan.

```sql
-- B-tree хорошо работает для этих запросов:
SELECT * FROM users WHERE email = 'user@example.com';        -- точный поиск
SELECT * FROM orders WHERE created_at BETWEEN ... AND ...;   -- диапазон
SELECT * FROM products ORDER BY price LIMIT 10;              -- сортировка

-- B-tree НЕ поможет здесь:
SELECT * FROM products WHERE name LIKE '%phone%';  -- префикс не определён
```

❗ На собесе часто спрашивают про `LIKE '%suffix%'` — индекс не используется, потому что нет чёткого начала диапазона в упорядоченном дереве. Для полнотекстового поиска нужны GIN/GiST индексы или `pg_trgm`.

**→ Уточняющий вопрос:** Для каких типов данных или запросов B-tree не подходит, и что использовать вместо него?

**↳ Ответ:** B-tree не подходит для: полнотекстового поиска (`LIKE '%text%'` без pg_trgm) → нужен GIN; геоданных → GiST; JSON с поиском по содержимому → GIN; массивов → GIN. Правило: если запрос не про равенство или диапазон упорядоченных значений — смотри в сторону GIN (для containment/overlap) или GiST (для геометрии/диапазонов).

## 3. Что такое составной индекс?

Составной (composite) индекс строится по нескольким колонкам одновременно: `CREATE INDEX idx ON orders(user_id, status, created_at)`. Ключи в дереве — это кортежи значений, упорядоченные слева направо. Это означает, что индекс эффективен только если в запросе используется **левый префикс** колонок.

В продакшне составные индексы незаменимы для часто повторяющихся паттернов запросов. Например, в сервисе заказов запрос «все активные заказы пользователя» — `WHERE user_id = ? AND status = 'ACTIVE'` — идеально покрывается индексом `(user_id, status)`. Если к нему добавить `created_at`, то и сортировка будет бесплатной.

```sql
CREATE INDEX idx_orders_user_status_date ON orders(user_id, status, created_at DESC);

-- использует индекс полностью:
SELECT * FROM orders WHERE user_id = 1 AND status = 'ACTIVE' ORDER BY created_at DESC;

-- использует только первую колонку:
SELECT * FROM orders WHERE user_id = 1;

-- НЕ использует индекс (нет левого префикса):
SELECT * FROM orders WHERE status = 'ACTIVE';
SELECT * FROM orders WHERE created_at > '2024-01-01';
```

❗ Распространённая ошибка — создать индекс `(a, b)` и ожидать, что запрос `WHERE b = ?` его использует. Порядок колонок в составном индексе критичен. На собесе могут дать запрос и попросить сказать, какой из индексов будет использован.

**→ Уточняющий вопрос:** Как называется оптимизация, когда индекс содержит все нужные для запроса колонки и обращение к heap не требуется?

**↳ Ответ:** Index-Only Scan (покрывающий индекс). Если все колонки в SELECT и WHERE есть в индексе — PostgreSQL читает только страницы индекса и не обращается к heap. Реализуется через `INCLUDE (col1, col2)` в PostgreSQL 11+. Правило: если один паттерн запроса повторяется часто — добавь INCLUDE нужных колонок для index-only scan.

## 4. Когда стоит создавать индекс?

Создавать индекс стоит, когда: колонка регулярно фигурирует в WHERE, JOIN ON, ORDER BY или GROUP BY; таблица большая (от ~10K строк, иначе seq scan зачастую быстрее); кардинальность высокая (много уникальных значений); чтений значительно больше, чем записей. Foreign key колонки в дочерних таблицах — почти всегда нужны.

На практике в e-commerce сервисе создают индексы на `orders.user_id` (выборка заказов пользователя), `orders.created_at` (временные диапазоны в дашборде), `products.category_id` (каталог по категориям). Каждый новый индекс проверяется через `EXPLAIN ANALYZE` до и после.

```sql
-- хорошие кандидаты для индексов:
CREATE INDEX idx_orders_user_id ON orders(user_id);              -- FK, высокая кардинальность
CREATE INDEX idx_orders_created_at ON orders(created_at DESC);   -- сортировка, диапазоны
CREATE UNIQUE INDEX idx_users_email ON users(email);             -- уникальность + поиск

-- плохие кандидаты:
CREATE INDEX idx_users_is_active ON users(is_active);  -- boolean: 2 значения, seq scan быстрее
```

❗ Часто спрашивают про индексы на FK: без них DELETE из родительской таблицы или JOIN ребёнка к родителю делает seq scan по всей дочерней таблице. PostgreSQL не создаёт индексы на FK автоматически — в отличие от MySQL InnoDB.

**→ Уточняющий вопрос:** Как PostgreSQL сам решает, использовать индекс или seq scan, и на что влияет этот выбор?

**↳ Ответ:** Оптимизатор оценивает стоимость: seq scan — дешёвый sequential IO; index scan — дорогой random IO. Если запрос вернёт >5–10% строк — seq scan выгоднее. Ключевой параметр: `random_page_cost` (дефолт 4.0, для SSD ставят 1.1–2.0) — его снижение «убеждает» оптимизатор использовать индексы агрессивнее. Основа решения — статистика кардинальности из `pg_statistics`.

## 5. Какие недостатки индексов?

Индексы замедляют модифицирующие операции: при каждом `INSERT`, `UPDATE`, `DELETE` PostgreSQL должен обновить все индексы на таблице. При bulk-загрузке (импорт миллионов строк) наличие 5 индексов может увеличить время операции в 3–5 раз. Кроме того, каждый индекс — это отдельная структура на диске, которая занимает место и требует обслуживания (VACUUM, ANALYZE).

Неиспользуемые индексы — скрытая проблема в продакшне: разработчик создал индекс на этапе разработки, запросы изменились, индекс теперь не используется, но INSERT/UPDATE продолжают его поддерживать. В PostgreSQL можно найти такие индексы через `pg_stat_user_indexes`.

```sql
-- найти неиспользуемые индексы
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY schemaname, tablename;

-- узнать размер индексов
SELECT indexname, pg_size_pretty(pg_relation_size(indexname::regclass)) AS size
FROM pg_indexes
WHERE tablename = 'orders';
```

❗ На собесе проверяют: понимает ли кандидат, что индексы надо периодически ревьювить и удалять лишние. Ещё одна ловушка — `UPDATE` на индексированную колонку создаёт новую версию строки и обновляет индекс, даже если значение не изменилось (Postgres 14+ оптимизирует это для HOT updates).

**→ Уточняющий вопрос:** Как называется механизм PostgreSQL, который позволяет обновить строку без обновления индекса, и при каких условиях он работает?

**↳ Ответ:** HOT (Heap-Only Tuple) update. Работает при двух условиях одновременно: обновляется колонка, не входящая ни в один индекс, И новая версия строки помещается на той же heap-странице (есть fillfactor-запас). Тогда B-tree не обновляется — экономия IO. Условие нарушается при изменении любой индексированной колонки или заполненности страницы.

## 6. Что такое кардинальность индекса?

Кардинальность — количество уникальных значений в колонке. Она определяет, насколько индекс избирателен (selective). Высокая кардинальность (UUID, email, timestamp) — запрос `WHERE email = ?` отберёт 1 строку из миллиона, индекс даёт огромный выигрыш. Низкая кардинальность (boolean, enum с 3 значениями, gender) — запрос `WHERE is_active = true` вернёт 50% таблицы, PostgreSQL скорее всего выберет seq scan.

В продакшне оптимизатор использует статистику кардинальности (собранную `ANALYZE`) для выбора плана. Если статистика устарела, оптимизатор может ошибиться: думает, что строк 100, а их 10 миллионов — и выбирает Index Scan вместо выгодного Hash Join.

```sql
-- посмотреть кардинальность колонок через статистику
SELECT attname, n_distinct
FROM pg_stats
WHERE tablename = 'orders';
-- n_distinct > 0 означает абсолютное кол-во уникальных значений
-- n_distinct < 0 означает долю (например, -1 = все значения уникальны)

-- пример: индекс на status бесполезен при 3 значениях
SELECT status, COUNT(*) FROM orders GROUP BY status;
-- ACTIVE: 40%, COMPLETED: 50%, CANCELLED: 10% — seq scan выгоднее
```

❗ Ловушка: кандидат создаёт индекс на `orders.status` и удивляется, почему его не используют. Оптимизатор трезво оценивает: если результат > ~5–10% строк таблицы, дешевле читать подряд. Частичный индекс (`WHERE status = 'ACTIVE'`) — способ решить эту проблему.

**→ Уточняющий вопрос:** Что такое частичный индекс и когда его применять вместо обычного?

**↳ Ответ:** Частичный индекс — с условием WHERE: `CREATE INDEX ON orders(user_id) WHERE status = 'ACTIVE'`. Содержит только строки, удовлетворяющие условию — меньше размер, быстрее обновление. Применять когда: запрос всегда фильтрует по конкретному значению низкокардинальной колонки (ACTIVE — 10% строк, COMPLETED — 90%). Обычный индекс на `status` не используется оптимизатором, а частичный — идеален.

## 7. Какие типы JOIN существуют?

INNER JOIN возвращает только строки, у которых есть совпадение в обеих таблицах. LEFT OUTER JOIN — все строки левой таблицы плюс совпадающие из правой (или NULL). RIGHT OUTER JOIN — зеркально. FULL OUTER JOIN — все строки обеих таблиц, NULL там, где нет совпадения. CROSS JOIN — декартово произведение (каждая строка с каждой). SELF JOIN — таблица с собой (например, иерархия сотрудников).

В реальных сервисах INNER JOIN используется для обязательных связей (`orders JOIN users` — заказ всегда принадлежит пользователю), LEFT JOIN — для опциональных (`users LEFT JOIN profiles` — профиль может отсутствовать). FULL OUTER JOIN встречается редко, обычно при синхронизации данных между источниками.

```sql
-- INNER: только заказы с известным пользователем
SELECT o.id, u.name FROM orders o INNER JOIN users u ON o.user_id = u.id;

-- LEFT: все пользователи, у некоторых может не быть заказов
SELECT u.name, COUNT(o.id) AS order_count
FROM users u LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- FULL OUTER: найти несоответствия между таблицами
SELECT a.id, b.id FROM table_a a FULL OUTER JOIN table_b b ON a.id = b.id
WHERE a.id IS NULL OR b.id IS NULL;
```

❗ На собесе часто просят объяснить, что вернёт LEFT JOIN, если в правой таблице нет совпадений — и почему потом нельзя делать `WHERE right_table.column = 'X'` (это превращает LEFT в INNER JOIN). Это классическая ошибка при написании сложных запросов.

**→ Уточняющий вопрос:** Что произойдёт с результатами LEFT JOIN, если добавить условие `WHERE правая_таблица.колонка IS NOT NULL`?

**↳ Ответ:** LEFT JOIN превратится в INNER JOIN по семантике. Строки без пары в правой таблице имеют NULL — условие `IS NOT NULL` их отсекает. Правило: фильтры на колонки правой таблицы в WHERE уничтожают смысл LEFT JOIN. Чтобы оставить LEFT JOIN и при этом фильтровать — переноси условие в ON clause: `LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'ACTIVE'`.

## 8. В чём разница между INNER JOIN и LEFT JOIN?

INNER JOIN возвращает строки только при наличии совпадения в обеих таблицах — строки без пары просто исчезают из результата. LEFT JOIN всегда сохраняет все строки левой таблицы; если в правой таблице нет совпадения, колонки правой заполняются NULL. Разница критична для отчётов: если нужно «все пользователи и количество их заказов», INNER JOIN скроет пользователей без заказов.

На практике ошибка выбора типа JOIN часто ведёт к silent data loss — продуктовый отчёт показывает 8000 пользователей вместо 10000, потому что 2000 без заказов пропали. Это трудно заметить без знания исходных данных.

```sql
-- INNER: пользователи БЕЗ заказов сюда не попадут
SELECT u.name, COUNT(o.id)
FROM users u INNER JOIN orders o ON u.id = o.user_id
GROUP BY u.id;

-- LEFT: ВСЕ пользователи, у кого нет заказов — count = 0
SELECT u.name, COUNT(o.id) AS order_count
FROM users u LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id;

-- Ловушка: это LEFT JOIN, но работает как INNER из-за WHERE:
SELECT u.name FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 'ACTIVE';  -- строки с NULL в o.status отфильтруются!
```

❗ Классическая ловушка собеса: показывают запрос с LEFT JOIN и WHERE по колонке правой таблицы без `IS NULL` проверки. Кандидат должен объяснить, что фильтр `WHERE o.status = 'ACTIVE'` исключает строки с NULL, и запрос фактически становится INNER JOIN.

**→ Уточняющий вопрос:** Как правильно написать запрос «все пользователи, у которых нет ни одного заказа»?

**↳ Ответ:** `SELECT u.* FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE o.id IS NULL`. Или через NOT EXISTS: `WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id)`. Правило: «найти записи без пары» — это всегда LEFT JOIN + WHERE правая.id IS NULL или NOT EXISTS. NOT EXISTS часто читается яснее и хорошо оптимизируется планировщиком.

## 9. Что лучше: JOIN или подзапрос?

В большинстве случаев JOIN предпочтительнее: оптимизатор лучше умеет работать с ними — выбирает алгоритм (Hash Join, Nested Loop, Merge Join), строит оптимальный порядок соединений, использует статистику. Подзапрос в SELECT (scalar subquery) или коррелированный подзапрос может выполняться для каждой строки внешнего запроса, что катастрофически медленно на больших таблицах.

Вместе с тем подзапросы бывают уместны и даже более читаемы: EXISTS вместо JOIN для проверки наличия, подзапрос для агрегации до JOIN-а (чтобы не дублировать строки), CTE для сложной логики. Современные оптимизаторы (PostgreSQL 12+) зачастую сами переписывают подзапрос в JOIN.

```sql
-- медленный коррелированный подзапрос O(n*m):
SELECT u.name,
  (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS cnt
FROM users u;

-- быстрый эквивалент через JOIN + GROUP BY:
SELECT u.name, COUNT(o.id) AS cnt
FROM users u LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- EXISTS часто быстрее чем JOIN для проверки наличия:
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.status = 'ACTIVE');
```

❗ На собесе могут дать запрос с коррелированным подзапросом в SELECT и попросить переписать через JOIN. Ключевой принцип: всегда смотри EXPLAIN ANALYZE — иногда оптимизатор сам справляется, иногда нет.

**→ Уточняющий вопрос:** В чём разница между `IN (подзапрос)` и `EXISTS (подзапрос)` по производительности?

**↳ Ответ:** `IN` материализует весь результат подзапроса и проверяет каждый элемент. `EXISTS` останавливается при первом совпадении (short-circuit) — выгодно когда совпадений много или подзапрос большой. С NULL — ловушка: `IN` с NULL в результате подзапроса может не вернуть ожидаемых строк (NULL = NULL не работает). Правило: `EXISTS` предпочтительнее для проверки наличия; `IN` с небольшим статическим списком констант — нормально.

## 10. Что такое коррелированный подзапрос?

Коррелированный подзапрос — это подзапрос, который ссылается на колонку из внешнего (родительского) запроса. Он не может быть вычислен один раз и закэширован: он перевыполняется заново для каждой строки внешнего запроса. Сложность — O(n × m), где n — строки внешнего запроса, m — строки во вложенном.

Типичный пример — «найти последний заказ каждого пользователя»: можно написать через коррелированный подзапрос, но это будет делать запрос к `orders` для каждого из 100K пользователей. Правильное решение — оконная функция или JOIN с подзапросом-агрегатом.

```sql
-- коррелированный подзапрос: медленно (100K запросов к orders)
SELECT u.id, u.name,
  (SELECT MAX(o.created_at) FROM orders o WHERE o.user_id = u.id) AS last_order
FROM users u;

-- быстрый вариант через оконную функцию:
SELECT DISTINCT u.id, u.name,
  MAX(o.created_at) OVER (PARTITION BY u.id) AS last_order
FROM users u JOIN orders o ON u.id = o.user_id;

-- или через JOIN с подзапросом:
SELECT u.id, u.name, lo.last_order
FROM users u
LEFT JOIN (SELECT user_id, MAX(created_at) AS last_order FROM orders GROUP BY user_id) lo
  ON u.id = lo.user_id;
```

❗ Коррелированный подзапрос в SELECT — это почти всегда красный флаг при code review. На собесе могут показать такой запрос и спросить, что с ним не так. Важно объяснить не только «это медленно», но и почему — per-row execution.

**→ Уточняющий вопрос:** Как EXPLAIN покажет коррелированный подзапрос, и на что смотреть в плане?

**↳ Ответ:** EXPLAIN покажет узел `SubPlan` — он выполняется для каждой строки внешнего запроса. Смотри на `loops=N` в EXPLAIN ANALYZE: если loops >> 1 и стоимость подзапроса высокая — это красный флаг O(n×m). `InitPlan` — выполняется один раз (некоррелированный подзапрос). Правило: видишь SubPlan с loops=тысячи — переписывай на JOIN или оконную функцию.

## 11. В чём разница между WHERE и HAVING?

WHERE фильтрует строки **до** группировки — работает с отдельными строками таблицы, не может использовать агрегатные функции. HAVING фильтрует **группы** после GROUP BY — работает с результатами агрегации, поэтому `HAVING COUNT(*) > 5` корректно, а `WHERE COUNT(*) > 5` — синтаксическая ошибка.

С точки зрения производительности WHERE предпочтительнее там, где возможно: он уменьшает число строк до группировки, и GROUP BY обрабатывает меньше данных. HAVING применяется только тогда, когда условие требует агрегата.

```sql
-- WHERE фильтрует до GROUP BY (эффективнее):
SELECT user_id, COUNT(*) AS cnt
FROM orders
WHERE created_at >= '2024-01-01'   -- фильтр по сырым строкам
GROUP BY user_id
HAVING COUNT(*) > 5;               -- фильтр по группам (агрегат)

-- Ошибка: нельзя использовать агрегат в WHERE:
SELECT user_id FROM orders
WHERE COUNT(*) > 5  -- ERROR: aggregate functions are not allowed in WHERE
GROUP BY user_id;
```

❗ Типичная ловушка: кандидат путает, куда поставить условие фильтрации. Если условие по обычной колонке ставится в HAVING вместо WHERE — это работает, но неэффективно: GROUP BY обрабатывает лишние строки. Интервьюер может спросить, где лучше писать `HAVING user_id = 5` или `WHERE user_id = 5`.

**→ Уточняющий вопрос:** Можно ли использовать в HAVING алиас, объявленный в SELECT? Зависит ли ответ от СУБД?

**↳ Ответ:** В PostgreSQL — нет: HAVING выполняется до SELECT, алиасы оттуда там недоступны. В MySQL/BigQuery — часто можно (расширение стандарта). Логический порядок: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. Правило: для переносимого SQL в HAVING всегда повторяй выражение, а не алиас — `HAVING SUM(amount) > 1000`, а не `HAVING total > 1000`.

## 12. Что делает GROUP BY?

GROUP BY разбивает результат на группы по уникальным комбинациям значений указанных колонок и позволяет применять агрегатные функции к каждой группе. После GROUP BY в SELECT можно использовать только колонки, перечисленные в GROUP BY, и агрегатные функции — всё остальное даст ошибку (или неопределённое поведение в MySQL).

В продакшне GROUP BY — основа любых аналитических запросов: выручка по дням, количество заказов по пользователям, среднее время обработки по категориям. При работе с большими таблицами GROUP BY требует сортировки или хеширования (Hash Aggregate / Sort Aggregate в EXPLAIN) и может быть узким местом.

```sql
-- выручка по дням за последний месяц
SELECT DATE_TRUNC('day', created_at) AS day,
       COUNT(*)                      AS order_count,
       SUM(total_amount)             AS revenue
FROM orders
WHERE created_at >= NOW() - INTERVAL '30 days'
GROUP BY DATE_TRUNC('day', created_at)
ORDER BY day;

-- ошибка: product_name не в GROUP BY и не агрегат
SELECT user_id, product_name, COUNT(*)  -- ERROR в PostgreSQL
FROM order_items
GROUP BY user_id;
```

❗ В PostgreSQL строгий режим: нельзя выбрать колонку, не входящую в GROUP BY и не обёрнутую в агрегат. MySQL по умолчанию это разрешает (возвращая произвольное значение), что приводит к непредсказуемым результатам. На собесе проверяют понимание этого различия.

**→ Уточняющий вопрос:** Как GROUP BY обрабатывает NULL-значения — попадают ли строки с NULL в одну группу?

**↳ Ответ:** Да, все NULL в группирующей колонке попадают в одну группу. PostgreSQL считает NULL = NULL для GROUP BY (хотя в WHERE — NULL != NULL). Это важно помнить при GROUP BY по nullable FK — строки без связи образуют свою группу с `col = NULL`. Проверить: `SELECT col, COUNT(*) FROM t GROUP BY col` — строки с NULL col окажутся в отдельной строке результата.

## 13. Когда использовать HAVING?

HAVING используется исключительно тогда, когда условие фильтрации требует агрегатной функции: «показать только пользователей, сделавших более 10 заказов», «найти категории с суммарной выручкой больше миллиона». Простые условия по колонкам всегда нужно ставить в WHERE — они выполняются раньше и сокращают объём данных для GROUP BY.

Типичный сценарий в продакшне: аналитический дашборд показывает топ-клиентов по объёму покупок. Запрос группирует заказы по `user_id`, считает сумму, и HAVING отсекает тех, кто купил меньше порога.

```sql
-- топ-клиенты: пользователи с суммой заказов > 100,000
SELECT u.name, SUM(o.total_amount) AS total_spent
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'COMPLETED'         -- WHERE: фильтр до группировки
GROUP BY u.id, u.name
HAVING SUM(o.total_amount) > 100000  -- HAVING: фильтр после агрегации
ORDER BY total_spent DESC;

-- найти дубликаты email в таблице:
SELECT email, COUNT(*) AS cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

❗ Один из любимых вопросов: «почему это неэффективно — `GROUP BY user_id HAVING user_id = 5`?» Потому что сначала группируются все пользователи, потом фильтруется один. Правильно: `WHERE user_id = 5 GROUP BY user_id`. Разница в порядке выполнения SQL: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY.

**→ Уточняющий вопрос:** В каком порядке логически выполняются клаузы SQL-запроса?

**↳ Ответ:** FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT/OFFSET. Это логический, а не физический порядок (оптимизатор меняет физику). Следствие: нельзя использовать алиас из SELECT в WHERE или GROUP BY; нельзя использовать агрегат в WHERE. Зная этот порядок — всегда знаешь, где что можно писать и почему.

## 14. Что такое оконные функции (Window Functions)?

Оконные функции вычисляют значение для каждой строки на основе «окна» — набора связанных строк — не сворачивая результат в одну строку, как делает GROUP BY. Каждая строка остаётся в результате, но получает дополнительно вычисленное значение. Синтаксис: `функция() OVER (PARTITION BY ... ORDER BY ... ROWS/RANGE ...)`.

Это одна из самых мощных возможностей SQL для аналитики. В продакшне они используются для рейтингов (топ N товаров в каждой категории), running totals (нарастающая выручка), lag/lead для сравнения с предыдущим/следующим значением, дедупликации (найти последнюю запись на пользователя).

```sql
-- нарастающая выручка по дням (running total)
SELECT
  order_date,
  daily_revenue,
  SUM(daily_revenue) OVER (ORDER BY order_date) AS running_total
FROM daily_sales;

-- топ-1 заказ каждого пользователя по сумме
SELECT * FROM (
  SELECT
    user_id, order_id, total_amount,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total_amount DESC) AS rn
  FROM orders
) t WHERE rn = 1;

-- сравнение с предыдущим периодом
SELECT
  month,
  revenue,
  LAG(revenue) OVER (ORDER BY month) AS prev_month_revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) AS delta
FROM monthly_revenue;
```

❗ Частая ошибка — путать оконные функции с GROUP BY. GROUP BY уменьшает количество строк; оконные функции — нет. Также: нельзя использовать оконную функцию в WHERE или HAVING (только в SELECT или ORDER BY), потому что она выполняется после них. Нужен подзапрос или CTE.

**→ Уточняющий вопрос:** Почему нельзя написать `WHERE ROW_NUMBER() OVER (...) = 1` и как это обойти?

**↳ Ответ:** Оконные функции выполняются после WHERE — WHERE не может фильтровать по тому, что ещё не вычислено. Обойти: вынести в подзапрос или CTE — `SELECT * FROM (SELECT *, ROW_NUMBER() OVER (...) AS rn FROM t) sub WHERE rn = 1`. Правило: любой фильтр по оконной функции — всегда через подзапрос или CTE.

## 15. Что делает ROW_NUMBER()?

`ROW_NUMBER()` присваивает каждой строке в окне уникальный последовательный номер, начиная с 1. При одинаковых значениях сортировки строки всё равно получают разные номера (порядок между ними не определён). Это делает её идеальной для дедупликации и паджинации.

В продакшне `ROW_NUMBER()` — стандартный способ получить «последнюю запись на сущность»: последний статус заказа, последнее действие пользователя, актуальная цена товара. Паттерн: оборачиваем в подзапрос и фильтруем `WHERE rn = 1`.

```sql
-- дедупликация: оставить последнее событие на пользователя
SELECT user_id, event_type, created_at
FROM (
  SELECT
    user_id, event_type, created_at,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
  FROM user_events
) t
WHERE rn = 1;

-- паджинация (альтернатива LIMIT/OFFSET для больших таблиц):
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (ORDER BY created_at DESC) AS rn
  FROM orders
) t
WHERE rn BETWEEN 101 AND 200;
```

❗ Вопрос: «чем ROW_NUMBER отличается от RANK?» ROW_NUMBER всегда уникален — при совпадении значений порядок произвольный. RANK при совпадении даёт одинаковый ранг и пропускает следующий номер. Для дедупликации нужен именно ROW_NUMBER, иначе можно получить несколько строк с rn=1.

**→ Уточняющий вопрос:** Что произойдёт, если в `ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC)` у двух строк одного пользователя одинаковый `created_at`?

**↳ Ответ:** Обе строки получат разные ROW_NUMBER (1 и 2), но порядок между ними недетерминирован — при разных запусках может меняться. Для стабильного результата добавь тай-брейкер: `ORDER BY created_at DESC, id DESC` — уникальный id гарантирует детерминированный порядок. Правило: ORDER BY в оконной функции должен однозначно упорядочивать строки, иначе результат непредсказуем.

## 16. Что делает RANK() и DENSE_RANK()?

Оба ранжируют строки в окне с учётом одинаковых значений. RANK при совпадении даёт одинаковый ранг, а следующий ранг пропускается: строки с одинаковым значением получают ранг 2, следующая — ранг 4 (ранг 3 пропущен). DENSE_RANK тоже даёт одинаковый ранг при совпадении, но следующий ранг идёт подряд без пропуска: 2, 2, 3.

В продакшне DENSE_RANK удобен для топ-N категорий при наличии «ничьих»: если нужно «топ-3 продукта по продажам» и два продукта с одинаковыми продажами поделили второе место, DENSE_RANK вернёт три позиции (1, 2, 2, 3), RANK вернёт (1, 2, 2, 4) — и четвёртую строку вместо третьей.

```sql
SELECT
  product_id,
  sales_amount,
  RANK()       OVER (ORDER BY sales_amount DESC) AS rank,
  DENSE_RANK() OVER (ORDER BY sales_amount DESC) AS dense_rank,
  ROW_NUMBER() OVER (ORDER BY sales_amount DESC) AS row_num
FROM product_sales;

-- Результат при sales: 1000, 900, 900, 800:
-- rank:        1, 2, 2, 4
-- dense_rank:  1, 2, 2, 3
-- row_number:  1, 2, 3, 4

-- топ-3 продукта в каждой категории:
SELECT * FROM (
  SELECT *, DENSE_RANK() OVER (PARTITION BY category_id ORDER BY sales DESC) AS dr
  FROM products
) t WHERE dr <= 3;
```

❗ Интервьюер может показать таблицу с данными и спросить, что вернёт каждая функция. Нужно чётко помнить разницу: RANK пропускает номера, DENSE_RANK — нет, ROW_NUMBER — всегда уникален. Ещё вопрос: «когда предпочесть DENSE_RANK перед RANK?» — когда важна непрерывность позиций в топ-N выборке.

**→ Уточняющий вопрос:** Как получить топ-3 пользователя по сумме покупок в каждой стране, включая все «ничьи» на третьем месте?

**↳ Ответ:** Использовать DENSE_RANK: `SELECT * FROM (SELECT user_id, country, SUM(amount) total, DENSE_RANK() OVER (PARTITION BY country ORDER BY SUM(amount) DESC) dr FROM orders GROUP BY user_id, country) t WHERE dr <= 3`. DENSE_RANK нужен, а не ROW_NUMBER, чтобы при «ничьих» на 3-м месте все равные попали в выборку — DENSE_RANK не пропускает номера.

## 17. Как работает MVCC в PostgreSQL?

MVCC (Multi-Version Concurrency Control) — механизм, при котором PostgreSQL хранит несколько версий одной строки одновременно. При UPDATE старая версия строки не перезаписывается — создаётся новая версия, а старая помечается как невидимая для новых транзакций. Каждая транзакция видит согласованный «снимок» данных на момент своего старта (или первого запроса, в зависимости от уровня изоляции), не блокируясь другими транзакциями на чтение.

Это даёт ключевое преимущество: читатели не блокируют писателей, писатели не блокируют читателей. В высоконагруженных системах это критично: тяжёлый аналитический запрос, читающий миллионы строк, не мешает одновременным INSERT/UPDATE от пользователей. Именно поэтому PostgreSQL хорошо держит смешанную OLTP+аналитику нагрузку.

```sql
-- Демонстрация: транзакция видит снимок на момент BEGIN
BEGIN;
-- в момент BEGIN таблица orders содержит 100 строк
-- другая транзакция вставляет 50 новых строк и коммитит

SELECT COUNT(*) FROM orders;  -- вернёт 100, а не 150
-- транзакция видит снимок, который был на момент BEGIN (READ COMMITTED — на момент запроса)
COMMIT;

-- посмотреть dead tuples (результат MVCC):
SELECT relname, n_dead_tup, n_live_tup
FROM pg_stat_user_tables
WHERE relname = 'orders';
```

❗ Главная ловушка — не понимать, что MVCC создаёт «мёртвые» версии строк (dead tuples), которые занимают место и не удаляются автоматически без VACUUM. На таблице с интенсивными UPDATE-ами без VACUUM таблица будет расти в размере, а запросы замедляться из-за необходимости просматривать мёртвые версии.

**→ Уточняющий вопрос:** При каком уровне изоляции транзакция в PostgreSQL получает снимок — в момент BEGIN или в момент первого запроса?

**↳ Ответ:** При READ COMMITTED (дефолт) — снимок создаётся для каждого запроса отдельно. Два последовательных SELECT в одной транзакции могут видеть разные данные. При REPEATABLE READ и SERIALIZABLE — снимок создаётся при первом запросе после BEGIN и остаётся неизменным. Правило: если нужна стабильность данных внутри транзакции — повышай уровень изоляции или используй FOR SHARE.

## 18. Что такое VACUUM в PostgreSQL?

VACUUM — процесс уборки мёртвых версий строк (dead tuples), накопившихся из-за MVCC. Обычный VACUUM помечает страницы как доступные для повторного использования, но не возвращает место операционной системе. `VACUUM FULL` переписывает таблицу полностью в новый файл — возвращает место ОС, но требует `ACCESS EXCLUSIVE` блокировку на всё время работы (таблица недоступна). `VACUUM ANALYZE` совмещает уборку со сбором статистики.

В продакшне autovacuum запущен постоянно и справляется в большинстве случаев, но есть ситуации, когда ручной VACUUM необходим: после массовых DELETE/UPDATE (bulk clean), если autovacuum не успевает из-за большой нагрузки, или если таблица сильно «раздулась» (bloat). `VACUUM FULL` производится в окно обслуживания — в рабочее время нельзя.

```sql
-- проверить bloat и необходимость VACUUM
SELECT
  schemaname,
  tablename,
  n_dead_tup,
  n_live_tup,
  ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct,
  last_vacuum,
  last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY dead_pct DESC;

-- ручной VACUUM без блокировки:
VACUUM ANALYZE orders;

-- VACUUM FULL — только в maintenance window (блокирует таблицу!):
VACUUM FULL orders;
```

❗ Путаница VACUUM и VACUUM FULL — классика собеса. Обычный VACUUM не блокирует и не возвращает место ОС. VACUUM FULL блокирует таблицу полностью и нужен редко. Ещё ловушка: `transaction ID wraparound` — если VACUUM не запускался очень долго, PostgreSQL может заморозить базу для предотвращения катастрофы с xid. Autovacuum следит за этим, но при его отключении — риск.

**→ Уточняющий вопрос:** Что такое table bloat, как он возникает и как его измерить?

**↳ Ответ:** Table bloat — раздутие физического размера таблицы из-за мёртвых версий строк (dead tuples), которые VACUUM не успел убрать. Возникает при интенсивных UPDATE/DELETE без достаточного VACUUM. Измерить: расширение `pgstattuple` → `SELECT * FROM pgstattuple('orders')` (показывает `dead_tuple_percent`). Косвенно: `n_dead_tup / (n_live_tup + n_dead_tup)` из `pg_stat_user_tables`.

## 19. Зачем нужен ANALYZE?

ANALYZE собирает статистику о распределении данных в таблицах: количество уникальных значений, гистограммы значений, процент NULL-ов — и записывает это в `pg_statistics`. Планировщик запросов использует эту статистику для оценки количества строк (cardinality estimation) и выбора оптимального плана: какой индекс использовать, какой алгоритм JOIN выбрать (Hash Join vs Nested Loop), в каком порядке соединять таблицы.

Без актуальной статистики оптимизатор принимает решения на основе устаревших данных и ошибается в оценках кардинальности. Например: таблица `orders` выросла с 10K до 50M строк, но ANALYZE не запускался — планировщик думает, что строк 10K, выбирает Nested Loop вместо Hash Join и запрос работает в 100 раз медленнее.

```sql
-- запустить ANALYZE вручную (без блокировки)
ANALYZE orders;
ANALYZE VERBOSE orders;  -- с подробным выводом

-- посмотреть статистику по колонке
SELECT
  attname,
  n_distinct,
  correlation,  -- насколько физический порядок совпадает с логическим
  most_common_vals,
  most_common_freqs
FROM pg_stats
WHERE tablename = 'orders' AND attname = 'status';

-- проверить когда последний раз запускался ANALYZE
SELECT tablename, last_analyze, last_autoanalyze
FROM pg_stat_user_tables
WHERE tablename = 'orders';
```

❗ Ловушка: разработчик делает массовую загрузку данных (`INSERT INTO ... SELECT ...` на 10M строк) и сразу запускает сложный аналитический запрос. Autovacuum ещё не успел запустить ANALYZE — план запроса строится на старой статистике. Решение: `ANALYZE table_name` сразу после bulk-загрузки. Хорошая практика: `VACUUM ANALYZE` после значительных изменений в таблице.

**→ Уточняющий вопрос:** Что такое `default_statistics_target` и когда его стоит увеличивать для конкретной колонки?

**↳ Ответ:** `default_statistics_target` (дефолт 100) — количество записей в гистограмме для каждой колонки. Выше значение → точнее оценки оптимизатора → дольше ANALYZE. Увеличивать для конкретной колонки: `ALTER TABLE orders ALTER COLUMN amount SET STATISTICS 500` — когда оптимизатор систематически ошибается в оценке строк для неё (видно по расхождению rows= vs actual rows= в EXPLAIN ANALYZE). Полезно для колонок со скошенным распределением.

## 20. Что такое explain plan?

`EXPLAIN` показывает план выполнения запроса, который построил оптимизатор: дерево операций (Seq Scan, Index Scan, Hash Join, Sort, Aggregate и т.д.), оценочную стоимость (`cost`) и ожидаемое количество строк. `EXPLAIN ANALYZE` реально выполняет запрос и добавляет фактическое время и реальное количество строк. `EXPLAIN (ANALYZE, BUFFERS)` также показывает операции с буферным кешем (сколько страниц читалось из кеша vs с диска).

Это основной инструмент оптимизации в продакшне. Самые важные узлы плана — те, у которых высокая фактическая стоимость и большое расхождение между ожидаемым (rows=10) и реальным (rows=1000000) количеством строк. Расхождение означает устаревшую статистику. Seq Scan на большой таблице — сигнал для добавления индекса.

```sql
-- базовый EXPLAIN:
EXPLAIN SELECT * FROM orders WHERE user_id = 12345;
-- Seq Scan on orders  (cost=0.00..2847.00 rows=1 width=112)
--   Filter: (user_id = 12345)

-- после создания индекса:
EXPLAIN SELECT * FROM orders WHERE user_id = 12345;
-- Index Scan using idx_orders_user_id on orders  (cost=0.43..8.45 rows=1 width=112)
--   Index Cond: (user_id = 12345)

-- полный анализ с буферами:
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT u.name, COUNT(o.id)
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id;
```

❗ Ловушки: 1) `EXPLAIN` без `ANALYZE` показывает только оценку — plan != reality, нужны реальные цифры. 2) `EXPLAIN ANALYZE` реально выполняет запрос — для `DELETE/UPDATE` оборачивай в транзакцию и делай `ROLLBACK`. 3) Стоимость (cost) — относительная единица, не миллисекунды, сравнивать имеет смысл только узлы одного плана. 4) Сайт `explain.depesz.com` или `explain.tensor.ru` упрощают анализ сложных планов.

**→ Уточняющий вопрос:** Что означают цифры `cost=0.00..2847.00` в выводе EXPLAIN, и как интерпретировать расхождение между `rows=1` в плане и `rows=50000` в `ANALYZE`?

**↳ Ответ:** `cost=0.00..2847.00` — две условные единицы стоимости: первая — startup cost (получить первую строку), вторая — суммарная стоимость узла. Единицы не миллисекунды — сравнивай только узлы одного плана. Расхождение `rows=1` (план) vs `rows=50000` (факт) — устаревшая статистика: оптимизатор ошибся и выбрал неоптимальный алгоритм. Решение: `ANALYZE table`.

## 21. Как оптимизировать медленные запросы?

Методичный процесс: 1) `EXPLAIN (ANALYZE, BUFFERS)` — найти узлы с высокой стоимостью и расхождением оценок. 2) Проверить наличие нужных индексов — seq scan на большой таблице в месте фильтрации или JOIN-а требует индекса. 3) `ANALYZE` — если оценки далеки от реальности, статистика устарела. 4) Переписать запрос: коррелированные подзапросы → JOIN, `SELECT *` → только нужные колонки, сложные условия в WHERE для использования индексов.

Дальше — по ситуации: составные индексы для покрытия всего запроса (index-only scan), партиционирование для очень больших таблиц, денормализация для тяжёлых JOIN-ов, materialized views для аналитических агрегатов, которые считаются редко. Иногда помогает `SET enable_seqscan = off` в сессии, чтобы принудить использование индекса для диагностики — в продакшне не использовать.

```sql
-- шаг 1: смотрим план
EXPLAIN (ANALYZE, BUFFERS)
SELECT u.name, COUNT(o.id), SUM(o.total_amount)
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
WHERE o.status = 'COMPLETED' AND o.created_at >= NOW() - INTERVAL '90 days'
GROUP BY u.id, u.name
HAVING SUM(o.total_amount) > 1000
ORDER BY SUM(o.total_amount) DESC
LIMIT 100;

-- шаг 2: добавляем покрывающий индекс под фильтр
CREATE INDEX idx_orders_status_date_user
  ON orders(status, created_at DESC)
  INCLUDE (user_id, total_amount);
  -- INCLUDE — добавляет колонки только в leaf nodes для index-only scan

-- шаг 3: materialized view для тяжёлого агрегата
CREATE MATERIALIZED VIEW mv_user_stats AS
SELECT user_id, SUM(total_amount) AS total_spent, COUNT(*) AS order_count
FROM orders WHERE status = 'COMPLETED'
GROUP BY user_id;

CREATE UNIQUE INDEX ON mv_user_stats(user_id);

REFRESH MATERIALIZED VIEW CONCURRENTLY mv_user_stats;  -- без блокировки чтения
```

❗ Системная ошибка — оптимизировать «на глаз» без EXPLAIN ANALYZE. Ещё одна: добавлять индексы бездумно, не проверяя, используются ли они. В продакшне полезно включить `pg_stat_statements` — он хранит статистику всех запросов (avg time, calls, total time) и позволяет найти самые дорогие запросы системно, а не «угадывать» проблему.

**→ Уточняющий вопрос:** Что такое `pg_stat_statements` и как его использовать для нахождения самых медленных запросов в продакшне?

**↳ Ответ:** `pg_stat_statements` — расширение, накапливающее статистику по каждому уникальному запросу: количество вызовов, суммарное и среднее время, строки. Включается через `CREATE EXTENSION pg_stat_statements`. Находишь самые дорогие: `SELECT query, calls, mean_exec_time, total_exec_time FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10`. Правило: смотри на `total_exec_time` — запрос с mean=1ms и calls=1M дороже, чем mean=1s и calls=5.
