## 1. Что такое топик (topic) в Kafka?

Топик — это именованный, append-only лог сообщений, разбитый на одну или несколько партиций. Продюсеры пишут в топик, консьюмеры читают из него независимо и не разрушают данные при чтении. В отличие от классических очередей (RabbitMQ), сообщение остаётся в топике после прочтения до истечения retention.

Внутри Kafka каждый топик — это директория на файловой системе каждого брокера, содержащая сегментные файлы (`.log`, `.index`). Сообщения упорядочены внутри каждой партиции (гарантируется total order), но порядок между разными партициями не гарантирован. Именно поэтому для заказа, который проходит шаги `CREATED → PAID → SHIPPED`, все события нужно класть в одну партицию (через ключ `orderId`).

В микросервисах на обработке заказов типично создают топики `order-events`, `payment-events`, `notification-events`. Сервис оплаты читает `order-events`, публикует в `payment-events`. Сервис уведомлений читает оба топика и рассылает письма. Все три сервиса независимы, связь только через Kafka.

```yaml
# конфигурация топика через AdminClient или kafka-topics.sh
bin/kafka-topics.sh --create \
  --topic order-events \
  --partitions 12 \
  --replication-factor 3 \
  --config retention.ms=604800000 \
  --bootstrap-server localhost:9092
```

❗ На собесе ловят на том, что «топик» часто путают с «очередью». В очереди сообщение удаляется после получения. В Kafka — нет. Несколько Consumer Group могут читать один топик независимо, каждая со своим offset. Также ловят на утверждении «порядок гарантирован» — уточняйте: порядок гарантирован только внутри партиции.

**→ Уточняющий вопрос:** Если порядок событий для одного заказа важен, как вы обеспечите его при нескольких партициях в топике?

## 2. Что такое партиция (partition) и зачем она нужна?

Партиция — это физически отдельный упорядоченный лог внутри топика. Именно партиция является единицей параллелизма: разные партиции могут лежать на разных брокерах, и разные консьюмеры одной группы читают разные партиции одновременно. Без партиций Kafka был бы обычной очередью с одним производителем и одним потребителем.

Каждая партиция представляет собой набор сегментных файлов на диске. Новые сообщения дописываются в конец активного сегмента (append-only — это и есть источник высокой производительности: sequential I/O). Каждое сообщение получает монотонно возрастающий offset внутри партиции. Реплики партиции распределяются по разным брокерам для отказоустойчивости.

Допустим, сервис обработки платежей получает 50 000 событий в секунду. С 1 партицией один консьюмер не справится. С 12 партициями — 12 консьюмеров обрабатывают параллельно, каждый получает ~4 000 событий/с. Число партиций определяет максимальный параллелизм на стороне консьюмера.

❗ Ловушка: нельзя уменьшить число партиций у существующего топика (только увеличить). При увеличении нарушается маппинг hash(key) % partitions — сообщения с одним ключом могут пойти в другую партицию, порядок в пределах ключа нарушится. Планируйте партиции с запасом на старте.

**→ Уточняющий вопрос:** Как вы определяете оптимальное количество партиций для нового топика при проектировании системы?

## 3. Как данные распределяются по партициям?

Механизм зависит от наличия ключа. Если ключ задан — `hash(key) % numPartitions` (точнее, murmur2 hash). Один и тот же ключ всегда попадёт в одну и ту же партицию (пока не изменилось число партиций), что гарантирует порядок событий для данной сущности. Если ключ не задан — используется sticky partitioner (Kafka 2.4+): продюсер заполняет батч в одну партицию, потом переключается, что эффективнее старого round-robin (меньше маленьких батчей).

Внутри DefaultPartitioner сначала проверяется наличие ключа. Если ключ есть — `Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions`. Если нет — выбирается «прилипшая» партиция для текущего батча (sticky partitioner), что снижает число сетевых вызовов и улучшает батчинг.

На практике: для топика `payment-events` ключом ставят `userId` или `orderId`. Все события одного заказа (INITIATED, PROCESSING, COMPLETED) попадут в одну партицию, и консьюмер обработает их строго по порядку. Для событий аудит-лога, где порядок не важен, ключ не задают — данные распределяются равномерно.

```java
// Producer с явным ключом
ProducerRecord<String, String> record = new ProducerRecord<>(
    "payment-events",
    order.getId().toString(),   // ключ → определяет партицию
    objectMapper.writeValueAsString(paymentEvent)
);
producer.send(record);
```

❗ Ловушка: если использовать ключи с неравномерным распределением (например, `countryCode` где 90% — одна страна), одна партиция станет hotspot. Все события этой страны попадут к одному консьюмеру — bottleneck. Решение: составной ключ (`countryCode + userId`) или кастомный Partitioner.

## 4. Что такое ключ сообщения и как он влияет на партиционирование?

Ключ сообщения — это произвольный byte[], который продюсер прикладывает к сообщению. Его основная функция — детерминированный выбор партиции через `murmur2(key) % partitions`, что гарантирует попадание всех сообщений с одним ключом в одну партицию и, следовательно, сохранение порядка для этого ключа.

Помимо партиционирования, ключ используется при log compaction: Kafka хранит только последнее сообщение для каждого уникального ключа. Это удобно для state store — топик с compaction и ключом `orderId` содержит актуальное состояние каждого заказа. При чтении с начала топика консьюмер восстановит текущее состояние всех заказов.

В сервисе заказов: ключ = `orderId`. События `ORDER_CREATED`, `PAYMENT_RECEIVED`, `SHIPPED` для одного orderId всегда идут в одну партицию. Консьюмер (например, сервис складского учёта) читает их строго по порядку и не рискует обработать SHIPPED раньше ORDER_CREATED.

```java
// Spring Kafka — отправка с ключом
@Service
public class OrderEventPublisher {
    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public void publish(OrderEvent event) {
        kafkaTemplate.send("order-events", event.getOrderId(), event);
        // orderId как ключ → гарантия порядка по заказу
    }
}
```

❗ Ловушка: ключ `null` не означает ошибку — сообщение просто уйдёт через sticky partitioner. Но если часть продюсеров пишет с ключом, а часть без — для одного orderId порядок сломается (события без ключа пойдут в случайные партиции). Убедитесь, что все продюсеры конкретного топика используют ключ согласованно.

**→ Уточняющий вопрос:** В чём разница между использованием ключа для партиционирования и использованием ключа для log compaction?

## 5. Что такое Consumer Group?

Consumer Group — это логическая группа консьюмеров, идентифицированная `group.id`. Kafka автоматически распределяет партиции топика между членами группы: каждая партиция отдаётся ровно одному консьюмеру группы. Это обеспечивает горизонтальное масштабирование потребления. Две разные группы читают один топик независимо — у каждой свой offset.

Offset каждой группы хранится в внутреннем топике `__consumer_offsets`. Координатор группы (один из брокеров) управляет membership: принимает heartbeats от консьюмеров и инициирует rebalance при изменениях. При старте консьюмер находит координатора, отправляет JoinGroup, получает назначенные партиции.

Типичный паттерн: топик `order-events` читают три независимые группы — `billing-service` (обрабатывает платежи), `warehouse-service` (резервирует товар), `notification-service` (отправляет письмо клиенту). Все три получают каждое событие независимо. Внутри `billing-service` развёрнуто 3 инстанса — они делят партиции между собой.

```yaml
# application.yml Spring Kafka consumer
spring:
  kafka:
    consumer:
      group-id: billing-service
      bootstrap-servers: kafka:9092
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "com.example.events"
```

❗ Ловушка: если два микросервиса случайно используют один `group.id`, они будут делить партиции между собой — каждый получит только половину сообщений. Это сложно обнаружить в логах. Соглашение: `group.id` = имя сервиса, уникальное в системе.

**→ Уточняющий вопрос:** Что произойдёт с offset консьюмера группы, если все инстансы сервиса перезапустятся?

## 6. Как работает балансировка консьюмеров в группе?

Балансировка (rebalance) — это процесс переназначения партиций между активными членами Consumer Group. Координатор группы собирает всех участников через фазу JoinGroup, затем лидер группы (первый присоединившийся консьюмер) запускает алгоритм распределения (assignor) и отправляет результат обратно через SyncGroup. Все консьюмеры получают свои партиции и начинают чтение.

Существуют несколько стратегий assignor: `RangeAssignor` (по умолчанию, делит диапазонами по топикам), `RoundRobinAssignor` (по кругу через все топики), `StickyAssignor` (минимизирует перемещения при rebalance), `CooperativeStickyAssignor` (инкрементальный rebalance без полной остановки, Kafka 2.4+).

В сервисе обработки заказов с 12 партициями и 3 инстансами каждый получит по 4 партиции. При добавлении 4-го инстанса — rebalance, каждый получит по 3. Стратегия CooperativeStickyAssignor позволяет переназначить только «лишние» партиции, не останавливая обработку на остальных.

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
        session.timeout.ms: 30000
        heartbeat.interval.ms: 10000
```

❗ Ловушка: при eager rebalance (стратегии Range, RoundRobin) все консьюмеры временно теряют все партиции — чтение полностью останавливается на время rebalance. При большой группе и частых деплоях это заметная задержка. Решение: CooperativeStickyAssignor (incremental rebalance) + StaticMembership (`group.instance.id`).

## 7. Можно ли иметь больше консьюмеров, чем партиций?

Можно, но лишние консьюмеры будут полностью простаивать — ни одной партиции им не достанется. Kafka назначает каждую партицию ровно одному консьюмеру группы. Если консьюмеров 5, а партиций 3 — трое работают, двое ждут. При падении одного из активных лишний консьюмер подхватит его партицию.

Это поведение заложено архитектурно: единица параллелизма в Consumer Group = партиция. Именно поэтому рекомендуется проектировать число партиций с запасом (например, 12, а не 3), чтобы при горизонтальном масштабировании сервиса можно было добавить инстансы и они сразу получат партиции.

На практике: сервис платежей стартует с 2 инстансами при 12 партициях — каждый читает 6. Нагрузка выросла — добавили 10 инстансов, теперь каждый читает по 1. При 13 инстансах — 12 работают, 1 простаивает. Это горячий резерв: при падении любого из 12 резервный мгновенно подхватит его партицию без ожидания нового деплоя.

❗ Ловушка: на собесе часто спрашивают «почему не увеличить партиции под число консьюмеров?». Ответ: увеличение партиций возможно, но нарушает порядок для ключей (hash rerout). Кроме того, каждая партиция — overhead на брокере (файловые дескрипторы, память для репликации). Десятки тысяч партиций могут перегрузить ZooKeeper/KRaft.

## 8. Что произойдёт при добавлении нового консьюмера в группу?

Новый консьюмер отправляет брокеру-координатору JoinGroup request. Координатор уведомляет всех текущих членов группы о необходимости rebalance (при следующем poll или heartbeat). Все консьюмеры ревокируют свои текущие партиции, отправляют JoinGroup, лидер группы пересчитывает назначения, SyncGroup рассылает результат. Каждый консьюмер получает новые партиции и возобновляет чтение с последнего закоммиченного offset.

Время rebalance зависит от `session.timeout.ms` (время обнаружения упавшего консьюмера) и числа членов группы. При eager rebalance всё чтение останавливается на всё это время. CooperativeStickyAssignor делает это инкрементально: только новому консьюмеру передаются партиции, остальные не прерываются.

В k8s при rolling deploy сервиса обработки уведомлений: каждый новый pod регистрируется в группе — происходит rebalance. При 10 репликах и медленном rebalance это 10 последовательных остановок. С CooperativeStickyAssignor и StaticMembership (`group.instance.id` = имя пода) переподнявшийся pod просто «переподключается» к тем же партициям без полного rebalance.

```yaml
spring:
  kafka:
    consumer:
      properties:
        group.instance.id: ${HOSTNAME}  # статическое членство — pod name
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

❗ Ловушка: при eager rebalance консьюмеры вызывают `onPartitionsRevoked` колбэк. Если там не сделать `commitSync` незакоммиченных offset — при rebalance они потеряются и другой консьюмер перечитает сообщения заново (дубли). Всегда коммитьте перед ревокацией партиций.

**→ Уточняющий вопрос:** Какой колбэк нужно реализовать в `ConsumerRebalanceListener` для надёжного коммита offset при rebalance?

## 9. Какие гарантии доставки сообщений предоставляет Kafka?

Kafka из коробки предоставляет три уровня гарантий в зависимости от конфигурации: at-most-once (может потерять, не повторит), at-least-once (не потеряет, но может повторить) и exactly-once (не потеряет и не повторит — через transactional API). Это гарантии на уровне Kafka, не учитывая внешние системы (БД, HTTP-вызовы).

At-most-once достигается авто-коммитом offset до обработки: если сервис упал после коммита, но до обработки — сообщение потеряно. At-least-once — коммит после обработки: если сервис упал после обработки, но до коммита — сообщение будет перечитано и обработано дважды. Exactly-once требует координации между продюсером и консьюмером через транзакции Kafka.

В платёжном сервисе at-least-once допустимо, если операция создания платежа идемпотентна (проверяет уникальность по `paymentId`). Для финансовых операций критично не создать платёж дважды — именно здесь нужна идемпотентность на уровне бизнес-логики или exactly-once семантика.

❗ Ловушка: exactly-once в Kafka — это гарантия within Kafka (между продюсером и брокером, а также в Kafka Streams). Если консьюмер записывает в PostgreSQL, эта транзакция не входит в Kafka EOS. «Exactly-once» в распределённых системах с внешними системами требует дополнительных механизмов (идемпотентные ключи в БД, outbox pattern).

**→ Уточняющий вопрос:** Как обеспечить exactly-once при записи событий из Kafka в PostgreSQL без использования транзакций Kafka?

## 10. В чём разница между at-most-once, at-least-once и exactly-once?

**At-most-once** — сообщение доставляется 0 или 1 раз. Offset коммитится до обработки. При сбое во время обработки сообщение теряется. Подходит для некритичных данных: метрики, логи кликов, где потеря нескольких событий приемлема ради скорости.

**At-least-once** — сообщение доставляется 1 или более раз. Offset коммитится после обработки. При сбое после обработки (но до коммита) сообщение обработается повторно. Это стандартный режим Kafka. Требует идемпотентной обработки на стороне консьюмера. Подходит для большинства бизнес-событий при условии идемпотентности.

**Exactly-once** — ровно один раз. Достигается через idempotent producer + транзакции Kafka (для Kafka Streams или read-process-write в пределах Kafka). При записи во внешние системы requires дополнительных паттернов. Используется в финансовых системах, где важна точность расчётов.

```
Режим          | Потери | Дубли | Скорость | Сложность
at-most-once   | Да     | Нет   | Высокая  | Низкая
at-least-once  | Нет    | Да    | Средняя  | Средняя
exactly-once   | Нет    | Нет   | Низкая   | Высокая
```

❗ Ловушка: «exactly-once в Kafka» часто путают с «exactly-once в системе». Kafka EOS гарантирует, что сообщение будет записано/прочитано в Kafka ровно один раз. Но если консьюмер делает HTTP-запрос к внешнему API или пишет в Redis — это уже вне Kafka EOS. Реальный exactly-once требует идемпотентности на каждом шаге.

**→ Уточняющий вопрос:** Как реализовать идемпотентную обработку события платежа при at-least-once семантике?

## 11. Как настроить exactly-once семантику?

Exactly-once в Kafka состоит из трёх компонентов. Первый: idempotent producer (`enable.idempotence=true`) — брокер отслеживает sequence number и отбрасывает повторные сообщения от одного продюсера. Второй: транзакционный продюсер (`transactional.id`) — атомарная запись в несколько партиций/топиков. Третий: консьюмер с `isolation.level=read_committed` — читает только сообщения из завершённых транзакций.

Транзакционный продюсер при старте регистрируется у transaction coordinator (один из брокеров). Каждая транзакция получает уникальный `producerId` + `epoch`. Все сообщения в транзакции маркируются. `commitTransaction()` отправляет специальный control record в каждую партицию. Консьюмер с `read_committed` пропускает незакоммиченные и прерванные записи.

В Kafka Streams приложение, которое читает `order-events`, агрегирует и пишет в `order-summary`, использует EOS для гарантии: каждое входное сообщение породит ровно одну запись в выходном топике. Без EOS при сбое агрегация может задвоиться.

```java
// Конфигурация транзакционного продюсера
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "payment-producer-1");
props.put(ProducerConfig.ACKS_CONFIG, "all");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("payment-events", key, value));
    producer.send(new ProducerRecord<>("audit-log", key, auditValue));
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
}
```

❗ Ловушка: `transactional.id` должен быть уникальным на уровне инстанса продюсера, но стабильным при рестарте (не UUID). Если два инстанса имеют одинаковый `transactional.id`, второй «зомбифицирует» первого (увеличит epoch), и первый получит ProducerFencedException. В k8s `transactional.id` = имя пода или порядковый номер реплики.

## 12. Что такое offset в Kafka?

Offset — монотонно возрастающий целочисленный идентификатор сообщения внутри конкретной партиции. Каждое новое сообщение получает offset = предыдущий + 1. Это позиция в логе партиции: зная партицию и offset, можно точно идентифицировать любое сообщение. Консьюмер хранит свой «текущий offset» — позицию, до которой он обработал данные.

Внутри Kafka offset хранится в двух местах: как метаданные сообщения в `.log` файлах партиции, и как прогресс Consumer Group в служебном топике `__consumer_offsets`. Последний — обычный Kafka топик с 50 партициями и compaction, где ключ = `(group, topic, partition)`, значение = offset. При запуске консьюмер запрашивает свой offset у координатора группы.

Сервис аудита читает топик `user-actions`. После обработки 1000 событий и записи в Elasticsearch он коммитит offset 1000. При рестарте сервиса он продолжит с 1001, не перечитывая уже обработанные события. При reset offset на `--from-beginning` аудит переобработает всю историю.

```java
// Получение текущего offset консьюмера (для мониторинга)
Map<TopicPartition, Long> endOffsets = consumer.endOffsets(partitions);
Map<TopicPartition, OffsetAndMetadata> committed = consumer.committed(new HashSet<>(partitions));
long lag = endOffsets.get(tp) - committed.get(tp).offset(); // consumer lag
```

❗ Ловушка: offset монотонный, но не обязательно без пропусков. Транзакционные control records тоже занимают offset. Поэтому offset 100 после 99 не гарантирован — может быть 101 (100 занят control record). Не стройте логику на «следующий offset = текущий + 1».

**→ Уточняющий вопрос:** Где физически хранится информация о закоммиченных offset для Consumer Group?

## 13. Как работает commit offset?

Коммит offset — это запись в топик `__consumer_offsets` записи вида `{group=billing-service, topic=order-events, partition=3, offset=1542}`. Это означает «всё до offset 1542 в партиции 3 обработано группой billing-service». При следующем запуске/rebalance консьюмер запросит этот offset и продолжит с 1543.

Коммит происходит через брокера-координатора группы, который является лидером соответствующей партиции `__consumer_offsets`. При `commitSync()` клиент ждёт подтверждения от брокера. При `commitAsync()` — не ждёт, передаёт callback. Автокоммит (`enable.auto.commit=true`) делает это в фоне каждые `auto.commit.interval.ms` (по умолчанию 5000 мс).

В Spring Kafka с ручным подтверждением: после обработки события заказа сервис вызывает `acknowledgment.acknowledge()`, что буферизует коммит. Spring Kafka может коммитить батчами (`BATCH` режим) или после каждого сообщения (`RECORD` режим), управляя балансом между производительностью и точностью.

```java
@KafkaListener(
    topics = "order-events",
    groupId = "billing-service",
    containerFactory = "manualAckListenerContainerFactory"
)
public void handleOrderEvent(
    ConsumerRecord<String, OrderEvent> record,
    Acknowledgment acknowledgment
) {
    try {
        billingService.processPayment(record.value());
        acknowledgment.acknowledge(); // коммит только после успешной обработки
    } catch (NonRetryableException e) {
        // не коммитим — или отправляем в DLQ и потом коммитим
        sendToDlq(record);
        acknowledgment.acknowledge();
    }
}
```

❗ Ловушка: `commitAsync()` без callback не сообщает об ошибках. Если коммит упал (сетевая ошибка, rebalance), приложение не узнает об этом — offset не записан, сообщения будут перечитаны. Всегда передавайте callback в `commitAsync` для логирования и fallback.

**→ Уточняющий вопрос:** В каком порядке нужно вызывать `commitSync` и `commitAsync`, и почему перед закрытием консьюмера лучше использовать `commitSync`?

## 14. В чём разница между auto commit и manual commit?

**Auto commit** (`enable.auto.commit=true`, `auto.commit.interval.ms=5000`): Kafka-клиент сам периодически в фоне коммитит наибольший offset среди вернувшихся из `poll()` сообщений. Удобно, требует минимум кода, но опасно: коммит происходит по таймеру, независимо от того, обработаны ли сообщения.

**Manual commit** (`enable.auto.commit=false`): приложение явно вызывает `commitSync()` или `commitAsync()` после обработки. Полный контроль: коммит только после гарантированной обработки. Сложнее реализовать, но надёжнее. В Spring Kafka это `AckMode.MANUAL`, `AckMode.MANUAL_IMMEDIATE`, `AckMode.RECORD`, `AckMode.BATCH`.

Сценарий проблемы с auto commit: консьюмер получил 100 сообщений в `poll()`, начал обрабатывать. Через 5 секунд автокоммит записал offset 100. На сообщении 57 сервис упал. После рестарта читает с 101 — сообщения 57-100 потеряны. С manual commit: коммит произошёл бы только после обработки всех 100 (или каждого), упав на 57 — при рестарте прочитает с 57.

```yaml
# application.yml — отключить auto commit
spring:
  kafka:
    consumer:
      enable-auto-commit: false
      auto-offset-reset: earliest
    listener:
      ack-mode: MANUAL  # или RECORD, BATCH
```

❗ Ловушка: auto commit не коммитит после каждого сообщения — он коммитит последний offset из `poll()` каждые N миллисекунд. Если `poll()` вернул 500 сообщений, а обработка зависла на 3-м, автокоммит всё равно через 5 сек закоммитит все 500. При следующем рестарте 497 сообщений потеряны. Это классический вопрос на собесе.

**→ Уточняющий вопрос:** Что произойдёт, если вы вызовете `commitAsync()` внутри цикла обработки сообщений, но исключение будет выброшено до коммита?

## 15. Что такое rebalancing и когда он происходит?

Rebalancing — это протокол перераспределения партиций топика между консьюмерами одной группы. Во время rebalance каждый консьюмер временно теряет свои партиции (при eager rebalance) или часть из них (при cooperative). Цель: обеспечить, что каждая партиция назначена ровно одному активному консьюмеру.

Триггеры rebalance: новый консьюмер присоединяется к группе; консьюмер выходит (graceful shutdown — LeaveGroup request); консьюмер не прислал heartbeat за `session.timeout.ms`; консьюмер не вызвал `poll()` за `max.poll.interval.ms`; изменилось число партиций топика; изменилась подписка консьюмера.

В production самая частая причина «неожиданного» rebalance — `max.poll.interval.ms` (по умолчанию 5 минут). Если обработка батча занимает больше 5 минут (например, запись 10 000 записей в БД под нагрузкой), брокер считает консьюмер «мёртвым», исключает из группы, инициирует rebalance. Другой консьюмер подхватит эти партиции и перечитает те же сообщения — дубли в БД.

```yaml
spring:
  kafka:
    consumer:
      properties:
        max.poll.interval.ms: 600000      # увеличить если обработка долгая
        max.poll.records: 50              # уменьшить батч чтобы уложиться в таймаут
        session.timeout.ms: 30000        # время без heartbeat до исключения
        heartbeat.interval.ms: 10000     # частота heartbeat (< session.timeout / 3)
```

❗ Ловушка: `session.timeout.ms` и `max.poll.interval.ms` — разные вещи. `session.timeout.ms` — таймаут на heartbeat (фоновый поток). `max.poll.interval.ms` — таймаут на вызов `poll()` (основной поток). При долгой обработке в основном потоке heartbeat продолжает биться, но `max.poll.interval.ms` истекает — и это тоже rebalance.

**→ Уточняющий вопрос:** Как избежать rebalance из-за `max.poll.interval.ms` при долгой обработке сообщений?

## 16. Что такое репликация в Kafka?

Репликация — механизм хранения каждой партиции на нескольких брокерах. `replication.factor=3` означает 3 копии партиции на 3 разных брокерах. Одна из них — leader (принимает все операции чтения/записи), остальные — followers (асинхронно реплицируют). При падении лидера один из followers становится новым лидером.

Follower реплицирует, вытягивая (fetch) данные от лидера — аналогично обычному консьюмеру. Лидер хранит High Watermark (HW) — минимальный offset, до которого все ISR реплики синхронизированы. Консьюмеры читают только до HW, что гарантирует: прочитанное сообщение не «исчезнет» при последующей смене лидера.

В кластере из 5 брокеров топик `payment-events` с `replication.factor=3` и 12 партициями: каждый брокер хранит примерно по ~7 партиций (некоторые как лидер, некоторые как follower). При падении одного брокера 3-4 его лидирующих партиции переходят к другим брокерам из ISR — всего за секунды, без потери данных.

```yaml
# Создание топика с репликацией
bin/kafka-topics.sh --create \
  --topic payment-events \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --bootstrap-server localhost:9092
```

❗ Ловушка: `replication.factor` не может превышать число брокеров. В dev-окружении с 1 брокером `replication.factor=3` вызовет ошибку. В prod с 3 брокерами и `replication.factor=3` потеря любого брокера не приведёт к потере данных, но потеря двух из трёх (при `min.insync.replicas=2`) сделает запись невозможной (NotEnoughReplicasException).

## 17. Что такое leader и follower реплики?

**Leader** — единственная реплика партиции, которая обслуживает все операции чтения и записи от продюсеров и консьюмеров. Только leader принимает ProduceRequest от продюсеров и FetchRequest от консьюмеров (по умолчанию). Leader поддерживает список ISR и High Watermark.

**Follower** — пассивная реплика, которая непрерывно отправляет FetchRequest к лидеру, получает новые сообщения и применяет их к своему локальному логу. Follower не обслуживает клиентов. Если follower отстаёт больше чем на `replica.lag.time.max.ms` (по умолчанию 30 сек), он исключается из ISR.

В платёжном кластере партиция 3 топика `payment-events` — лидер на broker-2. Все 15 000 транзакций/с пишутся через broker-2. Broker-1 и broker-3 — followers, реплицируют данные. При planned maintenance broker-2 партиция 3 переедет на broker-1 или broker-3 из ISR — clients автоматически перенаправятся через metadata update.

Kafka 2.4+ поддерживает `fetch.from.nearest.replica` (Follower Fetching) — консьюмер может читать с ближайшей реплики в том же AZ, снижая межзональный трафик. Это не влияет на гарантии, так как читается до High Watermark.

❗ Ловушка: у лидера и фолловера одни и те же данные, но пишет/читает только лидер. Частая ошибка — предположить, что «чтение распределяется между репликами» (нет, если не включён follower fetching). Весь read/write трафик одной партиции идёт через одного брокера — лидера.

**→ Уточняющий вопрос:** Что происходит с производительностью, если все лидирующие партиции кластера сосредоточены на одном брокере?

## 18. Что такое ISR (In-Sync Replicas)?

ISR (In-Sync Replicas) — динамический список реплик партиции, которые достаточно близко синхронизированы с лидером. «Достаточно» означает: реплика не отставала от лидера дольше `replica.lag.time.max.ms`. Только реплика из ISR может стать новым лидером при failover, что гарантирует отсутствие потери данных.

Лидер хранит список ISR в ZooKeeper/KRaft. При каждом FetchRequest от follower'а лидер проверяет, укладывается ли тот в лимит по времени. Если follower задержался (GC pause, network issue, перегрузка диска) — лидер вычёркивает его из ISR. Как только follower догнал лидера — добавляется обратно. High Watermark = минимальный log end offset среди всех реплик в ISR.

В prod-кластере топика `order-events` с `replication.factor=3` и `min.insync.replicas=2`: нормальный режим — ISR=[broker1(leader), broker2, broker3]. Broker3 лаганул (долгий GC) — ISR=[broker1, broker2]. Запись продолжается (есть 2 из min 2). Broker2 тоже упал — ISR=[broker1]. `min.insync.replicas=2` не выполнен — продюсер с `acks=all` получает `NotEnoughReplicasException`.

❗ Ловушка: `acks=all` означает «подтверждение от всех ISR», а не «от всех реплик». Если ISR сократился до 1 (только лидер), `acks=all` = подтверждение только от лидера — надёжность падает. `min.insync.replicas=2` — страховка: если ISR < 2, запись блокируется, а не записывается с пониженными гарантиями.

## 19. Как Kafka обеспечивает отказоустойчивость?

Отказоустойчивость Kafka строится на нескольких уровнях. Репликация данных: каждая партиция хранится на нескольких брокерах (`replication.factor`). Автоматическое переключение лидера: при падении брокера ZooKeeper/KRaft инициирует leader election для затронутых партиций — новый лидер выбирается из ISR за секунды. Дисковая надёжность: sequential write на диск, fsync политика, сегментные файлы.

При падении брокера Kafka Controller (один из брокеров) обнаруживает это через ZooKeeper/KRaft сессию. Controller запускает leader election для всех партиций, у которых упавший брокер был лидером. Новый лидер — первый из ISR. Clients получают обновлённые metadata при следующем запросе — продюсеры и консьюмеры автоматически переключаются на нового лидера. End-to-end latency spike — секунды, не минуты.

Для сервиса платежей с `replication.factor=3`, `min.insync.replicas=2`, `acks=all`: потеря одного брокера из трёх — нулевая потеря данных, автоматическое восстановление за ~10-30 секунд. Потеря двух брокеров — запись блокируется (NotEnoughReplicasException), читать можно с оставшегося. Это сознательный выбор: остановить запись лучше, чем потерять финансовые данные.

```yaml
# Конфигурация для максимальной отказоустойчивости
# На уровне топика:
min.insync.replicas=2
replication.factor=3

# На уровне продюсера:
acks=all
enable.idempotence=true
retries=2147483647
delivery.timeout.ms=120000
```

❗ Ловушка: Kafka не защищает от split-brain при сетевом разделении (network partition). Если брокер изолирован от ZooKeeper, но доступен клиентам — может возникнуть ситуация с несколькими лидерами (до истечения ZK сессии). В KRaft (Kafka без ZooKeeper, Kafka 3.x) этот риск уменьшен благодаря встроенному Raft-консенсусу.

**→ Уточняющий вопрос:** Что происходит с продюсером при `acks=all`, когда ISR сокращается ниже `min.insync.replicas`?

## 20. Что такое producer acknowledgment и какие режимы существуют (acks=0,1,all)?

`acks` — конфигурация продюсера, определяющая, сколько брокеров должны подтвердить получение сообщения, прежде чем продюсер считает отправку успешной. Это главный рычаг баланса между производительностью и надёжностью.

**acks=0**: продюсер не ждёт никакого подтверждения — fire and forget. Максимальная скорость, нулевая надёжность. Потеря при сбое сети или брокера гарантирована. Применение: метрики кликов, некритичная телеметрия, где потеря 0.1% событий приемлема.

**acks=1**: ждёт подтверждения от лидера партиции. Лидер записал в свой лог — OK. Followers ещё могут не иметь сообщение. Если лидер упал сразу после ACK и до репликации — сообщение потеряно. Баланс: чуть медленнее acks=0, гораздо быстрее acks=all.

**acks=all (acks=-1)**: ждёт подтверждения от всех ISR реплик. Максимальная надёжность. В сочетании с `min.insync.replicas=2` и `replication.factor=3` — сообщение выживет при потере любого одного брокера. Применение: финансовые транзакции, заказы, платежи.

```java
// Spring Kafka Producer config для payment-events
@Bean
public ProducerFactory<String, PaymentEvent> producerFactory() {
    Map<String, Object> config = new HashMap<>();
    config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
    config.put(ProducerConfig.ACKS_CONFIG, "all");
    config.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
    config.put(ProducerConfig.RETRIES_CONFIG, Integer.MAX_VALUE);
    config.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 5);
    return new DefaultKafkaProducerFactory<>(config);
}
```

❗ Ловушка: `acks=1` не означает «записано надёжно». Лидер мог подтвердить и тут же упасть до репликации. Для платёжных систем `acks=1` — неприемлемый риск. На собесе спрашивают: «acks=all медленнее, но насколько?» — latency растёт пропорционально времени репликации, обычно +5-20 мс, что для большинства use case приемлемо.

**→ Уточняющий вопрос:** Как `MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION` влияет на порядок сообщений при `acks=1` с ретраями?

## 21. Что такое batch в Kafka producer?

Батч — это группа сообщений для одной партиции, накапливаемая в памяти продюсера и отправляемая одним сетевым запросом. Вместо N сетевых вызовов для N сообщений — один вызов для батча. Это кардинально увеличивает throughput и снижает load на брокер.

Управляется двумя параметрами: `batch.size` (байты, по умолчанию 16384) — максимальный размер батча для партиции; `linger.ms` (по умолчанию 0) — сколько ждать дополнительных сообщений перед отправкой неполного батча. `linger.ms=0` — отправить как можно быстрее (но всё равно может накопить несколько сообщений за время одного network round-trip). `linger.ms=10` — ждать 10мс для набора большего батча.

В сервисе аудит-лога, который пишет 100 000 событий/с: `batch.size=65536, linger.ms=5` позволяет сформировать батчи по 4-8 МБ и отправлять их вместо 100 000 мелких запросов. Throughput вырастает в 10-50x. Латентность увеличивается на `linger.ms` (5 мс) — для аудита это приемлемо, для real-time нотификаций — нет.

```java
// Конфигурация батчинга для высокого throughput
config.put(ProducerConfig.BATCH_SIZE_CONFIG, 65536);        // 64KB
config.put(ProducerConfig.LINGER_MS_CONFIG, 5);             // ждать 5мс
config.put(ProducerConfig.BUFFER_MEMORY_CONFIG, 67108864);  // 64MB buffer
config.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4"); // сжать батч
```

❗ Ловушка: `batch.size` — это лимит, а не гарантированный размер. Батч отправляется при достижении `batch.size` ИЛИ истечении `linger.ms`, что наступит раньше. При `linger.ms=0` батч с одним сообщением будет отправлен немедленно. Также: батч формируется per-partition, поэтому при 12 партициях и 16KB на батч буфер может занять 192KB только на один продюсер.

## 22. Как работает компрессия сообщений?

Компрессия в Kafka происходит на уровне батча на стороне продюсера. Продюсер формирует батч, сжимает весь батч целиком (`gzip`, `snappy`, `lz4`, `zstd`), отправляет на брокер. Брокер хранит данные в сжатом виде — не распаковывает. Консьюмер получает сжатый батч и распаковывает на своей стороне. Брокер экономит и дисковое место, и сетевой трафик.

Алгоритмы: `gzip` — лучшее сжатие, но CPU-интенсивен; `snappy` — быстрее, сжимает хуже; `lz4` — самый быстрый, хорошее сжатие (рекомендован для большинства случаев); `zstd` (Kafka 2.1+) — лучший баланс: сжатие как у gzip, скорость близко к lz4.

JSON-события обычно сжимаются в 5-10x. Топик `user-activity-events` без компрессии: 10 ГБ/час дисковой записи, 80 Мбит/с сетевого трафика. С `lz4`: ~1.5 ГБ/час дисковой записи, ~12 Мбит/с трафика. Критично для retention: при хранении на 7 дней разница — 1.5 ТБ против 168 ГБ.

```java
config.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");
// Компрессия per-broker (override producer settings):
// compression.type=producer — брокер хранит как получил от продюсера
// compression.type=lz4      — брокер принудительно перекомпрессирует
```

❗ Ловушка: если продюсер отправляет `snappy`, а брокер сконфигурирован с `compression.type=gzip` — брокер распакует и перепакует. Это CPU-overhead на брокере. Рекомендация: устанавливать `compression.type=producer` на брокере и управлять компрессией только на стороне продюсера. Также: сжатие работает только для батчей — одиночные сообщения не выигрывают.

**→ Уточняющий вопрос:** Влияет ли компрессия на время сериализации/десериализации и как это учитывать при выборе алгоритма?

## 23. Что такое idempotent producer?

Idempotent producer — продюсер с гарантией, что повторная отправка одного и того же сообщения (при retry) не создаст дубликат в партиции. Включается через `enable.idempotence=true`. Kafka присваивает продюсеру уникальный `PID` (Producer ID) и присваивает каждому сообщению sequence number. Брокер отслеживает последний принятый sequence number от данного PID для каждой партиции и отбрасывает повторы.

Без idempotence: сообщение отправлено, брокер записал, но ACK потерялся в сети. Продюсер делает retry — брокер записывает снова. Дубль. С idempotence: retry приходит с тем же PID + sequence number — брокер видит, что уже обработал, отвечает OK, но не записывает. Гарантия exactly-once в пределах одной партиции для одного producer сессии.

В сервисе создания заказов: метод `createOrder()` отправляет событие в Kafka. Сеть лагает, retry x3. Без idempotence — три дубля в `order-events`, три SMS клиенту «заказ создан». С idempotence — один раз в топике, одно SMS.

```java
// Включение idempotence (обязательно влечёт acks=all, retries=MAX_INT)
config.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
// При enable.idempotence=true Kafka автоматически выставляет:
// acks=all, retries=Integer.MAX_VALUE, max.in.flight.requests.per.connection<=5
```

❗ Ловушка: idempotence работает только в пределах одной producer сессии. Если сервис рестартанул, PID изменится — старые sequence numbers забыты, дублей при retry из новой сессии idempotence не предотвратит. Для cross-session exactly-once нужны транзакции. Также: idempotence не защищает от дублей на уровне бизнес-логики (дважды вызвали `send()` из кода).

**→ Уточняющий вопрос:** Чем idempotent producer отличается от транзакционного продюсера и когда недостаточно одного idempotence?

## 24. Как обрабатывать ошибки при чтении сообщений?

Ошибки при чтении делятся на: retriable (временные — БД недоступна, timeout) и non-retriable (постоянные — невалидный JSON, нарушение схемы, бизнес-ошибка). Для retriable — retry с exponential backoff, не коммитя offset. Для non-retriable — отправка в DLQ и коммит offset (чтобы не блокировать партицию).

Если не обработать non-retriable ошибку, один «битый» message заблокирует всю партицию навсегда: `while(true) { poll(); fail; }`. Весь backlog за этой партицией не обрабатывается. Consumer lag растёт. Сервис деградирует. Именно поэтому нужен явный error handling с DLQ.

В сервисе уведомлений: пришло событие с невалидным email (null). Отправка не имеет смысла. Сообщение идёт в топик `notification-events-dlq` с заголовками: `original-topic`, `original-partition`, `original-offset`, `error-message`, `error-timestamp`. Offset коммитится. Основная обработка продолжается. DLQ-сообщения разбирает дежурная команда.

```java
@KafkaListener(topics = "order-events", groupId = "notification-service")
public void handleOrderEvent(ConsumerRecord<String, OrderEvent> record,
                             Acknowledgment acknowledgment) {
    try {
        notificationService.send(record.value());
        acknowledgment.acknowledge();
    } catch (InvalidEmailException e) {
        // Non-retriable: отправляем в DLQ и коммитим
        dlqPublisher.send("order-events-dlq", record, e);
        acknowledgment.acknowledge();
    } catch (MailServiceUnavailableException e) {
        // Retriable: НЕ коммитим, пусть перечитается
        throw e; // Spring Kafka retry с backoff
    }
}
```

❗ Ловушка: бесконечный retry retriable ошибок — это denial of service самому себе. При недоступности БД на 10 минут миллионы retry создадут шторм запросов при её восстановлении. Используйте exponential backoff + max attempts. В Spring Kafka: `SeekToCurrentErrorHandler` с `FixedBackOff` или `ExponentialBackOff`.

## 25. Что такое DLQ (Dead Letter Queue)?

DLQ (Dead Letter Queue) — это специальный топик (или очередь в других системах), куда направляются сообщения, которые не удалось обработать после N попыток. Назначение: не потерять данные, не заблокировать обработку основного топика, сохранить проблемные сообщения для анализа и ручного реплея.

По соглашению DLQ-топик называется `<original-topic>-dlt` или `<original-topic>-dlq`. Сообщение в DLQ дополняется заголовками: оригинальный топик/партиция/offset, stacktrace, количество попыток, timestamp ошибки. Это позволяет потом понять, откуда пришло сообщение и почему упало.

В сервисе платежей: событие `payment-requested` с некорректной суммой (отрицательное значение) не пройдёт валидацию. После 3 попыток (на случай временной ошибки) уходит в `payment-requested-dlq`. Команда мониторит размер DLQ через алерт в Grafana. Инженер смотрит сообщение, фиксит баг продюсера, replay'ит исправленное событие вручную.

```java
// Spring Kafka — автоматический DLQ через DeadLetterPublishingRecoverer
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<String, Object> kafkaTemplate) {
    DeadLetterPublishingRecoverer recoverer =
        new DeadLetterPublishingRecoverer(kafkaTemplate,
            (record, ex) -> new TopicPartition(
                record.topic() + "-dlt",  // payment-events → payment-events-dlt
                record.partition()         // сохраняем партицию для порядка
            ));

    ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
    backOff.setMaxAttempts(3);

    return new DefaultErrorHandler(recoverer, backOff);
}
```

❗ Ловушка: DLQ без мониторинга — это корзина забытых данных. Если алерт на DLQ не настроен, сообщения тихо копятся месяцами. В системе заказов это может означать сотни незавершённых транзакций. Обязательно: алерт при появлении любого сообщения в DLQ или при превышении порога lag.

## 26. Как мониторить lag консьюмера?

Consumer lag = `endOffset(partition) - committedOffset(partition)` — сколько сообщений в партиции ещё не обработано группой. Это ключевая метрика здоровья Kafka-консьюмера. Нулевой lag — консьюмер не отстаёт. Растущий lag — консьюмер не справляется с нагрузкой или завис. Lag = несколько тысяч при пиках — норма, если потом сходит на нет.

Инструменты мониторинга: `kafka-consumer-groups.sh --describe` (базовый CLI), JMX метрики (`records-lag-max`, `records-lag-avg`), Prometheus + kafka-exporter (наиболее распространённый в k8s), Burrow (LinkedIn, специализированный lag monitor с историей трендов), Grafana dashboard поверх любого из них.

В сервисе обработки заказов: lag топика `order-events` вырос с 0 до 500 000 за 10 минут — алерт. Причина: один из 4 консьюмеров завис на `max.poll.interval.ms` из-за долгого запроса к БД. Rebalance произошёл, но оставшиеся 3 консьюмера не справляются с 12 партициями при пиковой нагрузке. Решение: добавить инстансы или уменьшить `max.poll.records`.

```java
// Программный мониторинг lag через AdminClient
try (AdminClient adminClient = AdminClient.create(props)) {
    Map<TopicPartition, OffsetAndMetadata> committed =
        adminClient.listConsumerGroupOffsets("billing-service")
            .partitionsToOffsetAndMetadata().get();

    Map<TopicPartition, ListOffsetsResult.ListOffsetsResultInfo> endOffsets =
        adminClient.listOffsets(committed.keySet().stream()
            .collect(Collectors.toMap(tp -> tp,
                tp -> OffsetSpec.latest()))).all().get();

    committed.forEach((tp, meta) -> {
        long lag = endOffsets.get(tp).offset() - meta.offset();
        meterRegistry.gauge("kafka.consumer.lag",
            Tags.of("topic", tp.topic(), "partition",
                String.valueOf(tp.partition())), lag);
    });
}
```

❗ Ловушка: lag=0 не всегда хорошо. Если топик пуст (нет продюсеров), lag=0, но сервис может быть мёртв. Мониторьте и `records-consumed-rate` — количество обработанных сообщений в секунду. Также: lag в байтах информативнее, чем lag в числе сообщений, если сообщения сильно разного размера.

**→ Уточняющий вопрос:** Как отличить ситуацию «консьюмер отстаёт из-за пиковой нагрузки» от «консьюмер завис и не читает вообще»?

## 27. Что такое retention policy?

Retention policy — правило, определяющее, как долго Kafka хранит сообщения в топике. Два режима: **delete** (по умолчанию) — сообщения удаляются по достижении порога; **compact** — хранится только последнее сообщение для каждого ключа (event sourcing / state store). Также можно комбинировать: `cleanup.policy=compact,delete`.

**Delete по времени**: `log.retention.hours=168` (7 дней) — сегменты лога старше 7 дней удаляются. Точнее: `log.retention.ms`, `log.retention.minutes`. Важно: удаляются целые сегменты, а не отдельные сообщения. Если активный сегмент содержит старое сообщение, он не удалится, пока не станет неактивным.

**Delete по размеру**: `log.retention.bytes=-1` (по умолчанию нет лимита по размеру). При установке: если суммарный размер партиции превышает лимит, старые сегменты удаляются. Применяется вместе с временным лимитом (удаляется при срабатывании любого).

Сценарии: топик `user-clickstream` — retention 24 часа (данные нужны только для real-time аналитики). Топик `order-events` — retention 30 дней (аудит, replay при инцидентах). Топик `user-profile-updates` — cleanup.policy=compact (всегда актуальное состояние профиля, нет ограничений по времени).

```yaml
# kafka-topics.sh или kafka AdminClient
bin/kafka-topics.sh --alter \
  --topic order-events \
  --config retention.ms=2592000000 \     # 30 дней
  --config retention.bytes=-1 \           # без лимита по размеру
  --bootstrap-server localhost:9092
```

❗ Ловушка: retention не означает «сообщение удалится ровно через N часов». Удаление происходит при запуске log cleaner (`log.cleaner.threads`) и затрагивает только неактивные сегменты. Задержка удаления может быть значительной. Не рассчитывайте retention как точный дедлайн — данные могут жить дольше (но не короче).

**→ Уточняющий вопрос:** В чём разница между cleanup.policy=delete и cleanup.policy=compact и когда применять каждый?

## 28. Как удаляются старые сообщения из топика?

Kafka хранит данные в сегментах — файлах фиксированного размера или временного диапазона. Активный сегмент (куда идёт запись) никогда не удаляется. Неактивные сегменты проверяются log cleaner потоком. При `cleanup.policy=delete`: если последний timestamp сегмента старше `retention.ms` или размер всей партиции превышает `retention.bytes` — сегмент удаляется целиком через `Files.delete()`.

При `cleanup.policy=compact`: log cleaner анализирует все неактивные сегменты, для каждого ключа определяет «tail» (самое свежее значение). Старые записи с тем же ключом заменяются tombstone записями (key + null value), затем tombstone тоже удаляются. Процесс называется log compaction. Compaction не затрагивает активный сегмент — только «хвост» лога.

Для топика `user-preferences` с compaction: пользователь 42 обновил тему с `light` на `dark` на `system`. В топике три записи с ключом `42`. После compaction останется только `42 → system`. При чтении нового консьюмера с начала топика он увидит только актуальные значения всех пользователей — полный snapshot состояния.

```
Сегментные файлы партиции:
00000000000000000000.log  ← может удалиться если старый
00000000000000012345.log  ← может удалиться если старый
00000000000000098765.log  ← активный (запись идёт сюда), не удаляется

Управление сегментами:
log.segment.bytes=1073741824  (1GB — создать новый сегмент)
log.roll.ms=604800000         (7 дней — создать новый сегмент по времени)
```

❗ Ловушка: compaction — асинхронный, фоновый процесс. Дубликаты ключей могут существовать в топике часами до очистки. Если консьюмер рассчитывает на то, что «в compacted топике каждый ключ встречается один раз» — он ошибается. Гарантируется только, что после полного compaction останется последнее значение. Во время чтения могут быть промежуточные.

**→ Уточняющий вопрос:** Как tombstone-записи (ключ + null) работают в compacted топике и зачем они нужны?

## 29. Можно ли читать сообщения из определённой партиции?

Да. Kafka предоставляет два API: `subscribe(topics)` — подписка на топик с автоматическим управлением партициями через Consumer Group; `assign(partitions)` — ручное назначение конкретных партиций без участия в Consumer Group и rebalance. При `assign()` `group.id` игнорируется для балансировки, но offset всё равно коммитится в `__consumer_offsets` если задан `group.id`.

С `assign()` можно также явно установить offset через `seek(partition, offset)`, `seekToBeginning()`, `seekToEnd()`. Это мощно для сценариев: переобработка конкретного диапазона событий, forensic analysis, реализация собственного управления партициями (например, в Kafka Streams или Flink).

Сценарий: incident investigation — нужно переобработать платежи из партиции 7 с offset 100500 до 200000 (сбой обработки за прошлую ночь). Без влияния на остальные партиции и текущих консьюмеров:

```java
KafkaConsumer<String, PaymentEvent> consumer = new KafkaConsumer<>(props);

TopicPartition partition = new TopicPartition("payment-events", 7);
consumer.assign(Collections.singletonList(partition));
consumer.seek(partition, 100500L); // начать с конкретного offset

while (true) {
    ConsumerRecords<String, PaymentEvent> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, PaymentEvent> record : records) {
        if (record.offset() > 200000L) break; // обработать только нужный диапазон
        reprocessPayment(record.value());
    }
}
```

❗ Ловушка: при `assign()` нет автоматического rebalance — если назначенный брокер с партицией недоступен, консьюмер не получит партицию от другого брокера автоматически. Ответственность за failover — на приложении. Также: смешивать `subscribe()` и `assign()` в одном консьюмере нельзя — IllegalStateException.

## 30. Как реализовать фильтрацию сообщений на стороне консьюмера?

В Kafka нет server-side filtering — брокер отдаёт все сообщения партиции консьюмеру, фильтрация происходит только на клиенте. Это означает: сетевой трафик и CPU десериализации тратятся даже на ненужные сообщения. Для систем с высоким процентом отфильтрованных сообщений это существенный overhead.

Три стратегии решения. **Фильтрация в консьюмере** — просто и работает при невысоком проценте отфильтрованных. **Отдельные топики** — события сегментируются при записи (продюсер пишет в `order-events-eu`, `order-events-us`) — консьюмер подписывается только на нужный. **Kafka Streams / ksqlDB** — роутинг потоков: один стрим читает `order-events`, фильтрует и пишет в `order-events-high-value` (сумма > 10000). Консьюмеры подписываются на специализированные топики.

В сервисе премиум-уведомлений: нужно обрабатывать только события `ORDER_COMPLETED` с суммой > 50 000 руб. Если такие события составляют 5% от всех, консьюмер читает 100%, десериализует 100%, обрабатывает 5%. При 1 млн событий/час — 950 000 «холостых» десериализаций. Kafka Streams-топология решает это на уровне потока.

```java
// Вариант 1: простая фильтрация в @KafkaListener (Spring Kafka)
@KafkaListener(topics = "order-events", groupId = "premium-notifications")
public void handle(OrderEvent event) {
    if (!"ORDER_COMPLETED".equals(event.getType())
            || event.getAmount().compareTo(BigDecimal.valueOf(50000)) < 0) {
        return; // игнорировать, но offset всё равно двигается
    }
    premiumNotificationService.send(event);
}

// Вариант 2: Kafka Streams — фильтрация на уровне потока
StreamsBuilder builder = new StreamsBuilder();
builder.<String, OrderEvent>stream("order-events")
    .filter((key, event) ->
        "ORDER_COMPLETED".equals(event.getType())
        && event.getAmount().compareTo(BigDecimal.valueOf(50000)) >= 0)
    .to("order-events-premium"); // отдельный топик для premium consumer
```

❗ Ловушка: фильтрация в консьюмере не освобождает от необходимости коммитить offset. Даже если сообщение проигнорировано, offset должен продвигаться — иначе при рестарте консьюмер перечитает все отфильтрованные сообщения заново. Spring Kafka с `@KafkaListener` коммитит offset автоматически после возврата из метода, включая ранний return.

**→ Уточняющий вопрос:** В каких случаях выгоднее создать отдельный топик для отфильтрованных событий вместо фильтрации на стороне консьюмера?
