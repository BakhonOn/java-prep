## 1. Что такое иммутабельный (неизменяемый) объект?

Иммутабельный объект — это объект, чьё состояние не может быть изменено после создания. Любая операция, которая «изменяет» значение, фактически возвращает новый объект с нужным состоянием, оставляя оригинал нетронутым. Классические примеры в Java: `String`, `Integer`, `BigDecimal`, `LocalDate`.

Механизм иммутабельности строится на нескольких правилах одновременно: поля объявлены `private final`, публичные методы не меняют состояние, а мутабельные зависимости (коллекции, даты, массивы) копируются при принятии и возврате. Ни одного правила в отдельности недостаточно — только их совокупность гарантирует неизменность.

На практике иммутабельные объекты встречаются везде: DTO в REST API (например, `OrderRequest` с `final`-полями), value-объекты в DDD (Money, Email, UserId), ключи в `HashMap`, результаты из кэша Guava/Caffeine. `LocalDate.of(2024, 1, 1).plusDays(7)` возвращает новый объект — оригинал не тронут.

❗ Ловушка собеса: многие отвечают «сделать все поля `final`» и считают задачу решённой. Но `final List<String> list` запрещает только переприсвоение ссылки — `list.add("x")` работает без ошибок. Иммутабельность поля ≠ иммутабельность объекта, на который оно указывает.

## 2. Какие преимущества даёт использование иммутабельных объектов?

Главное преимущество — безопасность в многопоточной среде без синхронизации. Если объект не меняется, несколько потоков могут читать его одновременно без `synchronized`, `volatile` или блокировок. Именно поэтому `String` безопасно передавать между потоками напрямую.

Второй ключевой плюс — надёжность как ключей в `HashMap`/`HashSet`. Хэш-код иммутабельного объекта стабилен на протяжении всей его жизни, поэтому ключ всегда находится в правильном бакете. Мутабельный ключ при изменении «теряется» в карте. Также иммутабельные объекты безопасно кэшировать: Guava Cache или Spring `@Cacheable` могут хранить иммутабельный объект без страха, что кто-то его испортит.

Дополнительные преимущества: упрощение рассуждений о коде (нет скрытых изменений состояния), безопасная передача между методами и сервисами без копирования, возможность кэшировать `hashCode()` (как делает `String`), естественная поддержка функционального стиля и Stream API.

❗ Ловушка собеса: интервьюер может спросить «а как ты тогда обновишь объект?». Правильный ответ — создать новый объект с изменёнными полями через Builder или `wither`-методы (`withName("...")` возвращает копию с новым именем). Record в Java 16+ поддерживает это через `record.withX(...)` при ручной реализации или через `with`-паттерн.

## 3. Как создать иммутабельный класс в Java?

Иммутабельный класс требует соблюдения пяти правил одновременно: объявить класс `final` (запрет наследования), все поля сделать `private final`, не предоставлять сеттеры, инициализировать все поля только в конструкторе, и выполнять defensive copy для мутабельных полей — как при принятии, так и при возврате.

```java
public final class Money {
    private final BigDecimal amount;
    private final String currency;
    private final List<String> tags;

    public Money(BigDecimal amount, String currency, List<String> tags) {
        this.amount = Objects.requireNonNull(amount);
        this.currency = Objects.requireNonNull(currency);
        this.tags = List.copyOf(tags); // defensive copy на входе
    }

    public BigDecimal getAmount() { return amount; }
    public String getCurrency() { return currency; }

    public List<String> getTags() {
        return tags; // List.copyOf уже возвращает неизменяемый список
    }

    public Money withAmount(BigDecimal newAmount) {
        return new Money(newAmount, this.currency, this.tags);
    }
}
```

Обратите внимание: `List.copyOf()` создаёт неизменяемую копию, поэтому геттер может возвращать её напрямую без дополнительной обёртки. Если бы поле хранилось как обычный `ArrayList`, то геттер обязан возвращать либо `Collections.unmodifiableList(tags)`, либо `new ArrayList<>(tags)`.

❗ Ловушка собеса: класс не объявлен `final`, и интервьюер спрашивает «что будет, если создать подкласс и добавить сеттер?». Подкласс сможет нарушить иммутабельность — для полиморфного кода объект будет казаться иммутабельным, но фактически изменяемым. Именно поэтому `final` на классе — обязательное правило.

## 4. Почему класс String является иммутабельным?

`String` сделан иммутабельным по нескольким взаимосвязанным причинам. Первая — безопасность: строки часто используются как имена классов в `Class.forName()`, пути к файлам, параметры подключения к БД. Если бы строка была мутабельной, злоумышленник мог бы передать безопасное значение, а потом изменить его до момента использования.

Вторая причина — String Pool. JVM может хранить один объект строки для всех литералов с одинаковым значением (`String a = "hello"; String b = "hello";` — это один объект). Это безопасно только потому, что строка не меняется: изменение через ссылку `a` не затронет `b`. Третья причина — кэширование `hashCode`: `String` вычисляет хэш-код при первом вызове и кэширует его в поле `hash`. Это возможно только при гарантии неизменности содержимого.

```java
String s = "hello";
// s.hash вычисляется один раз и кэшируется
int h1 = s.hashCode(); // вычислен
int h2 = s.hashCode(); // взят из кэша
```

❗ Ловушка собеса: «Строка хранит символы в `char[]` — массив же мутабельный, как String иммутабелен?». Ответ: массив `value` объявлен `private final` и никогда не возвращается наружу — только копии или результаты операций. Внешний код не имеет доступа к нему (кроме рефлексии).

**→ Уточняющий вопрос:** Что произойдёт, если изменить строку через рефлексию, обратившись к внутреннему массиву?

## 5. Какие последствия иммутабельности String?

Главное практическое последствие — каждая операция «изменения» (`+`, `substring`, `replace`, `toUpperCase`) создаёт новый объект в куче. Конкатенация строк в цикле через `+` — классическая ошибка производительности: `n` итераций создают `O(n)` промежуточных объектов и суммарно копируют `O(n²)` символов. Решение — `StringBuilder` или `StringJoiner`.

```java
// Плохо: O(n²) копирований
String result = "";
for (String s : list) result += s;

// Хорошо: O(n) через StringBuilder
StringBuilder sb = new StringBuilder();
for (String s : list) sb.append(s);
String result = sb.toString();
```

С другой стороны, иммутабельность позволяет безопасно передавать строки между потоками без копирования, использовать их как ключи в `HashMap` (хэш кэшируется), и переиспользовать через String Pool. `intern()` позволяет явно добавить строку в пул и получить ссылку на общий объект.

❗ Ловушка собеса: компилятор Java оптимизирует конкатенацию строковых литералов (`"a" + "b"` → `"ab"` на этапе компиляции), а в Java 9+ конкатенация через `+` использует `invokedynamic` и `StringConcatFactory`, что умнее, чем старый `StringBuilder`. Но в цикле оптимизация не работает — каждая итерация всё равно создаёт новые объекты.

**→ Уточняющий вопрос:** В чём разница между `new String("hello")` и `"hello"` с точки зрения String Pool?

## 6. Почему иммутабельные объекты thread-safe?

Потокобезопасность иммутабельных объектов следует из простого факта: если состояние объекта никогда не меняется, то нет ничего, что требовало бы синхронизации. Гонка данных (data race) возникает, когда один поток читает значение, пока другой его записывает. Если записи нет в принципе — нет и гонки.

Модель памяти Java (JMM) гарантирует, что `final`-поля, инициализированные в конструкторе, видимы всем потокам после завершения конструктора — без дополнительных `synchronized` или `volatile`. Это называется «freeze action» для `final`-полей. Поэтому иммутабельный объект, созданный в одном потоке, можно безопасно передать в другой через любой механизм.

```java
// Иммутабельный объект - безопасно шарить между потоками
final class Config {
    private final String host;
    private final int port;
    // ...конструктор и геттеры...
}

Config config = new Config("localhost", 8080);
// Можно передавать в любые потоки без синхронизации
executor.submit(() -> connect(config));
executor.submit(() -> monitor(config));
```

❗ Ловушка собеса: «А если поле `final`, но указывает на мутабельный объект — всё равно thread-safe?». Нет. `final List` гарантирует видимость ссылки на список, но не защищает от `list.add()` из другого потока. Для настоящей потокобезопасности мутабельное содержимое тоже должно быть защищено — или заменено на `List.copyOf()`.

**→ Уточняющий вопрос:** Что означает «freeze action» для `final`-полей в Java Memory Model?

## 7. Что такое ключевое слово final и как оно помогает в создании иммутабельных классов?

`final` в Java работает на трёх уровнях: поле, метод, класс. Для иммутабельности важны поля и класс. `final`-поле нельзя переприсвоить после инициализации в конструкторе — это запрещает методам изменить ссылку или примитивное значение. `final`-класс нельзя расширить наследованием, что исключает создание подклассов с мутабельным состоянием.

JMM даёт `final`-полям специальную гарантию: значения, записанные в `final`-поля в конструкторе, гарантированно видны другим потокам после завершения конструктора, даже без синхронизации. Это позволяет создавать иммутабельные объекты и передавать их между потоками без `volatile` или `synchronized`.

```java
public final class Point {
    private final int x; // нельзя переприсвоить
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }
    // нет сеттеров
}

// Point p = new Point(1, 2);
// p.x = 3; // ошибка компиляции
```

❗ Ловушка собеса: `final` на поле — это про ссылку, не про объект. `final int[] arr = {1, 2, 3}; arr[0] = 99;` — компилируется и выполняется. `final` запрещает `arr = new int[]{99}`, но не изменение элементов массива. Это принципиальное различие.

**→ Уточняющий вопрос:** Зачем нужен `final` на классе, если у нас уже `private final`-поля?

## 8. Достаточно ли сделать все поля final для иммутабельности?

Нет, недостаточно. `final` запрещает только переприсвоение ссылки, но не изменение объекта, на который она указывает. Класс с `final List<String> tags` всё равно мутабелен — можно вызвать `tags.add("x")` и изменить содержимое. Поле указывает на тот же список, но список изменился.

```java
public final class BrokenImmutable {
    private final List<String> items;

    public BrokenImmutable(List<String> items) {
        this.items = items; // ОШИБКА: храним переданную ссылку
    }

    public List<String> getItems() {
        return items; // ОШИБКА: отдаём внутреннюю ссылку
    }
}

List<String> list = new ArrayList<>(List.of("a", "b"));
BrokenImmutable obj = new BrokenImmutable(list);
list.add("c");                // изменяем через исходную ссылку
obj.getItems().add("d");      // изменяем через геттер
// obj.items теперь ["a", "b", "c", "d"] — иммутабельности нет
```

Для настоящей иммутабельности нужны три вещи вместе: `final`-ссылка + defensive copy при сохранении + возврат неизменяемого представления или копии из геттера. Только тогда внешний код не сможет изменить внутреннее состояние ни через какой путь.

❗ Ловушка собеса: это самый популярный вопрос-ловушка на тему иммутабельности. Кандидат говорит «все поля `final` — значит иммутабельный», интервьюер рисует пример с `List` и просит объяснить. Готовьте именно этот пример наизусть.

## 9. Что делать, если поле класса ссылается на мутабельный объект?

Нужно применить defensive copy в двух местах: при приёме значения в конструкторе (чтобы внешняя ссылка на оригинал не могла изменить внутреннее состояние) и при возврате значения из геттера (чтобы получатель не мог изменить внутренний объект через полученную ссылку).

```java
import java.util.Date;

public final class Event {
    private final String name;
    private final Date date; // мутабельный тип

    public Event(String name, Date date) {
        this.name = name;
        this.date = new Date(date.getTime()); // копия на входе
    }

    public String getName() { return name; }

    public Date getDate() {
        return new Date(date.getTime()); // копия на выходе
    }
}
```

Предпочтительная альтернатива — использовать изначально иммутабельные типы. Вместо `Date` — `LocalDate` или `Instant` (иммутабельны, нет нужды в defensive copy). Вместо `ArrayList` — `List.copyOf()` на входе, и можно безопасно возвращать тот же объект из геттера. Современный код должен избегать мутабельных `Date` и `Calendar` в пользу `java.time`.

❗ Ловушка собеса: копию нужно делать именно в конструкторе, до присвоения полю. Если написать `this.date = date; this.date = new Date(date.getTime());` — первая строка не имеет смысла, но хуже: если в конструкторе после присвоения что-то пойдёт не так, между двумя строками объект уже мутабелен.

**→ Уточняющий вопрос:** Чем `new Date(date.getTime())` отличается от `(Date) date.clone()`?

## 10. Что такое defensive copy (защитная копия)?

Defensive copy — приём, при котором вместо сохранения или возврата ссылки на мутабельный объект создаётся его независимая копия. Цель — разорвать связь между внутренним состоянием объекта и внешним кодом, который передал или получил объект.

Defensive copy нужна в двух точках: на входе (в конструкторе или фабричном методе) и на выходе (в геттере или методе, возвращающем мутабельное поле). Если сделать только одну из двух копий, защита неполная. Например, если копировать при сохранении, но отдавать оригинал из геттера — получатель может изменить объект через полученную ссылку.

```java
public final class Portfolio {
    private final List<String> assets;

    public Portfolio(List<String> assets) {
        // Defensive copy на входе: List.copyOf создаёт неизменяемую копию
        this.assets = List.copyOf(assets);
    }

    public List<String> getAssets() {
        // List.copyOf уже неизменяем — можно возвращать напрямую
        // Если бы хранили ArrayList — возвращали бы new ArrayList<>(assets)
        return assets;
    }
}
```

❗ Ловушка собеса: `Collections.unmodifiableList(assets)` в конструкторе — это не defensive copy! Это обёртка над оригиналом. Если внешний код держит ссылку на оригинальный список и изменит его, изменения будут видны через `unmodifiableList`. Настоящая копия — `new ArrayList<>(assets)` или `List.copyOf(assets)`.

**→ Уточняющий вопрос:** В чём разница между `List.copyOf()` и `new ArrayList<>(list)` для defensive copy?

## 11. Когда нужно делать defensive copy?

Defensive copy нужна всегда, когда иммутабельный класс принимает или возвращает мутабельный объект. Конкретные случаи: поле типа массив (`int[]`, `String[]`), коллекция (`List`, `Map`, `Set`), устаревшие мутабельные классы (`Date`, `Calendar`), любые объекты с публичными модифицирующими методами.

```java
public final class Snapshot {
    private final int[] values;

    public Snapshot(int[] values) {
        this.values = values.clone(); // копия массива на входе
    }

    public int[] getValues() {
        return values.clone(); // копия на выходе
    }

    public int getValue(int index) {
        return values[index]; // безопасно: отдаём примитив
    }
}
```

Defensive copy не нужна для примитивов (`int`, `long`, `boolean`) и иммутабельных объектов (`String`, `Integer`, `BigDecimal`, `LocalDate`, `LocalDateTime`). Также не нужна, если возвращаемый объект гарантированно неизменяем: `List.copyOf()` возвращает `ImmutableList`, который уже нельзя изменить.

❗ Ловушка собеса: для массивов используют `array.clone()` или `Arrays.copyOf(array, array.length)`, но не `System.arraycopy` для defensive copy (она корректна, но более многословна). Главный вопрос — делаете ли вы shallow copy или deep copy. `List.copyOf(list)` делает shallow copy: если элементы списка сами мутабельны — защита неполная.

**→ Уточняющий вопрос:** Если список содержит мутабельные объекты (`List<Date>`), достаточно ли `List.copyOf()`?

## 12. Как защитить коллекцию от изменений?

Есть три основных подхода. Первый — `Collections.unmodifiableList(list)`: возвращает view (обёртку), которая бросает `UnsupportedOperationException` при попытке изменения. Но это не копия — изменения исходного списка видны через эту обёртку. Второй — `List.of(...)` или `List.copyOf(list)` (Java 9+): создают полностью независимую неизменяемую копию. Третий — Guava `ImmutableList.copyOf(list)`: аналог `List.copyOf`, совместимый с Java 8.

```java
List<String> original = new ArrayList<>(List.of("a", "b", "c"));

// Вариант 1: unmodifiable (view, не копия)
List<String> view = Collections.unmodifiableList(original);
original.add("d"); // изменяем оригинал
System.out.println(view); // [a, b, c, d] — изменение видно!

// Вариант 2: List.copyOf (настоящая копия)
List<String> copy = List.copyOf(original);
original.add("e");
System.out.println(copy); // [a, b, c, d] — изменение НЕ видно
// copy.add("x"); // UnsupportedOperationException
```

Для иммутабельного класса правильный выбор — `List.copyOf()` при хранении в поле: создаём независимую копию и гарантируем, что никто не изменит её через исходный список. Из геттера можно возвращать эту же ссылку, так как `List.copyOf()` уже неизменяем.

❗ Ловушка собеса: `Collections.unmodifiableList` — одна из самых популярных ловушек. Код выглядит «защищённым», но если оригинальный список изменяется где-то ещё, обёртка это не скрывает. Всегда уточняйте: вам нужна защита от изменений из геттера, или полная изоляция от внешнего кода?

**→ Уточняющий вопрос:** Что вернёт `Collections.unmodifiableList(list).getClass()`?

## 13. Что такое Collections.unmodifiableList() и как это работает?

`Collections.unmodifiableList()` возвращает объект класса `Collections.UnmodifiableList` — обёртку (декоратор) над исходным списком. Методы чтения (`get`, `size`, `iterator`) делегируются оригиналу. Методы записи (`add`, `remove`, `set`, `clear`) выбрасывают `UnsupportedOperationException`. Это паттерн Decorator.

Ключевое слово — это view (представление), а не копия. Внутри хранится ссылка на оригинальный список. Если исходный список изменится, `unmodifiableList` немедленно отразит изменения. Таким образом, защита односторонняя: через обёртку изменить нельзя, но через оригинальную ссылку — можно.

```java
List<String> source = new ArrayList<>(List.of("a", "b"));
List<String> unmod = Collections.unmodifiableList(source);

// unmod.add("c");      // UnsupportedOperationException — защита работает
source.add("c");        // изменяем через оригинал
System.out.println(unmod); // [a, b, c] — видно изменение!

// Правильно для иммутабельного класса:
List<String> safe = List.copyOf(source); // реальная копия
source.add("d");
System.out.println(safe); // [a, b, c] — изменение не видно
```

❗ Ловушка собеса: не нужно путать `unmodifiableList` с иммутабельным списком. Первый — это «нельзя менять через эту ссылку», второй — «нельзя менять вообще». На собесе часто спрашивают именно это различие и просят написать пример, где `unmodifiableList` не защищает.

**→ Уточняющий вопрос:** Чем `List.of()` отличается от `List.copyOf()`?

## 14. В чём разница между shallow copy и deep copy?

Shallow copy (поверхностная копия) создаёт новый объект-контейнер, но копирует только ссылки на вложенные объекты — не сами объекты. Оба контейнера (оригинал и копия) ссылаются на одни и те же вложенные объекты. Изменение вложенного мутабельного объекта будет видно через обе ссылки.

Deep copy (глубокая копия) рекурсивно копирует и сам объект, и все вложенные объекты — создаётся полностью независимый граф объектов. Изменение в копии не затрагивает оригинал и наоборот.

```java
// Shallow copy: элементы Date — общие
List<Date> original = new ArrayList<>(List.of(new Date()));
List<Date> shallow = new ArrayList<>(original); // shallow copy
original.get(0).setTime(0); // меняем вложенный объект
System.out.println(shallow.get(0).getTime()); // 0 — видно изменение!

// Deep copy: элементы Date — независимые копии
List<Date> deep = new ArrayList<>();
for (Date d : original) deep.add(new Date(d.getTime()));
original.get(0).setTime(999);
System.out.println(deep.get(0).getTime()); // 0 — изменение не видно
```

`List.copyOf()` и `new ArrayList<>(list)` делают shallow copy: элементы не копируются. Для `List<Date>` или `List<List<...>>` этого недостаточно. Deep copy нужно реализовывать вручную или через сериализацию/рефлексию.

❗ Ловушка собеса: `List.copyOf(list)` — это shallow copy. Если элементы списка мутабельны (например, `List<Person>` где `Person` имеет сеттеры), то список неизменяем (нельзя добавить/удалить элемент), но сами объекты-элементы изменить можно через полученные ссылки.

**→ Уточняющий вопрос:** Когда в реальном коде нужна deep copy, а когда достаточно shallow?

## 15. Можно ли наследоваться от иммутабельного класса?

Технически — да, если класс не объявлен `final`. Но это опасно: подкласс может добавить мутабельные поля, переопределить геттеры или методы таким образом, что контракт иммутабельности будет нарушен. Код, принимающий тип «иммутабельного» класса, получит объект, который на самом деле может меняться — нарушение принципа подстановки Лисков в контексте иммутабельности.

```java
public class ImmutableBase {
    private final String value;
    public ImmutableBase(String v) { this.value = v; }
    public String getValue() { return value; }
}

// Подкласс ломает иммутабельность:
public class MutableChild extends ImmutableBase {
    private String extra;

    public MutableChild(String v, String e) {
        super(v);
        this.extra = e;
    }

    public void setExtra(String e) { this.extra = e; } // мутабельность

    @Override
    public String getValue() {
        return super.getValue() + extra; // результат меняется!
    }
}

ImmutableBase obj = new MutableChild("hello", " world");
// obj.getValue() => "hello world"
((MutableChild) obj).setExtra(" java");
// obj.getValue() => "hello java" — поведение изменилось!
```

Именно поэтому правило: иммутабельный класс должен быть `final`. Альтернатива — private конструктор с фабричными методами (тогда подкласс не может вызвать `super()` и практически не может быть создан извне).

❗ Ловушка собеса: «`String` не `final`» — это неверно. `String` как раз объявлен `final` начиная с самых ранних версий Java. Проверьте этот факт.

## 16. Почему иммутабельный класс должен быть final?

Если иммутабельный класс не `final`, подкласс может нарушить инвариант неизменности несколькими способами: добавить мутабельные поля, переопределить геттеры так, чтобы они возвращали меняющиеся значения, добавить методы, изменяющие унаследованное состояние через рефлексию или иные механизмы.

Проблема в том, что код, принимающий `ImmutableParent`, доверяет иммутабельности — не синхронизируется, кэширует значения, использует как ключ в Map. Если через полиморфизм подставить `MutableChild`, это доверие нарушается. Это особенно критично в многопоточных приложениях.

```java
// Безопасно только если SomeImmutable — final:
Map<SomeImmutable, String> cache = new HashMap<>();
SomeImmutable key = new SomeImmutable("data");
cache.put(key, "value");
// Если key.hashCode() или key.equals() может изменить поведение
// через переопределение в подклассе — кэш сломан
```

Альтернатива `final`-классу — сделать конструктор `private` и использовать статические фабричные методы. Тогда никто не сможет вызвать `super()` из подкласса, и наследование становится практически невозможным без взлома через рефлексию.

❗ Ловушка собеса: интервьюер может спросить «а что если нужна иерархия иммутабельных классов?». Ответ: используйте интерфейсы и sealed-классы (Java 17+). `sealed interface Shape permits Circle, Rectangle` — и каждый `permits`-класс может быть `final record`.

**→ Уточняющий вопрос:** Как sealed-классы в Java 17+ помогают строить иерархии иммутабельных типов?

## 17. Что произойдёт, если переопределить геттер в подклассе иммутабельного класса?

Если иммутабельный класс не `final`, подкласс может переопределить геттер и заставить его возвращать разные значения при разных вызовах — тем самым объект перестаёт быть иммутабельным с точки зрения наблюдателя. Это нарушает контракт, на который рассчитывает код, работающий с типом родителя.

```java
public class ImmutableValue {
    private final int value;
    public ImmutableValue(int v) { this.value = v; }
    public int getValue() { return value; }
}

public class TrickyChild extends ImmutableValue {
    private int counter = 0;

    public TrickyChild(int v) { super(v); }

    @Override
    public int getValue() {
        return super.getValue() + (counter++); // каждый вызов возвращает другое!
    }
}

ImmutableValue obj = new TrickyChild(10);
System.out.println(obj.getValue()); // 10
System.out.println(obj.getValue()); // 11
System.out.println(obj.getValue()); // 12 — объект "меняется"
```

Это классический пример нарушения принципа Лисков (LSP): `TrickyChild` нельзя использовать везде, где ожидается `ImmutableValue`. Код, кэширующий `getValue()` в предположении иммутабельности, получит некорректные результаты.

❗ Ловушка собеса: переопределение `hashCode()` или `equals()` в подклассе иммутабельного класса — ещё опаснее. Ключ в `HashMap` перестанет находиться, если `hashCode` стал зависеть от мутабельного состояния подкласса.

**→ Уточняющий вопрос:** Почему `final`-метод в родительском классе лучше, чем `final`-класс, если нужна частичная наследуемость?

## 18. Как работает String pool и как это связано с иммутабельностью?

String Pool (String Intern Pool) — специальная область в куче (в Java 8+ это часть Heap, а не PermGen), где JVM хранит уникальные экземпляры строковых литералов. Когда код использует строковый литерал `"hello"`, JVM проверяет пул: если строка уже есть, возвращает ту же ссылку; если нет — создаёт новый объект и добавляет в пул. Таким образом, несколько переменных могут указывать на один объект.

Это безопасно исключительно потому, что `String` иммутабелен. Если бы строки были мутабельны, изменение через одну ссылку затронуло бы все переменные, указывающие на тот же объект в пуле. Это было бы катастрофой для любого кода.

```java
String a = "hello";
String b = "hello";
String c = new String("hello"); // всегда новый объект, вне пула

System.out.println(a == b);         // true — одна ссылка из пула
System.out.println(a == c);         // false — c вне пула
System.out.println(a.equals(c));    // true — одинаковое содержимое

String d = c.intern(); // добавить c в пул (или получить уже существующий)
System.out.println(a == d);         // true — d теперь из пула
```

❗ Ловушка собеса: `new String("hello")` создаёт объект вне пула, даже если `"hello"` уже в нём есть. Вопрос «сколько объектов создаёт `new String("hello")`?» — ответ: один, если `"hello"` уже в пуле; потенциально два при первой встрече литерала. Сравнение строк всегда через `equals()`, не `==`.

**→ Уточняющий вопрос:** Что происходит с результатом `String.intern()` при сравнении через `==`?

## 19. Можно ли изменить значение String через рефлексию?

Технически — да. Можно получить доступ к приватному полю `value` (массив `byte[]` в Java 9+, `char[]` в Java 8) через рефлексию, снять модификатор `private` и изменить содержимое. Код скомпилируется и выполнится (с `--add-opens` в Java 9+).

```java
String s = "hello";
Field field = String.class.getDeclaredField("value");
field.setAccessible(true); // требует --add-opens java.base/java.lang=ALL-UNNAMED
byte[] value = (byte[]) field.get(s);
value[0] = 'J'; // меняем 'h' на 'J'
System.out.println(s); // "Jello"

// Но теперь сломан String Pool:
String other = "hello";
System.out.println(other); // тоже "Jello" — тот же объект из пула!
```

Последствия разрушительны: так как пул переиспользует объекты, изменение одного экземпляра может затронуть все строки с тем же значением в JVM. Кэшированный `hashCode` больше не соответствует содержимому. Объекты-ключи в `HashMap` потеряются. Это undefined behavior.

❗ Ловушка собеса: в Java 9+ по умолчанию включён strong encapsulation (`--illegal-access=deny` стал значением по умолчанию в Java 16). Без явного `--add-opens` рефлексия к `java.lang.String.value` бросит `InaccessibleObjectException`. Это целенаправленная защита от таких взломов.

**→ Уточняющий вопрос:** Почему взлом String через рефлексию может сломать код, который никогда не трогал эту строку?

## 20. Что такое Record и как он помогает создавать иммутабельные классы?

`record` — специальный вид класса, появившийся в Java 16 (preview в 14-15). Он автоматически генерирует: приватные `final`-поля, публичный конструктор со всеми компонентами, геттеры с именами компонентов (без `get`-префикса), `equals()`, `hashCode()`, `toString()`. Класс неявно `final` и не наследуется от других классов.

```java
// Запись — компактная иммутабельная структура
public record Point(int x, int y) {}

// Эквивалентно приблизительно такому классу:
public final class Point {
    private final int x;
    private final int y;
    public Point(int x, int y) { this.x = x; this.y = y; }
    public int x() { return x; }
    public int y() { return y; }
    // equals, hashCode, toString — автоматически
}

// Использование:
Point p = new Point(1, 2);
System.out.println(p.x()); // 1
System.out.println(p);     // Point[x=1, y=2]
```

Record идеален для DTO в REST API, value-объектов в DDD, результатов запросов. Можно добавить кастомный compact constructor для валидации:

```java
public record Range(int min, int max) {
    public Range { // compact constructor
        if (min > max) throw new IllegalArgumentException("min > max");
    }
}
```

❗ Ловушка собеса: Record не гарантирует полной иммутабельности, если компоненты — мутабельные объекты. `record Holder(List<String> items)` — поле `items` иммутабельно как ссылка, но список можно изменить через `holder.items().add("x")`. Нужен compact constructor с `List.copyOf(items)`.

## 21. Почему LocalDate, LocalDateTime иммутабельны?

`LocalDate`, `LocalDateTime`, `ZonedDateTime` и весь пакет `java.time` разработан Стивом Колборном (создатель Joda-Time) с иммутабельностью как фундаментальным принципом. Причины: потокобезопасность без синхронизации, предсказуемость (метод не может изменить аргумент), удобная передача в лямбды и Stream API, возможность использовать как ключи в Map.

Старые `java.util.Date` и `Calendar` были мутабельны, что порождало классические баги: метод получает дату, меняет её внутри себя, и у вызывающего кода дата неожиданно изменилась. С `LocalDate` это невозможно: `plusDays()`, `withMonth()`, `atStartOfDay()` — все методы возвращают новый объект.

```java
LocalDate today = LocalDate.now();
LocalDate nextWeek = today.plusDays(7); // новый объект
LocalDate endOfMonth = today.withDayOfMonth(today.lengthOfMonth()); // новый объект

System.out.println(today);       // исходная дата не изменилась
System.out.println(nextWeek);    // today + 7 дней
System.out.println(endOfMonth);  // конец месяца

// Безопасно передавать в несколько потоков:
LocalDate deadline = LocalDate.of(2024, 12, 31);
tasks.forEach(task -> task.setDeadline(deadline)); // ok, deadline неизменна
```

❗ Ловушка собеса: `LocalDate` не хранит время и часовой пояс. `LocalDateTime` не хранит часовой пояс. Если код работает с разными временными зонами, нужен `ZonedDateTime` или `OffsetDateTime`. Иммутабельность здесь не помогает, если выбран неправильный тип.

**→ Уточняющий вопрос:** Почему `LocalDate.now()` не подходит для хранения бизнес-даты в многозонном приложении?

## 22. В чём преимущества иммутабельных объектов для кэширования?

Иммутабельный объект можно безопасно кэшировать без копирования: гарантировано, что кэшированный объект никто не изменит, а значит, все получатели из кэша видят одно и то же корректное значение. Именно этот принцип используется в Spring `@Cacheable` — закэшированный результат метода возвращается напрямую, без создания копии.

```java
// Guava Cache с иммутабельными значениями — полностью безопасно
LoadingCache<String, ProductInfo> cache = CacheBuilder.newBuilder()
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(key -> productRepository.findByCode(key));

// ProductInfo — иммутабельный Record:
public record ProductInfo(String code, String name, BigDecimal price) {}

// Множество потоков могут получать один и тот же объект из кэша:
ProductInfo info = cache.get("SKU-001"); // безопасно для всех потоков
```

Дополнительные преимущества: `hashCode` можно вычислить один раз и закэшировать (как это делает `String`), объект можно использовать как ключ в `ConcurrentHashMap` без страха коллизий из-за изменения, иммутабельные объекты хорошо работают с `WeakReference`-кэшами — GC может их собрать, не боясь consistency-проблем.

❗ Ловушка собеса: «А если в кэше устаревший объект?» — это проблема инвалидации кэша, не иммутабельности. Иммутабельность гарантирует, что кэшированный объект не изменится неожиданно, но не управляет временем жизни кэша. Для этого нужны `expireAfterWrite`, `@CacheEvict` и т.д.

## 23. Как иммутабельность влияет на производительность?

Иммутабельность имеет как плюсы, так и минусы для производительности. Минус: каждое «изменение» создаёт новый объект в куче. При частых изменениях (например, построение строки в цикле через `+`) это создаёт значительную нагрузку на GC — много короткоживущих объектов, которые нужно собрать. Поэтому для частых изменений используют мутабельные «строители»: `StringBuilder`, `ArrayList`, `HashMap`, — а иммутабельный объект создают один раз в конце.

```java
// Плохо: n объектов String в куче
String result = "";
for (int i = 0; i < 1000; i++) result += i; // O(n²) копирований

// Хорошо: один StringBuilder, один итоговый String
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) sb.append(i);
String result = sb.toString();
```

Плюсы: отсутствие синхронизации (нет блокировок, нет cache-miss от `volatile`-чтений), безопасное шарение (не нужно копировать при передаче), кэширование `hashCode`, возможность более агрессивной оптимизации JIT-компилятором (объект не изменится — можно заинлайнить значения). В многопоточных системах выигрыш от отсутствия синхронизации часто перевешивает стоимость создания объектов.

❗ Ловушка собеса: JVM современные версии (особенно с Escape Analysis) могут вообще не помещать короткоживущие объекты в кучу — они создаются на стеке или вообще устраняются. Поэтому «создание новых объектов = медленно» — упрощение, которое не всегда верно на практике.

**→ Уточняющий вопрос:** Что такое Escape Analysis в JVM и как он влияет на стоимость создания иммутабельных объектов?

## 24. Что такое persistent data structures?

Persistent data structures (персистентные структуры данных) — иммутабельные структуры, которые при «изменении» не создают полную копию, а используют structural sharing: новая версия переиспользует максимально возможную часть старой структуры, создавая только новые узлы для изменённого пути. Это позволяет иметь несколько «версий» структуры одновременно с минимальными затратами памяти.

Классический пример — персистентный связный список: добавление в начало (`cons(element, list)`) создаёт один новый узел, который ссылается на старый список. Обе версии существуют одновременно и разделяют хвост. Аналогично для деревьев (например, `HashMap` как Hash Array Mapped Trie в Clojure): изменение одного листа требует пересоздания только узлов на пути от корня до листа — `O(log n)` новых узлов вместо `O(n)`.

```
// Схема: персистентный список
v1: [A] -> [B] -> [C]
v2: [D] -> [B] -> [C]  // structural sharing: [B]->[C] общие
// Добавление D в начало создало только один новый узел
```

В Java персистентные структуры из коробки не встроены, но доступны через библиотеки: Vavr (`io.vavr`), Eclipse Collections, PCollections. В функциональных языках (Clojure, Scala, Haskell) они являются стандартом.

❗ Ловушка собеса: «персистентный» в этом контексте не означает «сохранённый на диск» (persistence как durability). Это термин из теории структур данных — структура «помнит» все свои предыдущие версии. Не путайте с persistence в смысле JPA/Hibernate.

## 25. Есть ли недостатки у иммутабельных объектов?

Да, у иммутабельности есть реальные недостатки. Первый — накладные расходы на создание объектов: каждое «изменение» порождает новый объект. В высоконагруженных системах с частыми обновлениями (например, счётчики, накопление состояния) это создаёт давление на GC. Поэтому там используют AtomicLong, LongAdder или мутабельные аккумуляторы внутри метода.

Второй недостаток — более сложный API для «изменения». Вместо `user.setName("John")` нужен либо Builder, либо `wither`-метод: `user.withName("John")`. При десятках полей это многословно. Lombok `@Builder` и `@With` частично решают проблему, но добавляют зависимость.

```java
// Мутабельный подход — просто:
user.setName("John");
user.setEmail("john@example.com");

// Иммутабельный подход — многословнее:
User updated = user
    .withName("John")
    .withEmail("john@example.com"); // каждый with() создаёт новый объект

// Или через Builder:
User updated = user.toBuilder()
    .name("John")
    .email("john@example.com")
    .build();
```

Третий недостаток — циклические зависимости невозможны для полностью иммутабельных объектов: если `A` ссылается на `B`, а `B` должен ссылаться на `A`, и оба иммутабельны — создать их невозможно (курица-яйцо проблема). Решение — ленивая инициализация или мутабельный этап сборки графа.

❗ Ловушка собеса: «иммутабельность всегда лучше» — неверное утверждение. Для счётчиков, аккумуляторов, кэшей с обновлением, буферов ввода-вывода — мутабельность уместнее. Выбор определяется контекстом использования.

## 26. Как реализовать Builder pattern для иммутабельного класса?

Builder pattern для иммутабельного класса — стандартный способ создавать объекты с множеством опциональных полей без телескопического конструктора. Внутренний класс `Builder` накапливает значения через fluent-методы, а `build()` создаёт финальный иммутабельный объект через приватный конструктор.

```java
public final class UserProfile {
    private final String id;
    private final String name;
    private final String email;
    private final List<String> roles;
    private final LocalDate createdAt;

    private UserProfile(Builder builder) {
        this.id = Objects.requireNonNull(builder.id, "id required");
        this.name = Objects.requireNonNull(builder.name, "name required");
        this.email = builder.email;
        this.roles = List.copyOf(builder.roles); // defensive copy
        this.createdAt = builder.createdAt != null ? builder.createdAt : LocalDate.now();
    }

    public String getId() { return id; }
    public String getName() { return name; }
    public String getEmail() { return email; }
    public List<String> getRoles() { return roles; }
    public LocalDate getCreatedAt() { return createdAt; }

    public Builder toBuilder() { // для создания "изменённой" копии
        return new Builder(this);
    }

    public static Builder builder() { return new Builder(); }

    public static final class Builder {
        private String id;
        private String name;
        private String email;
        private List<String> roles = new ArrayList<>();
        private LocalDate createdAt;

        private Builder() {}

        private Builder(UserProfile p) { // для toBuilder()
            this.id = p.id;
            this.name = p.name;
            this.email = p.email;
            this.roles = new ArrayList<>(p.roles);
            this.createdAt = p.createdAt;
        }

        public Builder id(String id) { this.id = id; return this; }
        public Builder name(String name) { this.name = name; return this; }
        public Builder email(String email) { this.email = email; return this; }
        public Builder role(String role) { this.roles.add(role); return this; }
        public Builder createdAt(LocalDate d) { this.createdAt = d; return this; }

        public UserProfile build() { return new UserProfile(this); }
    }
}

// Использование:
UserProfile profile = UserProfile.builder()
    .id("u-1")
    .name("Alice")
    .email("alice@example.com")
    .role("ADMIN")
    .role("USER")
    .build();

// "Изменение" через toBuilder():
UserProfile updated = profile.toBuilder()
    .email("alice@new.com")
    .build();
```

❗ Ловушка собеса: Builder сам по себе мутабелен — это нормально. Главное, что итоговый объект, созданный `build()`, иммутабелен. Также важно делать defensive copy при передаче коллекций из Builder в объект, иначе внешний код, держащий ссылку на Builder, может повлиять на созданный объект.

**→ Уточняющий вопрос:** Lombok `@Builder` — он реализует defensive copy для полей-коллекций?

## 27. Можно ли использовать иммутабельные объекты как ключи в HashMap?

Да, и это идеальный вариант. `HashMap` требует от ключей стабильного `hashCode()` и корректного `equals()`. Иммутабельный объект гарантирует: его состояние (а значит, `hashCode` и `equals`) никогда не изменится. Ключ всегда находится в правильном бакете, `get()` всегда находит то, что было положено `put()`.

```java
public final class OrderId {
    private final String value;

    public OrderId(String value) {
        this.value = Objects.requireNonNull(value);
    }

    @Override
    public int hashCode() { return value.hashCode(); }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof OrderId)) return false;
        return value.equals(((OrderId) o).value);
    }
}

Map<OrderId, Order> orders = new HashMap<>();
OrderId id = new OrderId("ORD-001");
orders.put(id, new Order(...));

// Всегда найдём:
Order found = orders.get(new OrderId("ORD-001")); // equals() сработает корректно
```

`String`, `Integer`, `Long`, все стандартные обёртки, `UUID`, `LocalDate` — все иммутабельны и отлично работают как ключи. Record тоже идеален: `hashCode()` и `equals()` генерируются по всем компонентам, которые `final`.

❗ Ловушка собеса: иммутабельность ключа необходима, но недостаточна. `hashCode()` и `equals()` должны быть правильно реализованы. Объект может быть иммутабельным, но если `hashCode()` всегда возвращает 0 — HashMap превращается в связный список с O(n) для всех операций.

**→ Уточняющий вопрос:** Что происходит с контрактом `equals`/`hashCode`, если `hashCode` кэшируется в поле — нет ли риска при серилизации?

## 28. Что произойдёт, если изменить мутабельный ключ в HashMap?

Если изменить мутабельный объект, используемый как ключ в `HashMap`, его `hashCode()` изменится. Ключ окажется не в том бакете, где был помещён при `put()`. При вызове `get()` с тем же объектом HashMap вычислит новый `hashCode`, найдёт другой (или пустой) бакет, и запись не будет найдена — метод вернёт `null`. Запись «потеряна»: она существует в карте, но стала недоступной.

```java
List<String> key = new ArrayList<>(List.of("a", "b"));
Map<List<String>, String> map = new HashMap<>();
map.put(key, "value");

System.out.println(map.get(key)); // "value" — работает

key.add("c"); // изменяем ключ!
System.out.println(map.get(key)); // null — запись "потеряна"

// Запись всё ещё в map:
System.out.println(map.size()); // 1 — объект есть

// Найти можно только перебором:
map.forEach((k, v) -> System.out.println(k + " -> " + v));
// [a, b, c] -> value — вот оно
```

В реальных системах такой баг крайне трудно найти: Map не пустая, `size()` показывает правильное число, но `get()` возвращает `null`. Решение: никогда не использовать мутабельные объекты как ключи. Если нужна коллекция-ключ — `List.copyOf()` и обернуть в иммутабельный value object.

❗ Ловушка собеса: это не только теория. В продакшн-коде бывают случаи, когда `Map<SomeEntity, Something>` где `SomeEntity` имеет мутабельное состояние. После изменения сущности ключ теряется. JPA-сущности с `@Id` как ключи в Map — классический источник подобных багов, если id присваивается после вставки в БД.

**→ Уточняющий вопрос:** Почему `HashSet` имеет ту же проблему с мутабельными элементами, что и `HashMap` с мутабельными ключами?

## 29. Как правильно работать с коллекциями в иммутабельных классах?

Правильная работа с коллекциями в иммутабельных классах требует трёх шагов: на входе делать `List.copyOf()` (не хранить переданную ссылку), хранить результат как `final`-поле, на выходе возвращать либо тот же `List.copyOf`-объект (он уже неизменяем), либо обёртку `Collections.unmodifiableList()`. Никогда не отдавать прямую ссылку на внутренний `ArrayList`.

```java
public final class Classroom {
    private final List<String> students;

    public Classroom(List<String> students) {
        // List.copyOf — создаёт неизменяемую независимую копию
        // Бросает NPE если students == null или содержит null
        this.students = List.copyOf(students);
    }

    public List<String> getStudents() {
        // Можно возвращать напрямую: List.copyOf уже неизменяем
        return students;
        // Если бы хранили ArrayList: return Collections.unmodifiableList(students);
    }

    public Classroom withStudent(String student) {
        // "Изменение" — создаём новый объект
        List<String> updated = new ArrayList<>(students);
        updated.add(student);
        return new Classroom(updated);
    }
}

// Пример использования:
List<String> input = new ArrayList<>(List.of("Alice", "Bob"));
Classroom cls = new Classroom(input);
input.add("Charlie"); // не влияет на cls
cls.getStudents().add("Dave"); // UnsupportedOperationException
```

Для `Map` и `Set` аналогично: `Map.copyOf()`, `Set.copyOf()` (Java 10+). Для вложенных мутабельных объектов внутри коллекции нужна deep copy — `List.copyOf()` защищает контейнер, но не содержимое. Если элементы коллекции иммутабельны (String, Record, Integer) — `List.copyOf()` полностью достаточно.

❗ Ловушка собеса: `List.copyOf()` бросает `NullPointerException`, если список содержит `null`-элемент, тогда как `new ArrayList<>(list)` проглотит `null`. Это различие поведения важно при работе с потенциально `null`-содержащими данными от внешних API.
